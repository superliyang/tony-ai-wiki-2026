# 已有知识图谱专家问题地图

这张地图给当前已有知识图谱补一层“专家问题感”和“领域结构感”。它不是最终答案，而是每个领域进一步沉淀 `领域全貌图.md`、`专家问题清单.md`、`学习路径.md` 的起点。

## AI

入口：[[10-Knowledge/AI/专题总览]]

```text
目标：理解 AI 能力、模型、产品、组织和社会影响如何演化
角色：模型公司、开源社区、开发者、企业、监管者、用户
流程：研究突破 -> 模型能力 -> 产品化 -> 生态扩散 -> 组织采用
指标：能力边界、成本、可靠性、采用率、安全性
问题：什么能力是真突破，什么只是包装？什么会改变工作流？
解法：模型评估、产品案例、组织研究、时间线、失败复盘
失败：追热点、只看模型榜单、忽略采用成本和组织阻力
```

专家问题：

- 这个 AI 进展改变的是模型能力、产品体验、工作流，还是组织采用门槛？
- 它的核心瓶颈是数据、算力、推理成本、可靠性、分发，还是监管？
- 如果用户没有采用，是模型能力不足，还是 workflow friction 太高？
- 哪些能力会从 demo 变成基础设施，哪些只会停留在营销叙事？

## AI Engineering

入口：[[10-Knowledge/AI-Engineering/专题总览]]

```text
目标：把 AI 能力稳定、可评估、可运维地放进真实系统
角色：工程团队、平台团队、产品、风险/安全、业务 owner、终端用户
流程：需求 -> 数据/上下文 -> 模型/工具 -> 编排 -> 评估 -> 部署 -> 观测 -> 迭代
指标：任务成功率、延迟、成本、可追踪性、失败恢复、人工介入率
问题：什么时候用 workflow，什么时候用 agent？如何评估和回滚？
解法：RAG、tool use、eval、LLMOps、AgentOps、sandbox、observability
失败：过度 agent 化、无评估上线、上下文污染、权限过大、不可回滚
```

专家问题：

- 这个系统的失败边界在哪里？失败后是重试、降级、人工接管还是回滚？
- 评估指标是离线 benchmark，还是真实任务成功率？
- 哪些状态必须持久化，哪些上下文应该短期存在？
- Agent 的权限、审批、事实源和审计链如何设计？

## AI Open Source

入口：[[10-Knowledge/AI-Open-Source/专题总览]]

```text
目标：理解开源 AI 项目的生态位、成熟度、可采用性和组合方式
角色：维护者、贡献者、企业用户、云厂商、模型平台、插件生态
流程：项目出现 -> 社区增长 -> API 稳定 -> 生产采用 -> 商业化/治理
指标：活跃度、维护质量、文档、生态兼容、issue 质量、release 节奏
问题：这是基础设施、框架、工具、实验项目，还是生态入口？
解法：项目画像、生态位地图、adoptability checklist、替代品比较
失败：只看 star、忽略维护风险、忽略迁移成本和锁定风险
```

专家问题：

- 这个项目解决的是开发效率、运行稳定性、评估、部署，还是生态连接？
- 它和 LangGraph、AutoGen、OpenHands、MCP、A2A 等系统的边界是什么？
- 如果项目停更，迁移成本在哪里？
- 它适合原型、内部工具、生产系统，还是平台级能力？

## Advertising

入口：[[10-Knowledge/Advertising/广告投放专题总览]]

```text
目标：在预算、信号、归因和现金约束下获取可回收的高质量用户或转化
角色：广告主、UA/增长团队、DSP、SSP、ADX、媒体、MMP、验证商、财务、产品团队
流程：预算/目标 -> 受众/创意 -> 竞价/投放 -> 归因/测量 -> 增量校准 -> 预算再分配
指标：ROAS、CPA/CPI、LTV、Payback Period、CTR/CVR、IVT、增量 ROAS
问题：平台自报效果与真实增量效果不一致，短窗口和长窗口指标冲突，信号持续损失
解法：服务端归因、SKAN / AdAttributionKit、MMP、增量测试、MMM、反作弊、PMP/PG 质量控制
失败：只看 CPI、信平台 ROAS、短窗口放量、忽略 fraud、UA/产品/财务 KPI 分裂
```

专家问题：

- 在某个游戏品类和现金跑道下，D7、D30、D90 和 Payback Period 分别应该用于什么决策？
- 当平台 ROAS 和增量 ROAS 明显不一致时，预算、KPI 和团队激励应该如何校准？
- iOS ATT 和 Cookie 弱化后，哪些信号可以恢复，哪些只能通过概率和实验校准？
- Open Exchange、PMP、Programmatic Guaranteed 应该如何按量、质、欺诈和品牌安全权衡？

## International Payments

入口：[[10-Knowledge/International-Payments/专题总览]]

```text
目标：让跨境交易成功、安全、低成本、可结算、可对账
角色：商户、用户、发卡行、收单行、PSP、卡组织、风控、财务、合规
流程：下单 -> 支付 -> 认证 -> 授权 -> 清结算 -> 对账 -> 拒付
指标：授权率、支付成功率、拒付率、3DS 转化、成本、到账时间、对账差异
问题：成功率、风险、体验、成本、覆盖、合规之间如何权衡？
解法：本地收单、智能路由、3DS 策略、风控规则、备份通道、对账系统
失败：误杀、拒付激增、通道不稳定、汇率损失、对账差异、合规缺口
```

专家问题：

- 在某个国家、卡种、用户阶段下，授权率、3DS friction、拒付率和通道成本如何取舍？
- 这个支付策略更适合首单转化，还是老用户复购？
- 支付失败是 issuer decline、risk rule、3DS、通道路由、用户体验，还是本地支付偏好问题？
- 对账差异发生时，哪个系统是 source of truth？

## Security

入口：[[10-Knowledge/Security/专题总览]]

```text
目标：在业务可用的前提下降低攻击、滥用、合规和供应链风险
角色：攻击者、用户、业务、风控、安全、平台、合规、供应商
流程：资产识别 -> 威胁建模 -> 控制设计 -> 检测响应 -> 复盘改进
指标：风险暴露、误报率、漏报率、响应时间、业务摩擦、合规覆盖
问题：安全控制如何不压垮业务体验？哪些风险必须前置？
解法：身份、权限、WAF、风控、监控、供应链治理、incident response
失败：只堆工具、不建责任链；误报过高；无演练；忽略业务绕过路径
```

专家问题：

- 这个控制降低的是哪类风险？它增加了什么业务摩擦？
- 攻击者最可能绕过哪一层？检测信号是否足够可解释？
- 误报和漏报哪个成本更高？在什么业务阶段答案会变？
- 事故发生后，谁有权限停机、封禁、回滚或通知用户？

## Personal Knowledge System

入口：[[10-Knowledge/Personal-Knowledge-System/专题总览]]

```text
目标：让知识持续捕获、判断、沉淀、复用，并能被 AI 协作系统可靠读取
角色：Tony、Obsidian、Codex、Hermes、ChatGPT、GitHub、未来 Agent
流程：Capture -> Recall -> Judge -> Crystallize -> Reuse
指标：可找回性、可复用性、结构清晰度、更新成本、AI 可读性、复习率
问题：哪些内容进入 inbox，哪些成为 canonical？AI 如何不乱改？
解法：三层结构、review queue、地图、playbook、agent memory、Git checkpoint
失败：信息堆积、无 review、地图过期、重复笔记、Agent 写入边界失控
```

专家问题：

- 这个知识系统的 source of truth 是什么？哪些只是 projection？
- AI 生成内容什么时候能进入 canonical knowledge？
- 知识是按主题组织、项目组织、问题组织，还是学习路径组织？
- 如何让复习、重构、合并重复和新内容吸收成为持续流程？
