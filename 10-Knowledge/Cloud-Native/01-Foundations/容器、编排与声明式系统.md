# 容器、编排与声明式系统

## 这一页在回答什么

这一页回答的是：为什么云原生不是“把应用丢到云上”这么简单，而是逐步形成了 `container -> orchestration -> declarative API -> control loop` 这一整套系统。

## 容器解决了什么

容器让应用和依赖可以被更一致地打包与运行。它的关键价值不是“更轻”，而是把应用运行环境标准化，让同一个 workload 更容易在不同环境里迁移。

## 为什么还需要编排

当应用变成多个 container，并且需要扩缩容、滚动发布、故障恢复、服务发现和权限隔离时，单个主机上的 container runtime 已经不够了。这时就需要 orchestration system。

## 为什么声明式很关键

云原生系统最重要的转变之一，是从“命令式地一步步操作系统”转向“声明式地描述期望状态”。

你不再重点关心：
- 先执行什么命令
- 再创建什么资源
- 出故障后人工怎么补救

而是更关心：
- 我希望系统长成什么样
- 当前状态和期望状态差了什么
- 谁持续把它们收敛起来

## 控制循环是核心心智模型

Kubernetes、GitOps controller、operator 等都建立在同一个思想上：

- 观察当前状态
- 对比期望状态
- 做出收敛动作
- 重复这个循环

这就是后面学习 Kubernetes、GitOps 和 platform engineering 时最重要的心智模型。

## 读完这一页你应该带走什么

- 容器是标准化运行单元
- 编排是为了解决分布式运行问题
- 声明式系统比脚本堆叠更适合规模化运维
- 控制循环是云原生系统最重要的运行机制之一
