# 策略与供应链安全接入工作流

## 这个工作流在解决什么

云原生安全真正难的地方，不是“知道要更安全”，而是怎么把安全要求变成平台默认能力，而不是每个团队各自手工执行。

在实践里，这条线通常会落到两类能力：

- `policy as code`：用 admission / validation / mutation 去约束集群资源
- `supply chain verification`：用签名、attestation、policy controller 去约束可运行镜像

## 推荐工作流

1. 先定义最小安全边界
   - 哪些资源必须受策略约束
   - 哪些镜像必须签名或带 attestation
   - 哪些 namespace 先试点
2. 把规则写成平台资源
   - `Kyverno` policy
   - `ClusterImagePolicy` / admission policy
3. 先以 audit 或 warn 模式验证
   - 不要一上来就全量 enforce
   - 先看误报、例外和开发者摩擦点
4. 对接 CI/CD 与 artifact 流水线
   - 构建时生成签名
   - 保存 provenance / attestation
   - 部署前验证
5. 在集群侧做 admission enforcement
   - 未签名镜像拒绝
   - 不符合 policy 的资源拒绝或标记
6. 处理例外机制
   - 明确谁能申请例外
   - 例外是否限时
   - 是否要求补偿控制
7. 把结果纳入治理节奏
   - 通过率
   - 阻断率
   - 误伤率
   - 高风险 repo / namespace / team

## 关键判断

- 云原生安全不能只盯 runtime，必须把 build -> sign -> verify -> admit 接成一条链
- policy 设计得太早太重，会让 adoption 失败
- 先 `audit -> enforce`，比直接强推更容易走通
- 供应链安全最终会变成平台团队和安全团队共担的控制面
