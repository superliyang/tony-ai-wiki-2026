# 服务网格流量变更工作流

## 这个工作流在解决什么

当团队已经进入微服务与 Kubernetes 阶段后，流量变更不再只是改一个 `Ingress` 或调一个负载均衡，而是会同时牵动：

- `routing`
- `retries / timeouts / outlier detection`
- `mTLS and identity`
- `observability`
- rollback

所以 service mesh 的真实价值，不只是“有了一个 mesh”，而是把流量变更变成一个更可治理的 workflow。

## 推荐工作流

1. 明确这次变更属于什么类型
   - 新版本灰度
   - 服务切流
   - 安全策略收紧
   - 故障恢复与回滚
2. 先确认基线是否稳定
   - 当前错误率、延迟、成功率、上游依赖健康度
   - tracing / metrics / logs 是否可用
3. 在 mesh 层定义变更策略
   - traffic split
   - timeout / retry / circuit breaker
   - destination policy
   - mTLS / authorization policy
4. 在低风险范围验证
   - 小流量灰度
   - 单 namespace / 单 service / 单 tenant 试点
5. 观察平台级信号
   - 请求成功率
   - p95 / p99 latency
   - upstream failure
   - policy rejection
6. 只在观测稳定后逐步扩量
7. 预先定义 rollback 条件
   - 错误率阈值
   - latency 回退阈值
   - 关键 API 失败率
8. 变更结束后记录
   - 哪些流量策略有效
   - 哪些 mesh 能力真正减少了应用侧复杂度

## 关键判断

- 不要把 service mesh 当成单纯网络中间件，它更像平台级流量控制面
- 流量治理和安全治理在这里会自然耦合
- 如果没有可观测性，mesh 只会让系统更复杂，不会更安全
