# 工作负载形态选择工作流

## 简介

`工作负载形态选择工作流` 关注的是：当一个团队要上线新服务时，应该如何判断它更适合传统 Deployment、serverless-like runtime，还是 stateful storage-heavy 路径。

## 输入

- 服务是否长期在线
- 是否需要事件驱动或按需触发
- 是否要求 scale to zero
- 是否有持久化和状态需求
- 对启动延迟、吞吐、成本和治理的约束

## 步骤

1. 先判断 workload 是长期在线还是按需触发
2. 再判断它是偏 request-driven、event-driven，还是 stateful
3. 对 stateless 高弹性场景评估是否适合 Knative 这类更高抽象路径
4. 对 stateful 场景评估 StorageClass、CSI、数据位置与生命周期要求
5. 明确运行时、存储和平台治理边界后，再进入标准交付路径

## 输出

- workload shape decision
- runtime choice
- storage requirement summary
- downstream platform requirements

## 评估

- 选型是否真的降低了复杂度，而不是只追逐新技术
- serverless / runtime / storage 边界是否被说清楚
- 后续交付是否更容易进入标准平台路径
