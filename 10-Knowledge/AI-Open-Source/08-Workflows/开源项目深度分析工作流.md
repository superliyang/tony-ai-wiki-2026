# 开源项目深度分析工作流

## 目标

这条工作流不是为了把 repo 收进知识库，而是为了把一个开源项目研究到“能判断、能对比、能迁移、能动手”的程度。

## 第一阶段：认知层

先回答这几个问题：

1. 它解决什么问题
2. 为什么现在值得关注
3. 它替代谁，或者和谁互补
4. 适合什么场景，不适合什么场景
5. 它在整个 stack 的哪一层

如果这一层还没清楚，不要急着读代码。

## 第二阶段：架构层

先建立整体心智图，而不是先读函数。

重点回答：

- 核心模块有哪些
- 模块之间如何协作
- 核心对象模型是什么
- 启动流程是什么
- 请求或数据是怎么流转的
- 哪些是核心路径，哪些是边缘能力

这一阶段最好至少产出：

- 一张架构图
- 一张数据流图或请求流图
- 一段“工作原理”总结

## 第三阶段：实现层

只抓最关键的 20%。优先看：

- 启动主流程
- 一个典型请求的完整调用链
- 配置变更如何生效
- 核心任务如何调度
- 异常和失败怎么处理

这里的目标不是“把每个类都看一遍”，而是先把一条关键链路走通。

## 第四阶段：工程层

把项目当作工程系统看，而不是 demo：

- 包结构和模块划分是否合理
- 配置管理是否清晰
- 日志、监控、Tracing 是否成体系
- 测试用例如何组织
- 并发、缓存、队列、连接池怎么处理
- 性能优化点在哪里
- 升级兼容、恢复与回滚怎么做

## 第五阶段：社区与演进层

不要只学当前代码，也要看它是不是“活的”：

- maintainer / core contributors 是谁
- issue / PR / discussions 活跃度如何
- release cadence 如何
- roadmap 往哪走
- 社区主流使用方式是什么

## 第六阶段：实战层

真正掌握的标志不是“我看懂了”，而是：

- 你能本地跑起来
- 你能打断点或加日志
- 你能定位一条主流程
- 你能改一个小功能或修一个 bug
- 你能写一个最小扩展或插件
- 你能写出自己的项目判断总结

## 我们在知识库里的固定产物

每次做深度分析，至少产出：

1. 一张项目卡
2. 一张架构图
3. 一张数据流图或请求流图
4. 一段工作原理总结
5. 一段主流程 / 关键链路总结
6. 一段 tradeoff 判断
7. 一个下一步实验建议
8. 一组适配度标签判断（local/mac/production/learning）

## 什么时候算“学到了”

如果下面这些都能回答，就算比较扎实：

- 它为什么存在
- 为什么这么设计
- 这套设计在什么约束下成立
- 和相邻方案怎么比较
- 出问题会先坏在哪里
- 你能不能在自己的系统里复用它的设计
