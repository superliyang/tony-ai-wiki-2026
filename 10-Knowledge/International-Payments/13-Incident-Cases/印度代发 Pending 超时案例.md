# 印度代发 Pending 超时案例

## 异常现象

- 代发提交成功，但大量交易长时间停在 `pending`
- 客服无法明确回答收款人“钱到底到没到”
- 通道 API 看起来健康，但实际到账时效恶化
- 运营团队开始手动查单，效率迅速下降

## 第一反应最容易犯的错

- 只看 API 成功率，忽略到账时效
- 把所有 pending 都当成失败重发
- 没分清“审核中”“处理中”“外部回执未回”“真正失败”

## 更强的拆法

1. 把 pending 拆成审核积压、通道处理中、回执未回、状态同步异常
2. 看是否集中在特定通道、金额区间、受益人类型或资料缺失场景
3. 对比通道回执时间、内部状态更新时间和客服反馈时间
4. 判断是合规审核、异步回执还是状态机问题

## 最可能的根因路径

- 受益人资料质量不稳，审核积压放大
- 本地 rails 回执延迟，内部状态没及时同步
- payout 状态机不够细，运营只能看到一个模糊的 pending
- 客服没有统一状态解释口径，导致投诉升级

## 应对动作

- 建立 pending 超时分层看板
- 区分“待审”“处理中”“待回执”“需补资料”
- 对高频失败类型前置校验受益人信息
- 统一客服、运营、技术对到账状态的解释口径

## 应该拉谁进来

- 支付技术
- 合规 / 审核
- 支付运营
- 客服

## 该沉淀什么

- 把 pending 超时阈值写入负责人清单
- 把状态分类纳入 payout 看板模板
- 把这类异常回写到印度 payout 组合案例
