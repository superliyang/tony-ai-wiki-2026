# A2A 与银行转账支付

## 这页解决什么问题

`A2A`（Account-to-Account）与银行转账支付不是“卡支付的替代品”这么简单。它常常意味着更低成本、更强本地化，也意味着更高的异步复杂度、状态管理复杂度和对账要求。

## 常见形态

- 用户主动转账 `push payment`
- 商户发起账户扣款 `pull payment`
- 开放银行 `Open Banking` 场景
- 即时转账、本地银行 rail 或传统银行转账

## 这类方式最适合什么场景

- 大额交易
- B2B 收款
- 某些本地高渗透消费场景
- 低费率敏感场景
- payout / 账户到账相关场景

## 真正的难点

- 状态经常是异步的
- 用户“已付款”不等于系统“已确认”
- 回执、附言、对账和失败恢复要设计得更细
- 客服和运营必须知道如何解释“处理中”

## 为什么很多团队会踩坑

- 用卡支付心智理解银行转账
- 只看 API 成功，不看到账确认
- 不把订单、支付、对账和客服系统串起来
