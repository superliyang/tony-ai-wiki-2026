# 手机游戏端到端安全控制矩阵

> 这页把手机游戏业务按“业务链路”拆开，回答每个环节会有什么问题、做什么控制、看什么指标、留下什么证据。

## 控制总览

| 业务链路 | 主要风险 | 核心控制 | 关键指标 | 证据 |
|---|---|---|---|---|
| 注册 | 批量注册、羊毛党、小号农场 | 设备指纹、渠道校验、验证码、注册限速、黑产号码识别 | 注册成功率、批量设备数、异常渠道占比 | 注册日志、设备画像 |
| 登录 | 撞库、密码喷洒、代理池 | 多维限速、泄露密码、风险验证码、异常登录 step-up | 登录失败/成功比、撞库成功率、代理命中 | 登录风控日志 |
| 会话 | token 泄露、会话劫持、长期有效 token | refresh token 轮换、设备列表、异常下线、短期 access token | 异常会话数、session revoke 数 | 会话日志 |
| 找回 | 社工、客服滥用、找回后盗号 | 找回风控、延迟生效、冷却期、人工复核、证据强度分级 | 找回成功率、找回后资产转移率 | 找回记录、客服工单 |
| 换绑 | 盗号后接管账号 | 二次验证、旧设备通知、冷却期、高风险冻结 | 换绑失败率、异常换绑数 | 账号变更日志 |
| 客户端 | 改包、Hook、模拟器、脚本 | Play Integrity、App Attest、签名、反调试、资源校验 | 完整性失败率、模拟器占比 | 客户端安全日志 |
| API | 越权、重放、参数篡改 | 对象级授权、幂等、nonce、schema 校验、接口限速 | 4xx/5xx 异常、越权拦截 | API 网关日志 |
| 战斗 | 加速、外挂、内存改值 | 服务端权威、回放校验、行为模型、异常封禁复核 | 作弊命中率、误封率 | 战斗日志、回放 |
| 结算 | 奖励伪造、重复发奖 | 服务端结算、幂等、奖励账本、异常奖励检测 | 重复发奖数、异常产出 | 结算日志 |
| 经济 | 刷币、搬砖、盗卖、黑市交易 | 账本化、交易限额、异常路径冻结、经济监控 | 货币产消比、异常转移 | 经济账本 |
| 支付 | 伪造回调、重复发货、拒付 | 渠道验签、订单幂等、对账、退款风控 | 退款率、拒付率、对账差异 | 订单与对账日志 |
| 排行榜 | 刷分、机器人、异常战绩 | 结果校验、行为检测、排行榜延迟发布、人工复核 | 异常上榜数、刷分回滚 | 排行榜审计 |
| 社交聊天 | 诈骗、广告、未成年风险 | 内容审核、举报、黑名单、风控画像 | 举报率、诈骗命中 | 审核与举报记录 |
| GM 后台 | 越权、误操作、内部滥用 | RBAC/PAM、审批、双人复核、全量审计 | 高危操作数、未审批数 | 后台审计 |
| 数据平台 | PII 泄露、越权查询 | 数据分级、脱敏、最小权限、查询审计 | 敏感查询数、异常导出 | 数据访问日志 |
| CI/CD | 供应链投毒、密钥泄露 | SAST/SCA/Secrets、SBOM、签名发布、制品库权限 | 高危漏洞、泄露密钥 | 扫描报告、SBOM |

## 撞库专项控制

| 控制 | 说明 | 落地要点 |
|---|---|---|
| 多维限速 | 不只按 IP，也按账号、设备、ASN、国家地区、失败模式限速 | 风险分越高，限速越强 |
| 风险验证码 | 只对高风险登录触发，避免全量打扰 | 失败多、代理池、陌生设备、异常地区触发 |
| 泄露密码检测 | 注册、登录、改密时识别已泄露密码 | 不记录明文密码，使用安全的哈希/查询方案 |
| 新设备 step-up | 陌生设备成功登录前后要求二次验证 | 对高价值账号更严格 |
| 可疑会话撤销 | 高风险成功登录后撤销 session | 配合用户通知和申诉 |
| 换绑冷却期 | 登录后立即换绑风险高 | 高风险账号延迟生效 |
| 资产保护 | 高风险登录后限制交易、赠送、出售、大额消耗 | 先限制损失半径，再复核 |
| 用户通知 | 新设备、换绑、改密、敏感资产操作通知 | 给用户自助冻结入口 |
| 客服防社工 | 找回材料复用、批量申诉、异常来源识别 | 高风险工单升级审核 |

## 安全数据模型

最低要打通这些实体：

- `account_id`
- `role_id`
- `device_id / device_fingerprint`
- `ip / asn / geo`
- `app_version / channel`
- `integrity_verdict`
- `session_id`
- `order_id`
- `item_id / currency_id`
- `operation_id`
- `support_ticket_id`
- `trace_id`

如果没有统一 trace id，撞库后的“登录 -> 换绑 -> 资产转移 -> 客服申诉 -> 支付退款”会很难串起来。

## 成熟度分级

| 等级 | 状态 | 标志 |
|---|---|---|
| L0 | 被动救火 | 只有日志，没有规则和流程 |
| L1 | 单点防护 | 有验证码、限速、封 IP，但不联动资产 |
| L2 | 链路联动 | 登录风险能影响换绑、交易、客服和支付 |
| L3 | 平台化 | 账号、设备、经济、支付、客服、后台统一风控 |
| L4 | 对抗运营 | 有黑产样本、规则回归、红蓝演练、误伤复盘 |
