# 身份、密钥与访问平台

> 身份是云和 SaaS 时代的新边界；密钥是机器身份的密码。这个平台族决定“谁能做什么、凭什么、在什么条件下、留下什么证据”。

## 能力边界

| 子能力 | 解决的问题 | 常见平台形态 |
|---|---|---|
| IdP / SSO | 账号分散、认证体验割裂 | Okta、Microsoft Entra ID、Google Workspace |
| MFA / Passwordless | 密码泄露、钓鱼登录 | FIDO2/WebAuthn、Authenticator、Passkey |
| IAM / RBAC / ABAC | 权限过大、职责不清 | 云 IAM、应用 RBAC、策略引擎 |
| PAM / JIT | 高权限账号常驻 | CyberArk 类 PAM、临时提权、审批 |
| Secrets Management | API key、token、证书泄露 | Vault、OpenBao、云 Secret Manager |
| KMS / HSM | 密钥存储和使用不可控 | 云 KMS、HSM、BYOK/HYOK |
| 机器身份 | 服务间访问和证书生命周期 | SPIFFE/SPIRE、mTLS、证书自动轮换 |

## 落地闭环

1. 账号源统一：HRIS / IdP / 云账号 / SaaS 账号建立主数据。
2. 权限模型：定义角色、资源、操作、环境、审批条件。
3. 强认证：MFA、设备态势、地理位置、风险登录策略。
4. 高权限治理：PAM、JIT、break-glass、会话审计。
5. 密钥治理：集中存储、动态凭据、自动轮换、泄露检测。
6. 审计证据：登录、授权、提权、密钥读取、异常访问全部可查。

## 选型检查点

- 是否支持现有 IdP、云账号、SaaS、工单和审批系统？
- 是否能处理员工、外包、供应商、服务账号和机器身份？
- 是否支持 Just-in-Time access、权限到期、审批和 break-glass？
- 是否能把高权限操作记录到 SIEM 或审计平台？
- 是否能自动发现长期 secret、过期证书和未轮换凭据？

## 关键指标

- MFA 覆盖率
- 高权限账号数量
- 长期权限和长期 secret 数量
- 离职账号关闭时间
- 高权限审批和会话审计覆盖率
- secret 轮换成功率

## 典型陷阱

- 只上 SSO，不治理权限生命周期。
- 只做 MFA，不治理高权限和服务账号。
- 密钥进入 Vault，但应用仍把 secret 写在配置文件里。
- PAM 只管管理员账号，不管云 IAM、CI/CD token 和机器身份。
