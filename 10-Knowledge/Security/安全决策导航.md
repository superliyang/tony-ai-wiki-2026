# Security 决策导航

> 这页按“安全架构和治理里经常要做的取舍”进入。

## 决策 1：先补身份，还是先补网络边界？

- 判断标准：
  - 如果问题是账号滥用、云权限过大、API key 泄露，先补 [[./05-Topics/身份与访问安全|身份与访问安全]]
  - 如果问题是公网暴露、东西向移动、DDoS 或边界不清，先补 [[./05-Topics/网络安全|网络安全]]
- 高级判断：现代安全通常先补身份，因为云和 SaaS 环境里身份就是新边界。

## 决策 2：先买安全工具，还是先做资产与攻击面盘点？

- 判断标准：
  - 如果资产、入口、owner 不清，先做 [[./05-Topics/漏洞管理与攻击面管理|漏洞管理与攻击面管理]]
  - 如果已有清晰痛点和日志基础，再引入工具
- 高级判断：不知道保护什么时，工具只会制造更多告警。
- 关联：[[./02-Tools/工具平台分类索引|工具平台分类索引]]、[[./08-Playbooks/安全工具平台选型与落地 Playbook|安全工具平台选型与落地 Playbook]]

## 决策 3：SAST、DAST、SCA、IAST 先做哪个？

- 判断标准：
  - 依赖漏洞和 license 风险明显，先 SCA
  - Web/API 漏洞多，先 DAST + API testing
  - 代码质量和安全开发弱，先 SAST + code review
  - 需要运行时上下文，再考虑 IAST
- 关联：[[./05-Topics/安全工程与 DevSecOps|安全工程与 DevSecOps]]

## 决策 4：零信任是网络项目，还是身份项目？

- 判断标准：
  - 零信任不是单独网络项目，而是 `身份、设备、应用、数据、策略、持续验证` 的组合
  - 如果只能先做一层，优先 `身份 + 设备态势 + 应用访问`
- 关联：[[./05-Topics/零信任架构|零信任架构]]

## 决策 5：高风险动作自动化，还是人工审批？

- 判断标准：
  - 可逆、低风险、高频动作可以自动化
  - 不可逆、外部影响、涉及资金/数据/权限的动作需要审批
  - 自动化前必须有审计、回滚、dry-run
- 关联：[[./05-Topics/安全运营、检测与响应|安全运营、检测与响应]]

## 决策 6：漏洞发现后先修，还是先做风险接受？

- 判断标准：
  - 可被公网利用、已被利用、有敏感数据影响，优先修
  - 低可利用性、补丁风险高、业务影响大，可以进入风险接受，但必须有 owner、期限和补偿控制
- 关联：[[./05-Topics/漏洞管理与攻击面管理|漏洞管理与攻击面管理]]

## 决策 7：日志全收，还是按场景收？

- 判断标准：
  - 关键身份、云控制面、应用审计、数据访问、终端告警必须优先
  - 全量日志如果无人分析，会变成成本黑洞
- 关联：[[./05-Topics/安全运营、检测与响应|安全运营、检测与响应]]

## 决策 8：合规先行，还是风险先行？

- 判断标准：
  - 被监管或客户审计驱动时，合规先行能快速建立最低控制线
  - 真正降低事故概率时，必须回到风险先行
- 高级判断：成熟组织用合规证明控制存在，用风险管理决定控制优先级。
- 关联：[[./05-Topics/安全治理、风险与合规|安全治理、风险与合规]]

## 决策 9：数据加密、脱敏、Tokenization 怎么选？

- 判断标准：
  - 传输和存储保护：加密
  - 展示和分析最小化：脱敏 / masking
  - 支付、身份号等高敏字段替代：tokenization
  - 不该存的数据：直接不收集或删除
- 关联：[[./05-Topics/数据安全与隐私保护|数据安全与隐私保护]]

## 决策 10：地区合规由法务主导，还是安全主导？

- 判断标准：
  - 法律解释和监管口径由法务 / 隐私团队主导
  - 技术控制、证据、日志、数据流和事件响应由安全/工程主导
  - 最终需要共同 owner
- 关联：[[./05-Topics/地区合规与监管坐标|地区合规与监管坐标]]

## 决策 11：SOC 自建，还是托管 MDR？

- 判断标准：
  - 规模小、缺 24/7 能力，优先 MDR
  - 资产复杂、强监管、高敏数据，逐步自建核心检测与响应能力
  - 最差状态是买了 SIEM 但没人运营
- 关联：[[./06-Maps/安全运营与事件响应闭环图|安全运营与事件响应闭环图]]

## 决策 12：安全团队做 gatekeeper，还是 platform enabler？

- 判断标准：
  - 初期必须有 gate，防止高风险上线
  - 长期要把安全控制产品化到平台、模板、CI/CD、默认配置
  - 只做 gatekeeper 会被业务绕过
- 关联：[[./05-Topics/安全工程与 DevSecOps|安全工程与 DevSecOps]]

## 决策 13：遇到安全问题时，先诊断、先修复、先响应，还是先治理？

- 判断标准：
  - 事实不清、影响不明，先用 [[./08-Playbooks/安全快速诊断 Playbook|安全快速诊断 Playbook]]
  - 已确认攻击或泄露，先用 [[./08-Playbooks/安全事件响应 Playbook|安全事件响应 Playbook]]
  - 新系统上线或高风险变更，先用 [[./08-Playbooks/应用与 API 安全评审 Playbook|应用与 API 安全评审 Playbook]]
  - 问题反复出现，说明是架构和治理缺口，进入 [[./08-Playbooks/企业安全架构落地 Playbook|企业安全架构落地 Playbook]]
- 高级判断：成熟安全团队会把每次事件、漏洞和审计发现都转成平台默认控制、检测规则和审计证据。
- 关联：[[./06-Maps/安全问题解决工作台|安全问题解决工作台]]

## 决策 14：学习安全时先补知识点，还是先补可交付能力？

- 判断标准：
  - 如果连基本术语都不清楚，先补 [[./06-Maps/安全点线面能力地图|安全点线面能力地图]] 里的“点”
  - 如果知道术语但不会落地，进入 [[./05-Topics/安全能力模型：知道什么、会做什么|安全能力模型]] 和 [[./08-Playbooks/安全学习与实战路线 Playbook|安全学习与实战路线]]
  - 如果目标是解决真实工作问题，优先产出评审、诊断、响应、架构、治理文档
- 高级判断：真正的掌握不是“知道很多点”，而是能把点串成线，再把线组织成面，最后交付可复用成果。
- 关联：[[./08-Playbooks/Playbook 索引|Playbook 索引]]

## 决策 15：零信任先做员工访问，还是服务间访问？

- 判断标准：
  - 如果远程办公、外包、供应商访问风险最大，先做员工和第三方应用级访问
  - 如果云原生、微服务和生产服务复杂，尽早规划服务间身份、网络策略和 workload identity
  - 如果只能先做一件事，先统一身份、MFA 和高权限账号治理
- 高级判断：零信任不是一次性项目，而是从身份、设备、应用、数据、网络和持续监控逐层收紧。
- 关联：[[./08-Playbooks/零信任落地 Playbook|零信任落地 Playbook]]

## 决策 16：漏洞优先级看 CVSS，还是看攻击面？

- 判断标准：
  - CVSS 是漏洞本身严重性，不等于本组织风险
  - 公网暴露、在野利用、高权限、敏感数据、关键业务路径会显著提高优先级
  - 有强补偿控制、不可达、低权限、测试环境会降低优先级，但必须有证据
- 高级判断：漏洞管理要和攻击面管理合并，否则会修很多“分高但打不到”的问题，漏掉“分不高但真实暴露”的问题。
- 关联：[[./08-Playbooks/漏洞管理与攻击面管理 Playbook|漏洞管理与攻击面管理 Playbook]]

## 决策 17：供应链先做 SBOM，还是先做 CI/CD 加固？

- 判断标准：
  - 如果依赖不可见，先做 SCA 和 SBOM
  - 如果构建和发布权限过大，先做 CI/CD secret、runner、审批和 artifact 权限治理
  - 如果面向强监管或客户审计，SBOM、签名、provenance 和供应商评估要一起规划
- 高级判断：SBOM 是可见性，不是完整控制；如果部署链路不验证产物来源，SBOM 只是清单。
- 关联：[[./08-Playbooks/供应链安全评审 Playbook|供应链安全评审 Playbook]]

## 决策 18：SOC 先接更多日志，还是先定义检测场景？

- 判断标准：
  - 如果关键身份、云、应用、数据、终端日志缺失，先补最小日志源
  - 如果日志已经很多但告警无效，先按攻击场景设计检测
  - 如果告警无人处理，先补分诊、owner 和响应动作
- 高级判断：SOC 成熟度不是日志量，而是关键攻击路径是否能被看见、判断和响应。
- 关联：[[./08-Playbooks/SOC 检测工程 Playbook|SOC 检测工程 Playbook]]

## 决策 19：怎么判断自己真的掌握安全，而不是只看懂了？

- 判断标准：
  - 能解释术语，只是 L1
  - 能按 checklist 执行，是 L2
  - 能独立评审、响应和复盘，是 L3
  - 能设计流程、控制和指标，是 L4
  - 能在业务约束下做架构取舍并教别人，是 L5
- 高级判断：掌握程度要看产物，包括风险诊断、安全评审、事件复盘、检测规则、云基线、供应链评审、风险登记和管理层汇报。
- 关联：[[./07-Templates/安全能力自评表|安全能力自评表]]、[[./09-Case-Studies/案例复盘索引|案例复盘索引]]

## 决策 20：练安全时先打靶场，还是先做防御作品集？

- 判断标准：
  - 如果目标是理解漏洞原理，可以在授权靶场做局部练习
  - 如果目标是解决工作问题，优先做资产盘点、评审、基线、检测、响应和 GRC 作品集
  - 如果目标是面试，最好准备能讲清业务背景、风险判断、控制设计和验证结果的作品
- 高级判断：攻防技能有价值，但安全工程和安全架构岗位更看重你能否把风险降低、把控制落地、把证据讲清。
- 关联：[[./08-Playbooks/安全实战实验室路线 Playbook|安全实战实验室路线 Playbook]]、[[./10-Interview-Kit/安全面试专题总览|安全面试专题总览]]

## 决策 21：做红队、蓝队，还是紫队演练？

- 判断标准：
  - 如果目标是验证真实业务风险，可以做红队评估，但必须明确授权、范围、停止条件和证据规则
  - 如果目标是提升检测和响应，优先蓝队场景演练和 SOC 检测工程
  - 如果目标是快速补齐检测缺口，优先紫队演练，让攻击路径、日志、规则、分诊、响应同步迭代
- 高级判断：成熟组织不会把红队报告当终点，而会用紫队机制把发现转成检测规则、响应 playbook、控制改进和指标。
- 关联：[[./08-Playbooks/红队蓝队紫队演练路径 Playbook|红队蓝队紫队演练路径 Playbook]]

## 决策 22：单点工具优先，还是平台化优先？

- 判断标准：
  - 如果当前只有一个清晰痛点，例如 secret 泄露或依赖漏洞，先用单点工具跑通闭环
  - 如果问题跨身份、云、代码、日志、审计多个系统，优先设计平台集成
  - 如果组织流程还不清，先补 owner、SLA、例外和验证，再扩大工具范围
- 高级判断：平台化不是采购大套件，而是让资产、身份、风险、工单、检测、证据在同一条运营链路里流动。
- 关联：[[./06-Maps/安全工具平台能力地图|安全工具平台能力地图]]、[[./08-Playbooks/安全工具平台选型与落地 Playbook|安全工具平台选型与落地 Playbook]]

## 决策 23：自建、开源，还是采购商业平台？

- 判断标准：
  - 开源适合学习、透明可控和小范围自定义，但需要工程和运营能力
  - 商业平台适合快速覆盖、合规证据、跨系统集成和支持保障，但要防止锁定和成本黑洞
  - 自建适合核心差异化流程，例如风险排序、检测规则平台、内部证据流，但不适合重复造通用底座
- 高级判断：真正要比较的是总拥有成本，包括授权、数据量、维护人力、误报治理、集成开发和审计成本。
- 关联：[[./02-Tools/安全工具平台总览|安全工具平台总览]]、[[./07-Templates/安全工具选型评分表|安全工具选型评分表]]

## 决策 24：通用安全优先，还是行业控制优先？

- 判断标准：
  - 如果身份、资产、日志、漏洞、数据分类还很弱，先补通用安全主干
  - 如果已经进入支付、医疗、SaaS、游戏或 AI 产品场景，必须把控制映射到行业资产、监管、客户审计和业务风险
  - 如果目标是面试或架构表达，先讲通用主干，再讲行业差异，会更像专家
- 高级判断：行业安全不是新学一套概念，而是把通用能力迁移到行业资产、威胁、指标和证据里。
- 关联：[[./03-Industry-Controls/行业控制清单索引|行业控制清单索引]]、[[./06-Maps/行业安全控制迁移地图|行业安全控制迁移地图]]

## 决策 25：手游撞库时，先全局验证码，还是先风险分层？

- 判断标准：
  - 如果登录服务被打垮，先做入口限速、DDoS/Bot 防护和临时高风险挑战
  - 如果服务可用但攻击持续，优先做账号、IP、设备、ASN、地区、失败模式的风险分层
  - 如果已经有成功登录，优先保护换绑、改密、资产转移、充值退款和客服找回
  - 如果玩家体验敏感，不要全量强验证码，而要对高风险登录触发 step-up
- 高级判断：撞库防护不是“让攻击者无法尝试”，而是降低成功率、限制损失半径、缩短发现时间，并把规则沉淀进账号安全平台。
- 关联：[[./08-Playbooks/手游撞库与账号安全治理 Playbook|手游撞库与账号安全治理 Playbook]]、[[./08-Playbooks/手游账号安全检测规则库|手游账号安全检测规则库]]、[[./06-Maps/手机游戏端到端安全架构图|手机游戏端到端安全架构图]]、[[./06-Maps/游戏账号风控数据链路图|游戏账号风控数据链路图]]、[[./03-Industry-Controls/手机游戏端到端安全控制矩阵|手机游戏端到端安全控制矩阵]]
