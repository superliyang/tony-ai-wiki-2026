# 安全工具平台选型与落地 Playbook

> 用于回答：到底该上什么安全工具、怎么选、怎么接、怎么证明它不是摆设。

## 适用场景

- 公司准备采购 SIEM、CNAPP、SAST、DLP、GRC、PAM、Secrets 等平台。
- 已经买了工具，但告警没人管、修复不闭环、审计证据仍靠手工。
- 安全团队希望把能力从“人工专家”变成“平台默认能力”。

## Step 1：定义问题，而不是定义产品

先写清楚一句话：

`我们要降低什么风险，在什么资产范围内，由谁处理，用什么指标证明改善。`

示例：

- 不是“买 SIEM”，而是“让关键身份和云控制面攻击在 15 分钟内被发现并进入响应流程”。
- 不是“买 SAST”，而是“减少高危代码缺陷进入生产，并让开发者在 PR 阶段得到可修复建议”。
- 不是“买 GRC”，而是“把 SOC 2 / ISO 27001 控制证据从人工截图变成自动采集和到期复审”。

## Step 2：盘点前置条件

| 前置条件 | 需要确认什么 |
|---|---|
| 资产 | 系统、云账号、代码仓、数据源、终端、SaaS 是否有 owner |
| 身份 | 员工、外包、服务账号、机器身份是否统一 |
| 流程 | 工单、审批、例外、SLA、变更流程是否存在 |
| 数据源 | 日志、代码、云配置、漏洞、数据分类是否能接入 |
| 组织 | 谁分诊、谁修复、谁审批、谁验收 |
| 指标 | 用什么证明风险下降或控制有效 |

## Step 3：选择平台族

| 问题 | 优先平台 | 不宜先买的情况 |
|---|---|---|
| 账号和高权限混乱 | IAM / PAM / Secrets | 账号源和权限模型完全不清 |
| 漏洞进生产 | AppSec / DevSecOps | CI/CD 和代码 owner 不清 |
| 云资产暴露 | CNAPP / CSPM / CIEM | 云账号和标签体系不清 |
| 攻击不可见 | SIEM / SOAR / XDR | 最小日志源和分诊流程缺失 |
| 敏感数据不可见 | DSPM / DLP / Privacy | 数据地图和分类规则缺失 |
| 合规证据低效 | GRC / IRM | 控制库和控制 owner 不清 |

## Step 4：做 PoC，但只验证关键假设

PoC 不要变成“看 demo”。建议验证：

- 接入：关键数据源能否在 1-2 周内接入。
- 准确：能否发现你关心的真实风险，而不是堆告警。
- 修复：能否生成 owner、工单、建议和验证路径。
- 集成：能否接 SSO、工单、CI/CD、云、SIEM、GRC。
- 运营：安全团队和业务团队是否愿意用。
- 证据：是否能留下审计和管理层需要的证据。

## Step 5：用评分表做取舍

使用 [[../07-Templates/安全工具选型评分表|安全工具选型评分表]]。

建议权重：

- 问题匹配度：25%
- 集成能力：20%
- 运营闭环：20%
- 准确率与可解释性：15%
- 成本与复杂度：10%
- 证据与合规：10%

## Step 6：小范围上线闭环

先选一个业务线或资产域跑通：

1. 接入资产和数据源。
2. 定义前 10 个关键规则或控制。
3. 绑定 owner、SLA、工单和例外。
4. 每周复盘误报、漏报、修复阻塞。
5. 产出管理层指标和审计证据。

## Step 7：平台化扩展

当小闭环跑通后，再扩展：

- 从一个云账号扩到多账号。
- 从一个代码仓扩到核心代码仓。
- 从一类日志扩到关键日志矩阵。
- 从人工证据扩到自动证据。
- 从安全团队操作扩到工程平台默认能力。

## 验收清单

- 是否有清晰的问题陈述和成功指标？
- 是否有资产范围、owner 和例外机制？
- 是否接入 SSO、工单、CI/CD、云、SIEM 或 GRC 中至少两个关键系统？
- 是否能从发现走到修复验证，而不是只生成报告？
- 是否能输出趋势指标和审计证据？
- 是否有 30/60/90 天落地计划？

## 30/60/90 天计划

| 时间 | 目标 | 交付物 |
|---|---|---|
| 30 天 | 跑通一个最小闭环 | 接入清单、规则、owner、工单、指标 |
| 60 天 | 扩展到核心资产 | 覆盖率、SLA、例外、误报治理 |
| 90 天 | 进入平台化运营 | 自动证据、管理看板、复盘机制 |
