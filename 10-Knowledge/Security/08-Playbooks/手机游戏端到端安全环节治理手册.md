# 手机游戏端到端安全环节治理手册

> 这份手册回答：手机游戏从拉新、注册、登录、客户端、API、战斗、结算、经济、支付、客服、运营后台、数据平台到安全运营，每个环节会有什么问题、应该具体做什么、先做什么后做什么、如何演进成熟。

## 总体判断

手游安全不是一个“安全组件”问题，而是一条业务链路的韧性问题：

`获客注册 -> 登录会话 -> 账号找回 -> 客户端可信 -> API 服务 -> 游戏行为 -> 战斗结算 -> 虚拟经济 -> 支付退款 -> 社交内容 -> 客服运营 -> 数据后台 -> CI/CD 供应链 -> 安全运营`

任何一段失守，都会沿着链路放大：

- 撞库成功不是终点，后面会接换绑、资产转移、退款、客服申诉。
- 客户端被绕过不是终点，后面会接自动化、外挂、刷资源和黑市交易。
- GM 后台越权不是一个内部小问题，它可以绕过前面所有技术防线。
- 没有统一日志和证据链，安全团队只能“怀疑”，不能证明、处置和复盘。

## 端到端分层总览

| 环节 | 保护对象 | 常见问题 | 第一优先级 |
|---|---|---|---|
| 1. 获客与注册 | 新账号、渠道、活动成本 | 批量注册、羊毛党、设备农场 | 注册限速、设备画像、渠道校验 |
| 2. 登录与认证 | 账号控制权 | 撞库、密码喷洒、代理池 | 多维限速、风险验证、泄露密码治理 |
| 3. 会话与设备 | session、设备绑定 | token 泄露、异常会话、设备接管 | 短 token、刷新轮换、session revoke |
| 4. 找回与换绑 | 账号归属 | 社工、盗号后换绑、客服滥用 | 冷却期、证据分级、旧设备通知 |
| 5. 客户端完整性 | App、设备、资源包 | 改包、Hook、模拟器、Root/越狱 | 完整性信号服务端验证、关键逻辑服务端化 |
| 6. API 与网关 | 业务接口、对象权限 | 越权、重放、参数篡改、滥用 | 对象级授权、幂等、nonce、限速 |
| 7. 战斗与反作弊 | 公平性、段位、排行榜 | 加速、内存修改、封包篡改 | 服务端权威、回放校验、行为异常 |
| 8. 结算与奖励 | 奖励、任务、活动 | 重复发奖、伪造结果、活动套利 | 结算幂等、奖励账本、异常产出 |
| 9. 虚拟经济 | 货币、道具、交易 | 刷币、搬砖、盗卖、黑市 | 账本化、流向分析、风险冻结 |
| 10. 支付与退款 | 订单、发货、退款 | 伪造回调、重复发货、拒付套利 | 验签、幂等、对账、退款联动 |
| 11. 社交与内容 | 聊天、公会、UGC | 诈骗、广告、未成年人风险 | 内容审核、举报、账号画像联动 |
| 12. 客服与运营 | 工单、GM、活动 | 社工、越权、误操作、内部滥用 | RBAC、审批、双人复核、审计 |
| 13. 数据与隐私 | PII、画像、日志 | 越权查询、泄露、合规缺口 | 分类分级、脱敏、查询审计 |
| 14. CI/CD 与供应链 | 代码、SDK、热更、签名 | SDK 投毒、密钥泄露、恶意制品 | SCA、Secrets、SBOM、签名发布 |
| 15. 安全运营 | 日志、规则、Case | 看不见、响应慢、误伤不可控 | 统一信号、规则库、Case、复盘 |

## 1. 获客、渠道与注册

### 会有什么问题

- 黑产用设备农场、云手机、模拟器批量注册账号。
- 活动拉新被羊毛党批量领取，真实获客成本被放大。
- 渠道包、广告归因、邀请码被滥用或伪造。
- 手机号、邮箱、第三方登录被批量化、低成本化。
- 小号池长期潜伏，后续用于刷榜、搬砖、诈骗、撞库中转。

### 详细可以做什么

- 注册限速：按 IP、设备、ASN、手机号段、渠道、邀请人、地理位置组合限速。
- 设备画像：记录设备指纹、安装来源、模拟器、Root/越狱、完整性信号。
- 渠道校验：渠道包签名、归因参数校验、异常渠道比例监控。
- 活动风控：新人奖励延迟到账，低信誉账号限制转移和交易。
- 邀请关系图：识别一个邀请人拉大量低质量账号、同设备批量账号。
- 号码风险：识别虚拟号、接码平台、短时间多账号绑定同类号码。

### 解决步骤

1. 先统计注册来源、设备分布、Top IP/ASN、渠道异常占比。
2. 对明显异常来源做限速和验证码，不要先全量打扰正常用户。
3. 把注册奖励和交易能力解耦：新账号先可玩，但高价值资产转移延迟。
4. 把注册、活动、交易、登录数据打通，看小号是否进入黑产链路。
5. 对活动上线建立安全评审：奖励是否可搬运、是否可套利、是否有回滚手段。

### 演进路线

- L1：IP/设备限速。
- L2：渠道、设备、手机号、邀请关系联合规则。
- L3：注册信誉分影响活动奖励、交易和社交能力。
- L4：活动上线前有风控评审、灰度、监控和复盘机制。

## 2. 登录、认证与撞库

### 会有什么问题

- 外部泄露密码被黑产拿来批量尝试。
- 密码喷洒用少量常见密码打大量账号。
- 代理池、IDC、住宅代理绕过单 IP 限速。
- 撞库成功后立刻换绑、改密、资产转移、退款或客服申诉。
- 全局验证码误伤正常用户，导致登录转化下降。

### 详细可以做什么

- 多维限速：账号、IP、设备、ASN、国家地区、失败模式、App 版本。
- 风险验证码：只对高风险触发，不做粗暴全量验证码。
- 泄露密码治理：注册、登录、改密时检测弱密码和已泄露密码。
- Step-up：新设备、高价值账号、异常地区、登录后敏感动作触发二次验证。
- Session revoke：发现账号接管时撤销可疑 session 和同源设备 session。
- 用户通知：新设备登录、换绑、改密、敏感资产操作通知用户。

### 解决步骤

1. 0-2 小时：拉登录失败量、成功量、失败/成功比、Top IP/ASN/设备/地区。
2. 0-2 小时：对高失败率来源做多维限速和指数退避。
3. 2-24 小时：找出成功登录后的敏感动作链路。
4. 24-72 小时：上线高风险登录 step-up、可疑 session 撤销、敏感动作冷却。
5. 7-30 天：建立账号风险评分和账号安全中心。

### 演进路线

- L1：IP 限速、基础验证码。
- L2：账号/IP/设备/ASN 联合限速。
- L3：登录风险影响换绑、交易、退款、客服。
- L4：账号风险平台化，支持策略配置、Case 管理、误伤复核和指标看板。

## 3. 会话、Token 与设备管理

### 会有什么问题

- access token 生命周期过长，被窃取后可长期使用。
- refresh token 不轮换，被复制后长期有效。
- 设备列表不可见，用户不知道哪些设备登录过。
- 风险登录后无法撤销 session，只能改密码。
- 同账号多设备、模拟器、脚本端共存，难以判断真实玩家。

### 详细可以做什么

- access token 短时效，refresh token 轮换和绑定设备。
- session 绑定设备、App 版本、风险信号和最近活动。
- 用户可查看设备列表并远程下线。
- 后端支持按账号、设备、风险来源批量 revoke。
- 高风险登录后对旧设备通知，新设备敏感动作冷却。
- 对异常 session 增加 trace id，串联后续资产动作。

### 解决步骤

1. 盘点 token 类型、有效期、是否可撤销、是否绑定设备。
2. 先给高危场景加 revoke 能力：盗号、换绑、改密、封禁、客服找回。
3. 再建设设备列表和用户自助下线。
4. 最后把 session 风险接入风控策略引擎。

### 演进路线

- L1：token 有过期时间。
- L2：refresh token 轮换，支持撤销。
- L3：session 与设备、风险、敏感动作联动。
- L4：会话治理成为账号安全平台的一部分。

## 4. 找回、换绑、改密与客服申诉

### 会有什么问题

- 黑产撞库成功后立刻换绑邮箱/手机号。
- 客服被社工，错误帮助攻击者找回账号。
- 找回材料被重复使用，批量接管多个账号。
- 高价值账号找回缺少更高证据标准。
- 找回成功后立即转移资产，导致用户不可恢复。

### 详细可以做什么

- 证据强度分级：设备历史、充值记录、实名信息、长期行为、旧绑定信息。
- 冷却期：找回、换绑、改密后限制交易、赠送、退款、大额消耗。
- 旧设备/旧绑定通知：给原持有人冻结入口。
- 高风险工单升级：客服不能单人完成高价值账号接管。
- 工单风控：同设备/IP/材料提交多个找回请求触发告警。
- 全量审计：客服操作、审批、证据、结果必须可追溯。

### 解决步骤

1. 立即禁止高风险登录后无验证换绑。
2. 给找回后账号设置敏感资产冷却。
3. 梳理客服权限，拆分查询、验证、改绑、补偿、解封。
4. 建立找回证据评分和升级审核。
5. 把客服工单接入账号风险和经济风险。

### 演进路线

- L1：换绑改密需要二次验证。
- L2：找回后冷却，客服高危操作审计。
- L3：工单风险评分和多维证据分级。
- L4：客服、账号、设备、资产、支付全链路联动。

## 5. 客户端完整性与设备可信

### 会有什么问题

- APK/IPA 被改包、二次打包、注入恶意模块。
- Hook、Frida、Xposed、越狱/Root 绕过本地逻辑。
- 模拟器、云手机、脚本批量自动化。
- 热更资源、配置、脚本被篡改。
- 客户端存储敏感密钥、接口签名逻辑被逆向。

### 详细可以做什么

- Android 接入 Play Integrity，服务端验证 token，把 verdict 作为风险信号。
- iOS 使用 App Attest / DeviceCheck，为敏感接口提供 App 与设备相关信号。
- 资源包、热更、配置签名校验，异常版本进入隔离策略。
- 本地敏感数据最小化，不在客户端保存长期密钥。
- RASP/反调试/反 Hook 用于提高成本，但不把客户端当最终可信根。
- 模拟器和完整性失败不一定一刀切封禁，而是分层影响登录、交易、战斗、奖励。

### 解决步骤

1. 先确认关键业务是否依赖客户端本地判断。
2. 把战斗、奖励、库存、支付、活动规则迁移到服务端权威。
3. 接入完整性信号并服务端验证，进入风控画像。
4. 对异常设备做分层策略：观察、限权、隔离、复核、封禁。
5. 建立客户端版本、渠道、热更、资源 hash 监控。

### 演进路线

- L1：签名校验和基础加固。
- L2：完整性 verdict 服务端验证。
- L3：完整性、设备、行为、经济风险联动。
- L4：客户端风险服务平台化，支持策略、监控、样本和演练。

## 6. API、网关与服务端授权

### 会有什么问题

- Broken Object Level Authorization：改 `role_id/order_id/item_id` 操作他人资产。
- 重放请求导致重复领取、重复发货、重复结算。
- 参数篡改绕过价格、数量、奖励、冷却时间。
- API 缺少限速，被脚本刷接口。
- 只信客户端校验，服务端没有二次验证。

### 详细可以做什么

- 对象级授权：每个对象操作都校验账号、角色、服务器、租户或区服关系。
- 幂等：订单、奖励、任务、结算、发货、补偿都必须有幂等键。
- nonce/timestamp：高风险接口防重放。
- schema 校验：类型、范围、枚举、状态机校验。
- 网关限速：账号、IP、设备、接口、业务动作组合限速。
- trace id：从登录、API、经济、支付、客服贯穿到底。

### 解决步骤

1. 列出高危 API：登录、换绑、交易、奖励、支付、GM、客服。
2. 逐个检查对象级授权和状态机校验。
3. 给发货、奖励、补偿、退款补幂等。
4. 接入 API 日志、限速和异常行为检测。
5. 把 API 安全评审纳入上线门禁。

### 演进路线

- L1：基础鉴权和参数校验。
- L2：对象级授权、幂等、限速。
- L3：业务状态机和攻击行为检测。
- L4：API 安全门禁、自动化测试和风险指标闭环。

## 7. 战斗、匹配、排行榜与反作弊

### 会有什么问题

- 加速、内存修改、封包篡改、脚本操作。
- 客户端伪造战斗结果或关键状态。
- 排行榜刷分、段位异常、匹配操控。
- 误封影响高价值玩家体验。
- 作弊与虚拟经济联动，产出可变现资产。

### 详细可以做什么

- 服务端权威：关键状态、结算、奖励由服务端计算或验证。
- 回放校验：保留关键输入、时间线、战斗摘要，用于复核。
- 行为模型：速度、命中率、操作间隔、异常路径、战斗统计分布。
- 排行榜延迟发布：高风险成绩先进入复核。
- 封禁分级：观察、影子匹配、限制收益、临时封禁、永久封禁。
- 误伤复核：保留证据，提供申诉和回滚策略。

### 解决步骤

1. 确认哪些战斗逻辑仍由客户端决定。
2. 对高价值结算链路做服务端验证或回放抽检。
3. 建立作弊样本库和正常玩家基线。
4. 上线排行榜延迟发布和高风险复核。
5. 把作弊收益接入经济风险和封禁复盘。

### 演进路线

- L1：基础外挂检测和人工封禁。
- L2：服务端校验关键结算。
- L3：行为模型、回放复核、误伤流程。
- L4：反作弊、经济风控、账号风控联动对抗。

## 8. 结算、任务、活动与奖励

### 会有什么问题

- 重复请求导致重复领奖。
- 活动规则被脚本刷爆。
- 状态机缺口导致绕过前置条件领奖。
- 奖励配置错误造成经济事故。
- 活动结束后接口仍可调用。

### 详细可以做什么

- 奖励发放幂等：同一任务、同一活动、同一订单只能发一次。
- 状态机校验：资格、进度、冷却、领取状态、活动窗口。
- 活动风控评审：奖励是否可转移、是否可交易、是否可退款套利。
- 灰度与限额：新活动先小流量监控。
- 异常产出检测：单位时间货币/道具产出超过基线告警。

### 解决步骤

1. 梳理所有奖励入口和幂等键。
2. 对历史事故最多的活动先补状态机和风控规则。
3. 建立活动上线 checklist。
4. 对奖励配置做审批、灰度、回滚和监控。
5. 将异常产出接入经济风控。

### 演进路线

- L1：奖励接口幂等。
- L2：活动上线安全评审。
- L3：异常产出实时监控。
- L4：活动风控与账号/设备/经济画像联动。

## 9. 虚拟经济、交易与库存账本

### 会有什么问题

- 盗号后道具和货币快速转移。
- 外挂或活动漏洞造成资源异常产出。
- 多账号搬砖形成黑市供应链。
- 退款后虚拟资产未扣回。
- 缺少账本，事故后无法回滚和证明。

### 详细可以做什么

- 账本化：所有货币、道具、充值、消耗、赠送、交易、补偿可追溯。
- 风险冻结：高风险登录后限制交易、赠送、出售、大额消耗。
- 资产流向图：识别一对多、多对一、环状交易、黑市中转账号。
- 经济健康指标：产出、消耗、存量、集中度、异常流入流出。
- 回滚机制：误发、盗号、刷币后能按 trace 回滚。

### 解决步骤

1. 确认核心资产是否有完整账本。
2. 建立高风险登录后的资产保护策略。
3. 建立异常产出、异常转移、异常消耗规则。
4. 把交易、赠送、拍卖、邮件、背包接入同一经济视图。
5. 建立经济事故复盘模板。

### 演进路线

- L1：道具和货币有流水。
- L2：核心资产账本化，支持追溯。
- L3：风险冻结和资产流向分析。
- L4：经济风控平台化，联动账号、支付、客服和反作弊。

## 10. 支付、订单、退款与拒付

### 会有什么问题

- 伪造渠道回调或重放回调。
- 订单重复发货。
- 退款后道具没有扣回。
- 拒付、退款套利与盗号联动。
- 渠道对账差异长期无人处理。

### 详细可以做什么

- 渠道验签：所有支付回调服务端验签，不信客户端回传。
- 订单幂等：同一订单只能发货一次。
- 对账闭环：支付、发货、退款、拒付、扣回状态一致。
- 退款风控：高风险登录后退款进入人工审核。
- 资产联动：退款、拒付触发道具扣回、账号限制或 Case。

### 解决步骤

1. 检查所有支付渠道回调验签和证书管理。
2. 检查发货幂等和异常补单流程。
3. 打通订单、发货、库存、退款、拒付对账。
4. 对高风险账号的退款和资产转移加审核。
5. 建立支付风险看板和渠道异常日报。

### 演进路线

- L1：验签和订单幂等。
- L2：支付对账和退款扣回。
- L3：支付风险联动账号、设备、经济。
- L4：支付风控与黑产情报、Case 管理、合规证据联动。

## 11. 社交、聊天、公会与内容安全

### 会有什么问题

- 诈骗、广告、导流、赌博、外挂售卖。
- 黑产用公会和私聊做账号交易或洗资产。
- 未成年人保护、内容合规和举报处理压力。
- 被盗号账号用于发诈骗消息。

### 详细可以做什么

- 内容审核：文本、图片、昵称、公会名、签名、UGC。
- 举报机制：举报到处置、反馈、复核闭环。
- 社交限权：低信誉账号限制私聊、公会邀请、链接发送。
- 账号风险联动：被盗风险账号限制社交传播。
- 黑产图谱：社交关系、交易关系、设备关系、支付关系联动。

### 解决步骤

1. 标准化聊天、公会、举报日志。
2. 对低信誉新号和风险登录账号限制高传播动作。
3. 建立诈骗关键词、链接、联系方式检测。
4. 将举报与账号、设备、经济风险打通。
5. 建立内容安全和账号安全联合处置流程。

### 演进路线

- L1：关键词和举报。
- L2：账号信誉与社交限权。
- L3：诈骗链路和黑产关系图。
- L4：内容安全、账号安全、经济风控联合作战。

## 12. GM 后台、客服、活动运营与内部权限

### 会有什么问题

- 客服被社工后帮助黑产改绑、找回、解封。
- GM 权限过大，单人可发道具、补偿、封禁、解封。
- 运营活动配置错误造成经济事故。
- 内部账号被盗后大规模操作生产数据。
- 查询玩家 PII 缺少审计。

### 详细可以做什么

- MFA：后台、客服、GM、数据平台强制 MFA。
- RBAC/PAM：角色拆细，临时授权，按需提权。
- 高危审批：发道具、补偿、解封、改绑、批量查询、导出必须审批。
- 双人复核：高价值账号、高价值道具、大批量操作。
- 全量审计：operator、target、before/after、approval、reason、ticket。
- 异常检测：非工作时段、批量操作、越权操作、异常补偿。

### 解决步骤

1. 盘点后台系统和高危操作。
2. 禁止共享账号，强制 MFA 和最小权限。
3. 给高危操作加审批、双人复核和审计。
4. 把客服工单、GM 操作、账号风险、经济账本关联。
5. 建立内部权限月度 review 和离职回收。

### 演进路线

- L1：后台账号实名和 MFA。
- L2：RBAC、审批、审计。
- L3：PAM、临时权限、异常操作检测。
- L4：内部操作风险和业务风控联动，支持审计证明。

## 13. 数据、隐私与合规

### 会有什么问题

- 玩家 PII、未成年人信息、聊天、行为画像被越权查询或导出。
- 日志中记录明文密码、token、手机号、身份证等敏感信息。
- 数据留存过长，删除和导出请求无法响应。
- 跨地区运营涉及数据驻留、跨境、通知和监管要求。
- 风控画像滥用，缺少最小必要和访问审计。

### 详细可以做什么

- 数据分类分级：PII、未成年人、支付、账号安全、风控画像分级管理。
- 最小化采集：只采业务和安全必要字段。
- 加密与脱敏：存储加密，展示脱敏，导出审批。
- 查询审计：谁查了谁、为什么查、查了什么、是否有工单。
- 保留删除：日志、画像、工单、聊天按规则留存和删除。
- 合规证据：数据地图、处理目的、访问控制、事件响应、第三方 SDK。

### 解决步骤

1. 先做数据地图：账号、支付、聊天、行为、设备、客服、日志。
2. 识别敏感字段和高风险查询入口。
3. 对后台展示、导出、分析平台做脱敏和审批。
4. 建立数据访问审计和异常导出告警。
5. 与法务确定未成年人、地区合规、隐私政策和事件通知要求。

### 演进路线

- L1：敏感字段脱敏和权限控制。
- L2：数据地图、分类分级、查询审计。
- L3：DSPM/DLP/Privacy 流程化。
- L4：隐私治理与安全运营、GRC 证据自动化联动。

## 14. CI/CD、SDK、热更与供应链

### 会有什么问题

- 第三方 SDK 收集过多数据或存在漏洞。
- CI/CD secret 泄露导致制品、云账号、签名证书被滥用。
- 热更资源被篡改，绕过应用商店审核。
- 依赖漏洞、恶意包、构建环境污染进入生产。
- 渠道包管理混乱，灰产利用旧版本或异常包。

### 详细可以做什么

- SAST/SCA/Secrets：代码、依赖、密钥扫描进入 CI。
- SBOM：核心客户端、服务端、镜像、SDK 形成软件物料清单。
- 签名发布：构建产物、镜像、热更资源签名和校验。
- 制品库权限：谁能发版、谁能回滚、谁能改配置。
- SDK 评审：权限、数据收集、网络域名、版本、供应商安全证明。
- 热更门禁：审批、灰度、回滚、完整性校验和日志。

### 解决步骤

1. 盘点代码仓、CI/CD、制品库、签名证书、热更系统、第三方 SDK。
2. 先上线 secrets 扫描和高危依赖扫描。
3. 给生产发布和热更加审批、签名和回滚。
4. 给 SDK 接入建立安全和隐私评审。
5. 逐步建设 SBOM、制品签名和供应链证据。

### 演进路线

- L1：扫描发现 secret 和高危漏洞。
- L2：发布审批、签名、制品权限。
- L3：SBOM、SLSA 思路、供应商评估。
- L4：供应链风险进入 release gate 和 GRC 证据。

## 15. 日志、检测、响应与复盘

### 会有什么问题

- 登录、设备、经济、客服、支付、后台日志分散，无法串联攻击链。
- 告警只有技术字段，没有账号、资产、订单、工单上下文。
- 规则上线后误伤不可控，没有 shadow、灰度和回归。
- 事件结束后没有沉淀新规则、新控制和新流程。
- 领导只看到事故，没有看到风险下降和能力提升。

### 详细可以做什么

- 统一事件模型：account、role、device、ip、session、order、ticket、operator、trace。
- 检测规则库：撞库、ATO、资产搬运、客服社工、退款套利、GM 滥用、外挂。
- Case 管理：告警、证据、处置、复核、申诉、复盘。
- 处置编排：限速、验证码、session revoke、冻结、人工审核、通知。
- 指标看板：攻击规模、成功率、损失、误伤、MTTD、MTTR、规则覆盖率。
- 复盘闭环：每次事故产出控制改进、规则回归、上线门禁和演练场景。

### 解决步骤

1. 先定义最小统一字段，不追求一步到位数据湖。
2. 建立 P0/P1 检测规则：撞库、成功登录后敏感动作、资产搬运。
3. 建 Case 流程：谁分诊、谁处置、谁复核、谁对外沟通。
4. 对规则做 shadow、灰度、误伤复核和回归。
5. 每周复盘 top 风险、top 规则、top 绕过、top 误伤。

### 演进路线

- L1：有日志和人工查询。
- L2：有核心检测规则和告警分诊。
- L3：规则、Case、处置动作、误伤复核闭环。
- L4：检测工程、红蓝紫演练、业务风控和 GRC 证据联动。

## 横向落地步骤

### 第 0 步：明确安全目标

不要用“绝对安全”做目标。手游安全更现实的目标是：

- 降低撞库成功率。
- 限制账号接管后的损失半径。
- 维护游戏公平性。
- 保护虚拟经济稳定。
- 降低支付和退款损失。
- 让客服和运营不成为绕过通道。
- 缩短发现和处置时间。
- 留下可审计、可复盘、可汇报的证据。

### 第 1 步：建立资产和链路清单

最少列清：

- 核心账号链路：注册、登录、会话、找回、换绑、改密。
- 核心资产链路：角色、货币、道具、交易、赠送、库存。
- 核心资金链路：订单、渠道回调、发货、退款、拒付、对账。
- 核心权限链路：客服、GM、活动、数据、发布、热更。
- 核心日志链路：登录、设备、API、经济、支付、客服、后台。

### 第 2 步：先止血高损失链路

优先做：

- 多维登录限速。
- 高风险登录 step-up。
- 可疑 session 撤销。
- 换绑/找回/资产转移冷却。
- 高风险退款人工审核。
- 客服和 GM 高危操作审计。

### 第 3 步：补数据关联

统一关联键：

- `account_id`
- `role_id`
- `device_id`
- `ip / asn / geo`
- `session_id`
- `trace_id`
- `order_id`
- `item_id / currency_id`
- `ticket_id`
- `operator_id`

没有这些键，端到端安全只能靠猜。

### 第 4 步：建立策略与动作分层

风险动作不要一刀切：

- 记录：低风险、低置信度。
- 限速：自动化尝试、撞库扫描。
- Step-up：新设备、高价值账号、敏感动作。
- 冻结：高风险登录后资产转移。
- 人审：证据不充分但影响大。
- 封禁/回滚：确认恶意且证据完整。

### 第 5 步：沉淀平台能力

长期目标是四个平台：

1. 账号安全平台：登录、会话、MFA、找回、换绑、设备列表。
2. 游戏风控平台：反作弊、经济风控、活动风控、支付风控。
3. 内部权限平台：GM、客服、数据、发布、审批、审计。
4. 安全运营平台：日志、检测、Case、处置、复盘、指标。

## 30 / 60 / 90 天演进

### 30 天：能止血、能看见、能处置

- 完成撞库止血：限速、step-up、session revoke。
- 完成敏感动作保护：换绑、交易、退款、客服找回冷却和复核。
- 打通最小日志链路：登录、设备、账号变更、经济、支付、客服、后台。
- 上线第一批检测规则：撞库、ATO、资产搬运、客服社工、GM 异常。
- 建立领导大盘：攻击规模、成功率、损失、误伤、处置进展。

### 60 天：能联动、能复盘、能平台化

- 账号、设备、经济、支付、客服风险联动。
- 客户端完整性信号进入服务端策略。
- 经济账本和资产流向分析可用。
- 后台高危操作审批和审计闭环。
- 检测规则进入 shadow、灰度、回归流程。
- 形成周度安全复盘和业务安全 backlog。

### 90 天：能对抗、能治理、能证明

- 建立账号安全中心、经济风控、客户端风险服务、Case 管理。
- 安全评审进入活动上线、版本发布、SDK 接入、经济规则变更。
- 组织一次桌面演练：撞库到资产搬运、外挂到刷币、GM 滥用、退款套利。
- 对关键控制形成证据：owner、策略、日志、审批、指标、复盘。
- 用成熟度模型评估 L0-L4，并给下一季度安全路线图。

## 成熟度模型

| 等级 | 状态 | 典型表现 |
|---|---|---|
| L0 被动救火 | 攻击来了靠人工查日志 | 只知道登录失败多，不知道成功后发生什么 |
| L1 单点控制 | 有验证码、限速、客户端加固 | 各系统不联动，黑产换入口就绕过 |
| L2 链路联动 | 登录风险影响换绑、资产、退款、客服 | 能限制损失半径，有基础规则库 |
| L3 平台化 | 账号、设备、经济、后台、支付统一风控 | 策略可配置，Case 可追踪，指标可汇报 |
| L4 对抗运营 | 持续红蓝紫、样本、情报、规则回归 | 能稳定降低成功率、损失和响应时间 |

## 和官方标准怎么对齐

- 移动端：用 OWASP MASVS 对齐客户端存储、网络、平台交互、代码质量、反篡改和隐私要求。
- 应用/API：用 OWASP ASVS 和 OWASP API Security Top 10 对齐认证、授权、会话、输入校验、业务逻辑和 API 滥用。
- 撞库：用 OWASP Credential Stuffing Prevention Cheat Sheet 对齐多因素、限速、风险验证、泄露密码治理和监控。
- 自动化滥用：用 OWASP Automated Threats 分类梳理批量注册、撞库、刷接口、内容滥用和业务套利。
- 移动攻击技术：用 MITRE ATT&CK Mobile 建攻击技术到检测规则的覆盖矩阵。
- 软件供应链：用 NIST SSDF 对齐安全开发、依赖、构建、发布和漏洞响应。
- 安全治理：用 NIST CSF 2.0 的 Govern、Identify、Protect、Detect、Respond、Recover 梳理组织闭环。
