# 应用与 API 安全评审 Playbook

> 用途：评审一个新系统、新 API、重大变更、第三方开放接口或高风险业务逻辑。

## 一句话

应用/API 安全评审不是找几个漏洞名，而是确认：

`身份可信 -> 权限正确 -> 输入可控 -> 数据最小化 -> 业务不可滥用 -> 可检测可追责`

## 适用场景

- 新系统上线
- 新 API 对外开放
- 管理后台、支付、资金、权限、数据导出等高风险功能变更
- 第三方回调、Webhook、开放平台、SDK 接入
- AI agent、自动化工具或内部平台获得操作权限

## 评审输入

评审前至少要拿到：

- 系统用途和业务 owner
- 架构图和数据流
- API 列表和鉴权方式
- 角色与权限模型
- 敏感数据字段
- 第三方依赖和调用关系
- 日志、告警、审计方案
- 上线计划和回滚方案

## 核心检查面

### 1. 身份认证

要问：

- 用户、员工、服务、第三方分别怎么认证？
- token 生命周期、刷新、撤销和绑定策略是什么？
- 是否支持 MFA 或高风险操作二次验证？
- API key / secret 是否有 owner、过期、轮换、权限范围？

常见问题：

- 长期 token 不过期
- service account 权限过大
- 仅靠客户端参数判断身份
- 第三方回调没有签名校验

### 2. 授权与越权

要问：

- 每个 API 的资源 owner 如何判断？
- RBAC / ABAC 权限在哪里执行？
- 是否区分 horizontal privilege escalation 和 vertical privilege escalation？
- 管理端、租户边界、组织边界是否有统一校验？

高风险信号：

- 前端隐藏按钮代替后端授权
- 只校验登录，不校验资源归属
- 多租户系统没有 tenant isolation
- 管理接口和用户接口混用权限模型

### 3. 输入、输出与注入

要问：

- 所有外部输入是否有 schema validation？
- 文件上传、URL、回调地址、模板、表达式是否受限？
- 输出是否做 encoding 和敏感字段过滤？
- 错误信息是否泄露堆栈、SQL、token 或内部路径？

关注风险：

- SQL injection
- XSS
- SSRF
- command injection
- insecure deserialization
- path traversal

### 4. 业务逻辑滥用

安全评审不能只看技术漏洞。

要问：

- 是否可批量注册、撞库、刷接口、薅羊毛？
- 是否可绕过支付、优惠、库存、风控、审批？
- 是否有 rate limit、quota、behavior detection？
- 是否有关键动作幂等、重放保护和风控信号？

### 5. 数据安全

要问：

- API 是否返回过多字段？
- 日志是否记录敏感数据？
- 数据导出是否需要审批、脱敏、审计？
- 缓存、搜索索引、分析系统是否复制敏感数据？
- 删除、保留、访问、更正等隐私请求能否落地？

关联：[[./数据安全与隐私评审 Playbook|数据安全与隐私评审 Playbook]]

### 6. 依赖与供应链

要问：

- 依赖是否做 SCA？
- 镜像和 artifact 是否来源可信？
- CI/CD secret 是否暴露给构建脚本、PR、第三方 action？
- 第三方 SDK 是否收集数据或引入权限风险？

### 7. 可观测与响应

要问：

- 关键操作是否有审计日志？
- 登录失败、权限拒绝、数据导出、管理操作是否可查询？
- 异常行为是否有告警？
- 出事后能否定位账号、IP、设备、请求 ID、数据范围？

## Release Gate

建议阻断上线的条件：

- 未定义 owner 和回滚方案
- 高权限接口无后端授权
- 多租户边界未验证
- 敏感数据明文暴露或过度返回
- 第三方回调无签名校验
- API key / secret 无轮换和审计
- 关键日志缺失，出事后无法追溯

可以带风险接受上线的条件：

- 风险可解释
- 有补偿控制
- 有 owner
- 有截止时间
- 有复查机制
