# 面试决策导航

## 决策 -1：开放题先从模型名讲，还是先从全景栈讲？

- 适用场景：被问“你怎么看大模型 / AI 系统 / AI 领域面试”这类开放题。
- 判断标准：
  - 高级岗位优先从 `能力形成 -> 工程系统 -> 评测治理 -> 业务结果` 的全景栈讲。
  - 只有在岗位明确偏研究或底层训练时，才先把模型架构和训练算法放到更前面。
- 关联题库：[[./05-Question-Banks/14-大模型与 AI 全景题库|大模型与 AI 全景题库]]
- 回答母版：[[./06-Answer-Playbooks/大模型与 AI 全景回答母版|大模型与 AI 全景回答母版]]

## 决策 0：自我介绍先讲技术栈，还是先讲结果？

- 适用场景：开场 1-2 分钟自我介绍。
- 判断标准：
  - 高级岗位优先讲 `主线 + 代表结果 + 你的独特价值`，技术栈放到后面点到为止。
  - 只有在岗位非常偏底层研究时，才把技术方向放到更前面。
- 关联题库：[[./05-Question-Banks/00-开场与项目深挖题库|开场与项目深挖题库]]

## 决策 1：是继续堆 Prompt，还是升级到 Harness/Eval 体系？

- 适用场景：回答质量波动大、上线后频繁回归。
- 判断标准：
  - 若失败模式可枚举且可回归，优先升级 Eval/Harness。
  - 若需求还在探索，先保留轻量 Prompt 迭代。
- 关联题库：[[./05-Question-Banks/05-评测与上线治理题库|评测与上线治理题库]]

## 决策 2：长上下文 vs RAG，优先做哪个？

- 适用场景：多轮对话、文档问答质量下降。
- 判断标准：
  - 若核心问题是“记住历史上下文”，优先长上下文策略。
  - 若核心问题是“知识更新快且体量大”，优先 RAG + 改写 + rerank。
- 关联题库：[[./05-Question-Banks/02-推理与系统架构题库|推理与系统架构题库]]、[[./05-Question-Banks/04-RAG与数据工程题库|RAG与数据工程题库]]

## 决策 3：单 Agent 还是多 Agent？

- 适用场景：任务复杂、步骤依赖多、工具类型多。
- 判断标准：
  - 若任务可线性执行，优先单 Agent 降复杂度。
  - 若有明显并行子任务和专业化收益，再上多 Agent。
- 关联题库：[[./05-Question-Banks/03-Agent与工具调用题库|Agent与工具调用题库]]

## 决策 4：安全优先导致体验下降，怎么权衡？

- 适用场景：高风险业务（金融、医疗、客服自动外发）。
- 判断标准：
  - 高风险动作强制 HITL 与审批门禁。
  - 低风险动作采用自动化策略与抽样复核。
- 关联题库：[[./05-Question-Banks/06-安全与合规题库|安全与合规题库]]

## 决策 5：面试里先讲成功结果，还是先讲风险与事故？

- 适用场景：项目深挖、上线治理、资深岗位追问。
- 判断标准：
  - 先用 1 句话交代结果，证明项目不是失败样本。
  - 立刻补上风险、事故和治理动作，证明你不是只会讲喜报。
- 关联题库：[[./05-Question-Banks/00-开场与项目深挖题库|开场与项目深挖题库]]、[[./05-Question-Banks/07-团队与业务沟通题库|团队与业务沟通题库]]

## 决策 6：面试设计题时先讲组件，还是先讲业务目标与边界？

- 适用场景：系统设计题、场景题、开放架构题。
- 判断标准：
  - 先讲业务目标、风险边界和成功标准，再讲组件和链路。
  - 如果一上来就罗列组件，容易显得像拼积木而不是做设计。
- 关联题库：[[./05-Question-Banks/08-系统设计与场景题库|系统设计与场景题库]]

## 决策 7：质量还没完全稳，先优化成本还是先补治理？

- 适用场景：上线前后、成本飙升、质量波动明显。
- 判断标准：
  - 若错误会带来高风险或高投诉，先补治理和门禁。
  - 若质量已过线但成本不可持续，再优先做路由、缓存和容量优化。
- 关联题库：[[./05-Question-Banks/05-评测与上线治理题库|评测与上线治理题库]]、[[./05-Question-Banks/09-成本、性能与容量规划题库|成本、性能与容量规划题库]]

## 决策 8：事故后先修局部问题，还是先补系统门槛？

- 适用场景：重复事故、幻觉反复、Agent 失控、上线回归频繁。
- 判断标准：
  - 如果是单点缺陷且影响面小，先修局部问题。
  - 如果同类问题反复出现，必须优先补 `评测样本 / 发布门禁 / runbook / 告警` 这类系统门槛。
- 关联题库：[[./05-Question-Banks/10-项目复盘与事故题库|项目复盘与事故题库]]

## 决策 9：文档工作流该走 OCR / layout pipeline，还是直接赌 end-to-end multimodal model？

- 适用场景：合同抽取、票据识别、表单处理、复杂文档问答。
- 判断标准：
  - 若字段责任重、审计要求高、需 evidence span，优先保留 OCR / layout / validation 分层。
  - 若任务更偏开放理解、容错高、交互性强，可更积极利用 end-to-end multimodal model。
- 关联题库：[[./05-Question-Banks/11-多模态、语音、OCR 与 Computer Use题库|多模态、语音、OCR 与 Computer Use题库]]

## 决策 10：什么时候该上 Voice / Realtime，什么时候只保留文本交互？

- 适用场景：语音助手、客服、实时协作、移动端操作。
- 判断标准：
  - 若价值来自低摩擦输入、连续打断、自然交互，Voice / Realtime 值得做。
  - 若核心任务更偏准确、审计、可回放，文本往往更稳也更便宜。
- 关联题库：[[./05-Question-Banks/11-多模态、语音、OCR 与 Computer Use题库|多模态、语音、OCR 与 Computer Use题库]]

## 决策 11：Computer Use 是真的必要，还是更适合做显式 tool integration？

- 适用场景：浏览器操作、后台系统串联、没有稳定 API 的遗留系统。
- 判断标准：
  - 若系统 UI 漂移大、动作风险高、API 可补，优先 tool integration。
  - 若必须跨多个现成界面、没有统一 API、且保留足够 approval / sandbox，才考虑 computer use。
- 关联题库：[[./05-Question-Banks/11-多模态、语音、OCR 与 Computer Use题库|多模态、语音、OCR 与 Computer Use题库]]

## 决策 12：Assistant、Copilot，还是 Workflow Automation？

- 适用场景：产品定义、场景设计、业务落地排序。
- 判断标准：
  - 若价值来自低摩擦问答和辅助，优先 assistant / copilot。
  - 若价值来自流程吞吐、错误降低、审批串联，优先 workflow automation。
- 关联题库：[[./05-Question-Banks/12-产品落地、Rollout、ROI 与 Governance题库|产品落地、Rollout、ROI 与 Governance题库]]

## 决策 13：先追 Adoption，还是先追 ROI 可证明性？

- 适用场景：项目试点、扩张和管理层汇报。
- 判断标准：
  - 若系统还在找 workflow fit，先追 adoption 和 retained usage。
  - 若项目准备扩大投资，必须补上 ROI 证据和 owner 机制。
- 关联题库：[[./05-Question-Banks/12-产品落地、Rollout、ROI 与 Governance题库|产品落地、Rollout、ROI 与 Governance题库]]

## 决策 14：Context / RAG / Memory，优先补哪一层？

- 适用场景：多轮任务、个性化系统、长期任务或项目型 agent。
- 判断标准：
  - 当前轮任务材料不够，先补 context。
  - 事实来源缺失或更新快，先补 RAG。
  - 真正存在跨会话状态价值时，再补 memory。
- 关联题库：[[./05-Question-Banks/13-Memory、Planning 与 Multi-Agent题库|Memory、Planning 与 Multi-Agent题库]]

## 决策 15：单 Agent、显式 Workflow，还是 Multi-Agent？

- 适用场景：复杂任务设计、平台化、长任务执行。
- 判断标准：
  - 若路径明确且要可控，优先 workflow。
  - 若任务需要灵活探索但仍可控，先做单 agent。
  - 若存在明显并行收益和角色分工，再考虑 multi-agent。
- 关联题库：[[./05-Question-Banks/13-Memory、Planning 与 Multi-Agent题库|Memory、Planning 与 Multi-Agent题库]]
