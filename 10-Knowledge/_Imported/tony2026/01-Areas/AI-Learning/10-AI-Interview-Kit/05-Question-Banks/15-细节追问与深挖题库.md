# 细节追问与深挖题库

> 这组题专门练“面试官已经听完你的主回答，又继续往细节里钻”的情况。
> 目标不是背更多名词，而是让你在追问时还能保持 `边界、取舍、指标、风险、机制`。

## 一、底层原理追问

### Q1. 为什么 token 数会同时影响成本、延迟和质量？

- 考察点：是否能把底层输入单位接到工程指标。
- 回答骨架：token 越多，prefill 成本、上下文窗口、KV Cache、生成延迟和费用都会上升；过长 context 还会引入噪声和注意力稀释。
- 追问：你如何在 RAG 系统里控制 token budget？

### Q2. Attention 的系统代价是什么？

- 考察点：是否不只会背 Q/K/V。
- 回答骨架：attention 带来上下文聚合能力，但长上下文下计算和内存成本上升；推理侧要关注 KV Cache、context pruning 和长短请求分池。
- 追问：为什么长上下文不等于无限记忆？

### Q3. Temperature、top-p 在生产系统里怎么设？

- 考察点：是否理解输出随机性和风险等级。
- 回答骨架：高风险任务低随机、强结构；创意任务可放宽；不同步骤可不同采样策略；上线要绑定 eval 和回归。
- 追问：什么时候需要固定 seed 或强制 JSON schema？

## 二、训练与后训练追问

### Q4. SFT 后模型为什么仍然会一本正经地错？

- 考察点：是否理解 SFT 的目标边界。
- 回答骨架：SFT 学示范分布和指令格式，不保证事实 grounding；事实能力受 base model、训练数据和外部知识链路影响。
- 追问：你会用 fine-tuning 还是 RAG 修这个问题？

### Q5. DPO 和 PPO 的工程取舍是什么？

- 考察点：是否能从复杂度、稳定性和收益讲。
- 回答骨架：DPO 简化稳定、适合偏好数据直接优化；PPO 控制更强但训练复杂、调参难、成本高；应用团队通常先 DPO / SFT + eval。
- 追问：什么情况下 PPO 值得投入？

### Q6. 训练数据污染如何发现？

- 考察点：是否有数据治理意识。
- 回答骨架：检查 benchmark overlap、重复样本、异常高分、近似匹配、时间切分、canary 数据和人工抽样。
- 追问：污染发现后是删数据、重训，还是只修 eval？

## 三、推理与架构追问

### Q7. 成本突然翻倍，你第一小时怎么止血？

- 考察点：事故现场优先级。
- 回答骨架：先限流/降级/回退路由；查 token、重试、缓存、模型路由、异常用户、长上下文占比；临时关闭高成本路径。
- 追问：止血后如何防止复发？

### Q8. 模型路由如何避免越优化越复杂？

- 考察点：治理复杂度。
- 回答骨架：先定义少量稳定 tier；只对高价值/高风险任务升级；路由规则版本化；用离线 replay 和线上 A/B 验证。
- 追问：路由错误的监控指标是什么？

### Q9. 如何设计 fallback？

- 考察点：生产可用性。
- 回答骨架：模型 fallback、检索 fallback、工具 fallback、人工 fallback、只读/降级模式；每种 fallback 都要有触发条件和用户体验设计。
- 追问：fallback 会不会掩盖真实问题？

## 四、RAG 与数据追问

### Q10. Chunking 怎么做评测？

- 考察点：是否能从策略走到验证。
- 回答骨架：构造 query-doc relevance 集；比较 recall@k、MRR、answer faithfulness、引用准确率、成本和延迟。
- 追问：文档结构复杂时怎么切？

### Q11. RAG 答案引用错了怎么办？

- 考察点：grounding 细节。
- 回答骨架：检查检索召回、rerank、context packing、citation span、答案生成约束；必要时分离 evidence selection 和 answer generation。
- 追问：如何让模型“不知道就说不知道”？

### Q12. 权限过滤应该在检索前还是生成后？

- 考察点：安全边界。
- 回答骨架：必须尽量在检索前/索引层过滤，生成后过滤只能作为防线之一；不能让模型看到无权限内容。
- 追问：缓存和向量索引如何做租户隔离？

## 五、Agent 与 Tool Use 追问

### Q13. Tool calling 失败主要有哪些类型？

- 考察点：故障分类。
- 回答骨架：选错工具、参数错、状态过期、权限不足、工具异常、重试副作用、返回不可解析、模型误读结果。
- 追问：哪些失败适合自动重试？

### Q14. 高风险工具怎么设计 approval？

- 考察点：风险分级和人机协作。
- 回答骨架：按动作副作用分级；低风险自动，高风险 dry-run + 人审；审批界面展示 intent、diff、数据、回滚路径。
- 追问：如何避免审批疲劳？

### Q15. Agent 循环调用怎么防？

- 考察点：控制面设计。
- 回答骨架：预算、最大步数、状态检查、重复动作检测、终止条件、planner/verifier 分离、trace 告警。
- 追问：循环是模型问题还是 harness 问题？

### Q16. Sandbox 的最小边界是什么？

- 考察点：执行安全。
- 回答骨架：文件系统、网络、环境变量、凭证、进程、包安装、产物导出、日志审计都要有边界。
- 追问：什么时候必须断网执行？

## 六、Eval 与上线追问

### Q17. LLM-as-judge 怎么用才不危险？

- 考察点：评测可信度。
- 回答骨架：用明确 rubric、pairwise 对比、校准人工样本、多 judge、置信度和偏差分析；不能把 judge 当唯一真相。
- 追问：judge model 升级后怎么办？

### Q18. Bad case 如何进入回归集？

- 考察点：质量闭环。
- 回答骨架：分级、去重、标注根因、补 expected behavior、绑定版本、加入 release gate，定期清理过时样本。
- 追问：回归集越变越大怎么办？

### Q19. AI 系统灰度和传统系统灰度有什么不同？

- 考察点：AI 质量不确定性。
- 回答骨架：除错误率和延迟，还要看任务完成率、事实性、安全事件、人工接管、投诉和成本；需要 shadow/replay。
- 追问：灰度期间用户反馈和离线 eval 冲突怎么办？

## 七、安全与合规追问

### Q20. Prompt injection 和 jailbreak 的区别是什么？

- 考察点：威胁分类。
- 回答骨架：jailbreak 偏绕过模型行为策略；prompt injection 常发生在应用上下文里，诱导模型泄露、越权或滥用工具。
- 追问：RAG 文档里的恶意指令怎么处理？

### Q21. PII 日志怎么处理？

- 考察点：隐私工程。
- 回答骨架：最小化采集、脱敏、访问控制、保留周期、审计、删除机制；调试样本要二次处理。
- 追问：线上事故需要原始日志怎么办？

### Q22. 安全指标和业务指标冲突怎么办？

- 考察点：owner 判断。
- 回答骨架：按风险等级处理；不可逆/合规/外部副作用优先安全；低风险可灰度实验；把决策上升到 release policy。
- 追问：如何向业务解释不上线？

## 八、产品与组织追问

### Q23. AI 项目 adoption 很高但 ROI 不明显怎么办？

- 考察点：业务指标判断。
- 回答骨架：拆使用是否替代真实工作、是否节省高价值时间、是否提升质量；检查隐藏成本和用户是否只是新鲜感使用。
- 追问：如何设计下一轮实验？

### Q24. 如何判断一个场景不适合 AI？

- 考察点：克制能力。
- 回答骨架：低频、低价值、强确定性、错误不可接受、缺少数据、没有 owner、无法评测或无法兜底的场景不适合先做。
- 追问：如果老板坚持要做呢？

### Q25. 如何把个人 prompt 经验沉淀成团队能力？

- 考察点：机制化。
- 回答骨架：prompt registry、变更说明、eval 绑定、review、最佳实践、模板、反例库、培训和 owner 机制。
- 追问：如何防止 registry 变成没人看的文档？

## 九、角色化深挖

### Q26. Research 岗最容易追问什么？

- 考察点：模型能力形成。
- 回答骨架：architecture、data mixture、post-training、benchmark、ablation、failure analysis、scaling。
- 追问：你如何证明一个能力来自训练而不是 prompt trick？

### Q27. AI Platform 岗最容易追问什么？

- 考察点：平台化和治理。
- 回答骨架：model gateway、routing、registry、eval harness、observability、multi-tenant、cost governance、release gate。
- 追问：平台怎么支持多个业务线而不失控？

### Q28. Agent / Coding Agent 岗最容易追问什么？

- 考察点：执行系统。
- 回答骨架：tool use、sandbox、approval、artifact、repo context、test feedback、task eval、rollback。
- 追问：如何判断 agent 真完成任务，而不是只生成看似合理的输出？

### Q29. Product / Tech Lead 岗最容易追问什么？

- 考察点：落地和组织。
- 回答骨架：workflow fit、ROI、adoption、risk tier、rollout、stakeholder alignment、operating cadence。
- 追问：如何在业务速度和治理之间做取舍？

## 十、压轴综合追问

### Q30. 如果你只能用一张图讲 AI 系统，你画什么？

- 考察点：抽象能力。
- 回答骨架：能力形成链、系统执行链、评测上线链、业务闭环链四条线。
- 追问：这张图里哪个环节最容易成为瓶颈？

### Q31. 一个 AI 项目为什么 Demo 成功但上线失败？

- 考察点：生产经验。
- 回答骨架：场景边界不清、数据脏、权限复杂、指标缺失、成本高、延迟差、缺少 eval、没有回滚、用户 workflow 不匹配。
- 追问：你会如何提前发现？

### Q32. 你怎么证明自己不是只会调 API？

- 考察点：候选人自我定位。
- 回答骨架：讲能力来源、系统分层、上线治理、事故复盘、业务指标和团队机制；用真实项目说明 owner 范围。
- 追问：你最能代表这一点的项目是什么？
