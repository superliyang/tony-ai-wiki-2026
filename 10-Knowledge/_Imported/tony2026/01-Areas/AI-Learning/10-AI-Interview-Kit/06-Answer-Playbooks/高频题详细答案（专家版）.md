# 高频题详细答案（专家版）

> 这份文档给的是“能直接说出口”的详细答案版本。建议按 `30秒结论 -> 2分钟结构化回答 -> 5分钟深挖` 三层练习。
> 如果你想让面试官先建立“这是个 senior”的判断，先读 [[./资深候选人表达总纲|资深候选人表达总纲]] 再回来练这里。

## 0) 开场时 1 分钟自我介绍怎么答，才不会像流水账？

### 30 秒结论
我会避免从学校或早期经历平铺直叙，而是直接讲自己的职业主线、最擅长解决的问题，以及最近最有代表性的结果。

### 2 分钟结构化回答
1. **主线定位**：我长期做的是把 AI 能力变成可上线、可治理的系统，而不只是模型 Demo。
2. **核心能力**：我最擅长的是把模型、RAG、工具调用、评测和发布门禁串成完整交付链。
3. **代表成果**：最近最有代表性的项目是把某个 `AI assistant / agent / 知识库` 做到生产可用，重点负责架构取舍和风险治理。
4. **岗位匹配**：所以如果岗位需要的是能把 AI 系统做稳、做成、还能带团队形成方法的人，我比较匹配。

### 5 分钟深挖
- 自我介绍的目标不是“信息完整”，而是让面试官迅速知道你站在哪个级别上。
- 资深岗位要尽量减少“我会很多技术”，增加“我对结果负责过什么”。

## 0.5) 让你讲最能代表自己的项目，怎么答才显得真做过？

### 30 秒结论
我会按 `业务问题 -> 难点 -> 我的责任 -> 方案 -> 结果 -> 风险与复盘` 讲，而不是只讲技术栈。

### 2 分钟结构化回答
1. **业务问题**：先说明服务对象是谁，原来痛点是什么。
2. **关键难点**：说明为什么这不是“接个模型 API”就能解决。
3. **我的责任**：明确我负责的架构、治理和协作边界。
4. **核心方案**：讲主链路和关键控制点，而不是所有组件。
5. **结果与复盘**：给量化结果，再补上最大风险和后续改进。

### 5 分钟深挖
- 如果项目是团队做的，更要把“我做了什么决策、定义了什么门槛、推动了什么落地”讲清楚。
- 面试官更愿意听你讲一次真实的困难和修复，而不是一个完美但发虚的成功故事。

## 0.7) 怎么回答“为什么你适合更高级别岗位”？

### 30 秒结论
我会把答案从“我懂很多技术”切换成“我能定义边界、统一机制、守住上线质量、把个人经验变成团队能力”。

### 2 分钟结构化回答
1. **视角更高**：我不只解决单点技术问题，也负责质量、成本、风险和交付节奏的平衡。
2. **责任更重**：我不只写方案，也会定义发布门禁、回滚机制和协作方式。
3. **复用更强**：我会把一次项目沉淀成团队模板、评测集和工作流，而不是每次从零开始。

### 5 分钟深挖
- 高级岗位不是“做得更多”，而是“对系统级结果负责得更多”。
- 如果你能把 `个人判断 -> 团队机制` 这条线讲出来，资深感会明显增强。

## 1) 如何保证 Agent 调用工具（Function Calling）的可靠性？

### 30 秒结论
我会把它当成“分布式系统里的外部依赖调用”来做，不当成 prompt 技巧。核心是四层：**输入约束、执行控制、结果验证、失败补偿**。

### 2 分钟结构化回答
1. **输入约束**：使用严格 JSON Schema + 业务规则校验（金额范围、日期格式、枚举值）。
2. **执行控制**：超时、重试、熔断、并发限流；写操作必须带幂等键，避免重复扣费/重复创建。
3. **结果验证**：校验返回结构与业务断言；关键动作做二次确认（read-after-write）。
4. **失败补偿**：失败分级（可重试/需人工/必须回滚），并把失败样本沉淀到回归集。
5. **可观测**：我会盯工具成功率、P95 延迟、重试率、越权拦截率、人工接管率。

### 5 分钟深挖（面试官追问时）
- 高风险工具（付款、删除、发信）走审批门禁，不允许模型直接执行不可逆动作。
- 统一 Tool Gateway 做鉴权、审计、策略、配额，Agent 只拿短期最小权限 token。
- 落地上我会先做“失败类型字典”，把偶发失败转成可回归测试。

## 2) 多轮对话 KV Cache 爆了怎么办？

### 30 秒结论
我不会只靠加机器，会从**上下文治理 + 服务分层 + 路由策略**三层同时处理。

### 2 分钟结构化回答
1. **上下文治理**：滑动窗口保留最近轮次；旧轮次做结构化摘要（facts/decisions/open issues）；长期记忆外置到检索层。
2. **服务分层**：长/短请求分池，避免长请求拖垮全局尾延迟；为高价值请求预留容量。
3. **模型路由**：仅在必要回合用长上下文模型，其余走低成本模型 + RAG。
4. **指标闭环**：显存峰值、吞吐、首 token 延迟、质量回归分（防压缩后答非所问）。

### 5 分钟深挖
- 若业务是客服场景，我会优先保证稳定响应与成本可控，而不是追求每轮都全量上下文。
- 摘要必须可解释：保留关键事实 pin，避免“摘要漂移”。

## 3) RAG 里的 Query 改写怎么做？

### 30 秒结论
Query 改写的目标不是“改写得漂亮”，而是提高召回质量并减少语义漂移。

### 2 分钟结构化回答
1. **意图识别**：是问答、对比、排障还是生成任务。
2. **改写三层**：规范化（缩写/同义词）-> 结构化（实体/时间/约束）-> 多路检索（原 query + 扩展 query）。
3. **检索编排**：Hybrid（BM25+向量）+ rerank；按任务动态设置 top-k 和 chunk 粒度。
4. **防过改写**：原 query 保留并行召回，避免改写引入错误先验。
5. **评测**：Recall@k、nDCG、faithfulness、citation hit rate。

### 5 分钟深挖
- 法条、代码符号、合同条款等高精度场景，改写要非常保守。
- 线上要监控“改写前后命中差异”，判断改写是否真的有增益。

## 4) SFT 和 RLHF 的本质区别是什么？

### 30 秒结论
SFT 是监督拟合，RLHF 是偏好优化，二者优化目标和风险都不同。

### 2 分钟结构化回答
- **SFT**：按示范数据做 token-level 学习，优势是稳定、成本相对低。
- **RLHF**：用人类偏好信号（或奖励模型）做策略优化，优势是更贴近“人想要的回答”。
- **本质差异**：监督信号粒度、优化目标、稳定性和失效模式都不同。
- **工程实践**：通常 `Pretrain -> SFT -> Preference Optimization(DPO/PPO) -> 安全校准`。

### 5 分钟深挖
- RLHF 更容易出现 reward hacking，需要 KL 约束与离线回归集兜底。
- DPO 更稳更轻，PPO 控制力更强但工程复杂。

## 5) 多 Agent 场景里主 Agent 怎么动态派生 subAgent？

### 30 秒结论
先有任务图再有多 Agent。没有可并行收益时，不要为了“炫技”拆分。

### 2 分钟结构化回答
1. Planner 把目标拆成 DAG（依赖、并行、验收标准）。
2. Router 按能力、成本、SLA 选择子代理模板。
3. Coordinator 做冲突裁决与结果融合。
4. 状态总线只共享必要上下文，避免全量广播。
5. 预算与权限隔离，防无限递归和越权执行。

### 5 分钟深挖
- 我会限制最大深度、最大步骤数、最大 token 预算。
- 失败时优先回退到单 Agent 简化链路，保证可控。

## 6) 你如何设计可上线的 LLM 系统（不是 POC）？

### 30 秒结论
POC 关注“能跑起来”，生产系统关注“稳定可控可回滚”。

### 2 分钟结构化回答
- **控制面**：模型路由、策略配置、灰度开关。
- **数据面**：请求编排、缓存、检索、工具网关。
- **治理面**：评测门禁、审计、风险策略、回滚机制。
- **观测面**：质量、成本、延迟、安全四象限指标。

### 5 分钟深挖
- 我会用阶段门禁：离线通过 -> Shadow -> 灰度 -> 全量。
- 每一阶段都预设回滚阈值与责任人，不接受“出事再说”。

## 7) 如何做 Release Gate？

### 30 秒结论
Release Gate 的本质是“把主观判断前移为可执行阈值”。

### 2 分钟结构化回答
1. Gate 1：离线质量/安全回归必须过线。
2. Gate 2：Shadow 对比不劣化。
3. Gate 3：灰度实时监控（质量、错误、成本、投诉）。
4. Gate 4：触发阈值自动回滚。
5. Gate 5：复盘回灌评测集。

### 5 分钟深挖
- 重点业务（金融/医疗）必须有人工审批和变更审计。

## 8) Prompt Injection 和 Tool Abuse 怎么防？

### 30 秒结论
防注入不能只靠提示词，要靠**架构隔离 + 策略执行 + 审计追溯**。

### 2 分钟结构化回答
- System/Developer/User/Tool 输出上下文分级。
- 工具白名单 + 参数策略检查 + 最小权限 token。
- 高风险动作二次确认/HITL。
- 外部内容默认不可信，不允许直接转化为高风险指令。

### 5 分钟深挖
- 关键是“默认拒绝 + 明确允许”，而不是“默认允许 + 事后补救”。

## 9) 如何向面试官解释“为什么现在不能全量上线”？

### 可直接说
“不是我们做不出来，而是我们还没通过上线门禁。现在离线质量OK，但灰度阶段发现在高并发场景下工具失败率偏高，且人工接管率超阈值。如果此时全量，会把风险转嫁给用户和客服团队。我会先补强失败补偿与回归集，再做第二轮灰度，这样能把上线风险从不可控变成可控。”

## 10) 成本太高时你怎么做优化？

### 可直接说
我会先做成本分解：输入 token、输出 token、模型单价、重试率、缓存命中率、长上下文占比。通常先做三件事：
1. 路由优化（简单任务降级模型）；
2. 上下文治理（摘要/裁剪/RAG）；
3. 提升缓存命中与工具成功率（减少重试）。
然后用 A/B 证明质量不显著下降再扩展。

## 11) 面试官问你“你是怎么带团队做 AI 架构治理的？”

### 可直接说
我会把团队治理拆成三层：
- **研发层**：统一开发规范（prompt、tool schema、日志字段、评测模板）。
- **发布层**：统一门禁流程（离线、shadow、灰度、回滚）。
- **运营层**：统一复盘机制（事故分级、根因分类、回归集更新）。
这样团队不依赖“某个高手”，而是依赖体系。

## 12) 如果你是小白，怎么在 2 周内练到“像专家”

1. 先背 12 道高频题的 30 秒结论。
2. 每题补全 2 分钟结构化回答（按这份文档）。
3. 每天做 2 道 5 分钟追问模拟（让朋友扮演面试官）。
4. 每天复盘 3 个“我答崩了的点”，转成自己的答题卡。
5. 第 14 天做一场 60 分钟完整模拟（技术 + 架构 + 安全）。

## 13) 万能答题模板（建议背下来）

你可以用这个顺序回答大多数架构题：

1. **先定义问题边界**（业务目标、约束、风险）。
2. **再给架构分层方案**（控制面、数据面、治理面）。
3. **再说取舍**（为什么不用别的方案，代价是什么）。
4. **最后说验证闭环**（指标、评测、灰度、回滚、复盘）。

> 这套模板的作用是：让面试官快速判断你不是“会一点点工具的人”，而是“能对结果负责的人”。
