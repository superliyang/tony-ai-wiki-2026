## What I Actually Understand Now

- 面试表现的核心不只是知识点，而是：问题定义、架构取舍、风险边界、验证闭环。
- 同一个问题至少要准备三层回答：基础版（2 分钟）、架构版（5 分钟）、事故版（含回滚）。
- 顶级岗位面试里，追问往往在“为什么这么设计”和“失败时怎么兜底”。
- 真正拉开级别感的往往不是技术题本身，而是开场、自我定位、项目 owner 边界、事故复盘和跨团队推动方式。
- 想让面试官感觉“这是个 senior”，必须把答案持续拉回 `业务目标 -> 架构方案 -> tradeoff -> 指标 -> 风险 -> 机制化`。
- 如果想让知识面显得非常全，还必须主动准备系统设计、性能成本、容量规划和事故治理题，因为这些题最容易暴露“只懂模型、不懂系统”。
- 仅有通用题库还不够，最后一层 senior 感往往来自你能不能把同样的方法落到具体公司语境里。
- 面试的更高层本质是“能不能把 AI 当作一个完整系统来解释”：能力从哪里来、如何被工程化、如何被验证和治理、如何进入业务并沉淀成团队机制。
- 一道开放题最稳的回答顺序是：`问题边界 -> 底层原因 -> 系统方案 -> 关键取舍 -> 验证指标 -> 上线机制 -> 失败复盘`。
- 深挖题最稳的回答单位不是“更多概念”，而是 `一句话定义 -> 为什么重要 -> 工程边界 -> 生产验证 -> 常见掉坑`。

## What Still Feels Fuzzy

- 细节追问已经有地图和题库，但最高频的 5-10 个问题还可以继续扩成完整 2 分钟答案。
- 现在上帝视角已经补上，但还可以继续拆成 `Research / Applied ML / AI Platform / Agent / Product Lead` 五种角色的不同追问重点。
- 还缺少足够多按公司类型拆开的场景题包，尤其是 `voice / realtime`、`document AI`、`computer use`。
- 对成本模型（token、GPU、人力运维）的量化表达还可以更强。
- rollout / ROI / governance 相关问法现在已经有正式题库，但行业化案例和 adoption failure 模式还可以更厚。
- memory / planning / multi-agent 现在也有正式题库，但共享 memory、长任务和多 agent 的失败案例还可以再补深。
- 事故类回答模板还可以再补得更厚，尤其是“先止血、后复盘、再制度化”的细节。
