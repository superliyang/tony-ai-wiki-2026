# 开发者体验与平台指标

## 一句话理解

平台工程不能只看“平台搭出来没有”，还要看开发者是否更快、更少出错、更愿意采用平台默认路径。这就是 `developer experience` 和平台指标要一起看的原因。

## 为什么不能只看 DORA

`DORA metrics` 很重要，但它们更偏结果层，比如部署频率、变更前置时间、故障恢复时间、变更失败率。平台团队还需要补一层更贴近平台本身的指标：

- self-service 使用率
- golden path adoption
- 新服务创建耗时
- ticket 依赖度
- docs / template 使用率
- friction 热点

## 常见指标层次

- 结果层：交付速度、稳定性、变更质量
- 采用层：portal、catalog、template、golden path 的使用率
- 体验层：开发者等待时间、重复工作、常见阻塞点
- 治理层：标准路径覆盖率、policy 命中率、例外申请量

## 学习重点

- 好的平台指标不是只衡量“平台团队干了多少”，而是衡量“开发者是否更容易把事做对”
- adoption 与 reliability 需要一起看，不能只追求使用率
- 平台团队常常需要把 DORA、DX 和 governance 指标组合成一个 review rhythm
