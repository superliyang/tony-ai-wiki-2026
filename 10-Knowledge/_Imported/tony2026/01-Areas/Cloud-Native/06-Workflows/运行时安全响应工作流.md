# 运行时安全响应工作流

## 这个工作流在解决什么

运行时安全最怕两种极端：

- 只有告警，没有响应动作
- 只有强阻断，没有渐进式采用路径

所以平台团队真正需要的是一个既能逐步提高默认安全基线，又能在发现异常时快速处置的 workflow。

## 推荐工作流

1. 先定义默认基线
   - `RuntimeDefault seccomp`
   - capability 最小化
   - 禁止 privileged / unrestricted workload
2. 把高风险 workload 单独分级
   - 例外场景
   - 历史负债 workload
   - 敏感 namespace
3. 建立运行时可见性
   - drift
   - suspicious process execution
   - unexpected network behavior
4. 为异常定义响应动作
   - alert only
   - quarantine / isolate
   - deny future admission
   - 强制修复并回滚
5. 复盘“默认安全”与“例外机制”
   - 哪些规则误伤高
   - 哪些团队依赖例外过多
   - 哪些节点/namespace 经常触发异常
6. 把运行时发现反推到 build 和 policy
   - seccomp profile
   - image policy
   - admission guardrails

## 关键判断

- 运行时安全必须和 admission、policy、incident response 连起来，才能形成真正闭环
- 默认基线比“临时手工审查”更可持续
- 好的 runtime security 不是告警越多越好，而是误伤可控、例外有边界、响应动作明确
