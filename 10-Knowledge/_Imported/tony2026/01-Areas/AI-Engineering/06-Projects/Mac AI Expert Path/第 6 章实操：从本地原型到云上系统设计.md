# 第 6 章实操：从本地原型到云上系统设计

## 本章目标

把你在 `MacBook Pro Max M4` 上做出来的本地原型，翻译成一份像样的云上系统设计草案。

这一步非常关键，因为它会把“个人实验能力”转换成“团队工程判断能力”。

## 你要选一个原型作为起点

建议从下面二选一：

- 本地 `RAG` 文档问答原型
- 本地 `agent + tool use` 原型

不要同时做两个。先选一个最清楚、最有把握的，用它做迁移设计。

## 第一步：先把本地系统讲清楚

你要先写一页“当前本地原型说明”，至少包括：

- 目标用户是谁
- 输入是什么
- 输出是什么
- 模型怎么接入
- 检索或工具怎么接入
- 当前 trace / eval 做到什么程度
- 现在哪里最脆弱

如果你连本地原型都讲不清，就不可能做出好的云上设计。

## 第二步：明确哪些能力必须迁移

你要把系统拆成两类：

### 继续留在本地的

- 个人实验
- prompt 调试
- 小样本 eval
- notebook 验证
- 风险较低的离线分析

### 必须上云的

- 多用户共享访问
- 稳定 API / 服务接口
- 大一点的检索或工具编排
- 持续运行的 agent workflow
- 更强的日志、观测、安全和成本控制

## 第三步：画最小架构图

你至少要拆出这些模块：

- client / API gateway
- orchestration 或 app server
- model access layer
- retrieval / tool layer
- state / memory / storage
- observability
- eval / feedback loop
- security / secret management

你不用一开始把每个模块都做复杂，但要先把边界讲清楚。

## 第四步：把本地组件映射到云上角色

例如你可以这样想：

- `Ollama / MLX-LM` 在本地只是 runtime 验证手段
- 真到云上时，模型服务可能要换成更标准的 serving 路线
- 本地文件检索，云上就要变成对象存储 + index + service
- 本地 agent 脚本，云上就要变成 API + background worker + trace

关键不是“完全一比一复刻”，而是把角色映射清楚。

## 第五步：补齐生产化必需信息

### 1. 可靠性

- 请求失败怎么处理
- 工具调用失败怎么降级
- 是否需要重试
- 哪些路径可以容忍部分失败

### 2. 可观测性

- 你要记录哪些日志
- 哪些字段要进入 trace
- bad cases 如何沉淀
- eval 如何接入 release gate

### 3. 安全

- secret 怎么管
- tool 权限怎么控
- prompt injection 风险怎么降
- 用户数据和文档数据如何隔离

### 4. 成本

- 哪些请求最贵
- 哪些流量可以缓存
- 哪些模型必须高配，哪些可以降级
- 检索与工具调用链路如何避免无谓放大

## 第六步：写一份分阶段落地计划

不要写成“最终完美系统”，而要分阶段：

### Phase 1

- 单租户或小范围内部使用
- 最小 API
- 最小检索 / 工具链
- 基础 trace

### Phase 2

- 更稳定的 serving
- 更完整的 eval
- 审批 / guardrails / security gate
- 更好的缓存和成本控制

### Phase 3

- 多用户或多团队接入
- 更完整的 workflow / agent orchestration
- 组织级 observability 和 release governance

## 推荐文档结构

你可以把最终设计文档按这个顺序写：

1. 背景与目标
2. 当前本地原型说明
3. 目标用户与使用场景
4. 目标架构图
5. 模块拆分
6. 数据流与请求流
7. eval / observability / security
8. 成本与风险
9. phased rollout

## 本章输出物

完成本章后，你应该有：

- 一份“本地原型 -> 云上系统”的架构草图
- 一张模块边界图
- 一份 phased rollout 说明
- 一份成本 / 安全 / observability 的最低要求清单

## 这章真正训练的能力

这一步训练的已经不是“会不会写代码”，而是：

- 你能否抽象系统边界
- 你能否识别生产化问题
- 你能否把个人实验翻译成团队工程语言

## 常见误区

- 直接把本地 demo 视为生产架构
- 只画图，不写失败处理
- 不考虑 observability / security / cost
- 一上来就设计过于庞大的平台

## 最后一件事

如果你能把这一章做好，`Mac AI` 这条学习路径就真正闭环了：

- 本地跑通
- 本地训练
- 本地微调
- 本地应用
- 云上映射

这也是从“强个人实验者”走向“工程型 AI 专家”的关键一步。
