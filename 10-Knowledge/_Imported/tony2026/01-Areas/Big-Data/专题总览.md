## 这个领域解决什么

这里不把大数据当成工具清单，而是当成一套“数据从产生到被决策使用”的系统工程：

- 数据怎么被采集、传输、存储、计算和服务
- batch、streaming、warehouse、lakehouse、feature / vector / OLAP 这些层怎样分工
- 数据质量、血缘、权限、成本、SLA 和 ownership 怎么变成平台能力
- 当 AI、BI、运营分析和实时业务都要用同一批数据时，系统边界怎么划

## 先建立的五条线

1. `Data Lifecycle`：采集、传输、存储、计算、服务、治理
2. `Data Modeling`：ODS / DWD / DWS / ADS、维度建模、指标口径、semantic layer
3. `Batch + Streaming`：Spark、Flink、Kafka、CDC、event time、exactly-once、backfill
4. `Lakehouse / Warehouse`：warehouse、data lake、lakehouse、Iceberg / Delta / Hudi 类表格式
5. `Data Platform`：orchestration、metadata、quality、lineage、access control、cost / FinOps

## 和其他领域的边界

- 和 [[../Cloud-Native/专题总览|Cloud-Native]] 的关系：大数据平台通常跑在 Kubernetes / cloud / storage / observability 上，但本领域关注数据生命周期和数据产品。
- 和 [[../AI-Engineering/专题总览|AI-Engineering]] 的关系：AI 需要数据管道、datasets、feature、vector 和 eval 数据；本领域先补通用数据工程底座。
- 和业务领域的关系：支付、游戏、增长、风控都可以复用这里的数据平台判断，但具体业务口径留在对应领域。

## 本轮学习目标

- 不急着背 Hadoop / Spark / Flink 名词，而是先知道每一层在解决什么约束
- 能从一个业务问题反推数据链路、计算模式、存储形态和治理需求
- 能判断“该补 batch、streaming、lakehouse、指标体系，还是 data quality / lineage”
