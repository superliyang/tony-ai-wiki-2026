# 安全事件响应 Playbook

> 用途：当出现告警、入侵迹象、数据泄露、账号接管、云密钥泄露、勒索、供应链投毒或重大漏洞利用时，用这套流程组织响应。

## 一句话

事件响应的核心不是“立刻修一下”，而是：

`确认事实 -> 分级 -> 止血 -> 取证 -> 根除 -> 恢复 -> 通报 -> 复盘 -> 改进控制`

## 事件分级

### P0：重大事件

典型条件：

- 已确认数据泄露
- 生产系统被接管或中断
- 高权限云账号、密钥、CI/CD 被攻陷
- 勒索、破坏、供应链投毒正在发生
- 触发监管、客户或合同通报义务

要求：

- 立即组建 incident room
- 指定 incident commander
- 法务、隐私、公关、业务 owner 加入
- 冻结关键证据

### P1：高风险事件

典型条件：

- 公网漏洞疑似被利用
- 高权限账号异常
- 敏感数据访问异常
- 恶意软件或横向移动迹象

要求：

- 当日响应
- 明确止血动作和证据范围
- 建立复盘任务

### P2/P3：一般事件或可疑信号

典型条件：

- 单点告警
- 低影响漏洞利用尝试
- 可解释但需要跟踪的异常

要求：

- 记录、分析、关联、必要时升级

## 角色分工

- Incident commander：统一决策、节奏、升级。
- Security lead：判断攻击路径、止血、检测、取证。
- Engineering owner：执行隔离、修复、回滚、补丁。
- Infra / Cloud owner：处理云权限、网络、主机、容器。
- Legal / Privacy：判断通报、合规和证据要求。
- Comms / Support：处理客户沟通和外部说明。
- Business owner：判断业务影响和优先级。

## 响应流程

### 1. 确认与定界

要回答：

- 事件是否真实？
- 从什么时候开始？
- 涉及哪些资产、账号、数据、区域？
- 攻击者是否仍在系统内？
- 是否存在横向移动或持久化？

证据来源：

- 身份日志
- 云审计日志
- 应用日志
- 网络流量
- EDR / XDR
- CI/CD 和 artifact 日志
- 数据访问审计

### 2. 止血

常见动作：

- 禁用可疑账号和 token
- 轮换 API key、secret、证书
- 封禁恶意 IP、域名、User-Agent
- 临时关闭高风险接口
- 隔离主机、容器、工作负载
- 暂停发布流水线或第三方集成

注意：

- 止血动作要记录时间、执行人和影响。
- 不要在没有取证计划时直接清理所有痕迹。

### 3. 取证与分析

要保留：

- 原始日志
- 受影响系统快照
- 恶意文件或 payload
- 账号和权限变更记录
- 数据访问范围
- 时间线

要分析：

- 初始入口
- 权限提升路径
- 横向移动路径
- 数据访问或外传
- 持久化机制
- 检测为何触发或为何没触发

### 4. 根除与修复

动作包括：

- 修复漏洞或错误配置
- 移除恶意程序和持久化
- 轮换凭据
- 加固身份、网络、应用、云、供应链控制
- 补充检测规则
- 修复日志盲区

### 5. 恢复

恢复前检查：

- 攻击入口已关闭
- 凭据已轮换
- 日志和检测已覆盖
- 关键业务已验证
- 回滚或备份可信

### 6. 通报与沟通

要判断：

- 是否触发监管通报？
- 是否触发客户合同通报？
- 是否需要通知数据主体？
- 是否需要内部高层汇报？

不要在事实不清时过度承诺；但也不能因为害怕沟通而拖延必要通报。

### 7. 复盘

复盘要产出：

- 时间线
- 根因
- 影响范围
- 做得好的地方
- 做得差的地方
- 控制改进
- 检测改进
- 流程改进
- owner 和 deadline

## 高频场景

### 账号接管

优先动作：

- 禁用账号或强制登出
- 轮换凭据
- 检查 MFA、登录地点、设备、OAuth app
- 检查数据导出和权限变更

### 云密钥泄露

优先动作：

- 立即禁用或轮换密钥
- 查询密钥使用历史
- 检查新建资源、权限变更、数据访问
- 修复 secret 泄露来源

### 数据泄露

优先动作：

- 确认数据类型、数量、地区、数据主体
- 冻结证据
- 判断通报义务
- 修复访问控制和审计缺口

### 勒索或破坏

优先动作：

- 隔离受影响主机和账号
- 保护备份
- 判断横向移动
- 恢复前确认攻击者已被驱逐
