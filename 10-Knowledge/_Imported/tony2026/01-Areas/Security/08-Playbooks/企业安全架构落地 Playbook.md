# 企业安全架构落地 Playbook

> 用途：把安全从“零散工具和被动救火”推进到“可治理、可度量、可审计、可复盘”的企业能力。

## 一句话

企业安全架构不是画一张大图，而是建立一套能持续回答这些问题的系统：

`保护什么 -> 谁能访问 -> 怎么发布 -> 怎么检测 -> 怎么响应 -> 怎么证明 -> 怎么改进`

## 落地顺序

### 0. 先建立资产与 owner

没有资产清单，就没有安全架构。

必须最先建立：

- 业务系统清单
- 公网入口清单
- 高权限账号清单
- 数据资产清单
- 云账号与生产环境清单
- CI/CD、镜像仓库、依赖、第三方服务清单
- asset owner 与 risk owner

产物：

- asset inventory
- attack surface inventory
- crown jewels list

### 1. 身份先行

现代企业里，身份往往就是新边界。

最低控制线：

- SSO / IdP 统一身份入口
- MFA 覆盖员工和高风险用户
- RBAC / ABAC 管理权限
- PAM 管理高权限操作
- service account 和 API key 有 owner、过期、轮换、审计
- 离职、转岗、外包、供应商账号有自动回收流程

关联：[[../05-Topics/身份与访问安全|身份与访问安全]]、[[../05-Topics/零信任架构|零信任架构]]

### 2. 数据分类与最小化

先知道数据在哪里，才能谈加密和合规。

最低控制线：

- 数据分类分级
- 敏感字段目录
- 数据流向图
- 加密、脱敏、tokenization、DLP 策略
- 数据保留和删除策略
- 数据访问审计

关联：[[../05-Topics/数据安全与隐私保护|数据安全与隐私保护]]

### 3. 应用与 API 进入 SDLC

安全要嵌入研发流程，而不是上线前临时挡路。

最低控制线：

- threat modeling
- secure coding baseline
- SAST / SCA / secret scanning
- API 鉴权、授权、限流、审计
- 高风险变更 security review
- 上线 release gate

关联：[[../05-Topics/应用安全与 API 安全|应用安全与 API 安全]]、[[../05-Topics/安全工程与 DevSecOps|安全工程与 DevSecOps]]

### 4. 云与基础设施默认安全

云安全的第一原则：默认配置必须安全。

最低控制线：

- 云 IAM 最小权限
- 公网暴露自动发现
- 存储桶和数据库公开访问检测
- IaC scanning
- 容器镜像漏洞扫描
- Kubernetes RBAC、NetworkPolicy、secrets 管理
- 生产变更审计

关联：[[../05-Topics/云原生与基础设施安全|云原生与基础设施安全]]

### 5. 供应链可信

供应链安全要覆盖代码进入、构建、产物、发布和第三方。

最低控制线：

- SCA 和 license 检查
- SBOM
- artifact signing
- 构建环境隔离
- CI/CD secret 管理
- 第三方 SDK 和供应商评估
- 关键依赖升级与应急机制

关联：[[../05-Topics/供应链安全|供应链安全]]

### 6. 安全运营闭环

控制不是“存在”就够，还要能被监控和响应。

最低控制线：

- 身份、云、应用、数据、终端、网络日志接入
- SIEM / detection rules
- 告警分级和 on-call
- incident response role 和流程
- tabletop exercise
- post-incident review

关联：[[../05-Topics/安全运营、检测与响应|安全运营、检测与响应]]

### 7. GRC 与证据系统

治理层负责让安全变成组织机制。

最低控制线：

- policy 和 control library
- risk register
- exception management
- audit evidence
- third-party risk
- 安全指标和管理层汇报

关联：[[../05-Topics/安全治理、风险与合规|安全治理、风险与合规]]

## 成熟度分层

### Level 1：被动救火

- 没有完整资产清单
- 主要靠人工经验和临时处理
- 安全工具分散，缺少 owner
- 审计时临时补证据

### Level 2：基础控制

- 身份、资产、日志、漏洞管理初步建立
- 关键系统有安全评审
- 高风险问题能分级处理
- 但控制仍然依赖人工推动

### Level 3：平台化

- 安全基线进入默认模板
- CI/CD 有自动检查
- 高风险动作有审批、审计、回滚
- SOC 能稳定分诊和响应

### Level 4：风险驱动

- 安全优先级和业务风险绑定
- 控制有效性可度量
- 红队、演练、事故复盘持续反哺架构
- 合规证据自动化沉淀

## 架构师要交付的文档

- 企业安全目标与边界
- 资产与攻击面清单
- 身份与访问模型
- 数据分类与流向图
- 应用/API 安全基线
- 云与基础设施安全基线
- 供应链安全基线
- 日志、检测、响应架构
- 风险登记册与审计证据模型

## 最小可行路线

如果资源有限，按这个顺序做：

1. 资产和公网入口盘点
2. 高权限身份和密钥治理
3. 关键数据分类与访问审计
4. 应用/API 上线安全评审
5. 云配置和供应链扫描
6. 关键日志接入和事件响应流程
7. 风险登记和审计证据沉淀
