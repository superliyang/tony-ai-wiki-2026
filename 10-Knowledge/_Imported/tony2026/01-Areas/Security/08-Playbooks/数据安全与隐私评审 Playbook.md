# 数据安全与隐私评审 Playbook

> 用途：评审一个涉及个人信息、敏感数据、数据导出、分析建模、跨境传输、第三方共享或长期保存的数据处理活动。

## 一句话

数据安全与隐私评审的核心是：

`少收集 -> 分清楚 -> 控访问 -> 加保护 -> 可审计 -> 能删除 -> 可证明`

## 评审输入

至少需要：

- 数据处理目的
- 数据类型和字段
- 数据主体和地区
- 数据来源
- 数据流向图
- 存储位置和保留期限
- 访问角色和权限
- 第三方共享对象
- 安全控制和审计日志

## 核心检查面

### 1. 数据最小化

先问：

- 这个字段是否真的需要？
- 是否能用聚合、脱敏、token 或匿名化替代？
- 是否存在“先收了以后可能有用”的数据？
- 是否能缩短保存时间？

安全上最强的控制，是不收集不必要的数据。

### 2. 分类分级

建议至少分为：

- 公开数据
- 内部数据
- 机密数据
- 受监管数据
- 高敏数据：身份号、支付数据、认证凭据、健康数据、生物识别、儿童数据等

每个等级要绑定：

- 访问规则
- 加密规则
- 日志规则
- 导出规则
- 保留删除规则

### 3. 访问控制

要问：

- 谁能访问？
- 凭什么访问？
- 是否最小权限？
- 是否有审批？
- 是否有定期复核？
- 是否能追溯到具体人、系统、服务账号？

高风险信号：

- 共享账号
- 永久权限
- 数据库直连
- 批量导出无审批
- 生产数据复制到测试环境

### 4. 加密、脱敏与 Tokenization

选择逻辑：

- 传输保护：TLS
- 存储保护：encryption at rest
- 展示最小化：masking
- 分析降敏：pseudonymization / anonymization
- 高敏字段替代：tokenization
- 密钥保护：KMS / HSM / key rotation

关键判断：

- 加密不等于权限控制。
- 脱敏不等于匿名化。
- tokenization 适合高敏标识符，但要保护 token vault。

### 5. 数据流和第三方共享

要问：

- 数据是否进入日志、缓存、搜索、BI、ML、客服系统？
- 是否共享给供应商、广告、风控、支付、分析工具？
- 第三方是否再处理或再转移？
- 合同、DPA、SCC、审计证明是否齐全？

### 6. 保留、删除与数据主体请求

要问：

- 数据保留多久？
- 到期是否自动删除？
- 删除是否覆盖备份、缓存、派生数据？
- 用户访问、更正、删除、导出、撤回同意如何处理？
- 删除请求和审计留痕如何平衡？

### 7. 事件通报准备

要提前知道：

- 哪些数据泄露需要通报？
- 向谁通报：监管、客户、数据主体、合作方？
- 通报时限是什么？
- 需要哪些证据证明影响范围？

## 上线阻断项

- 未定义数据处理目的
- 收集明显超出目的的数据
- 高敏数据无访问控制或审计
- 生产敏感数据进入测试环境且无脱敏
- 数据导出无审批、无记录、无水印或无追溯
- 第三方共享无合同和安全评估
- 跨境传输没有合法机制和记录
- 无法处理删除、访问、更正等基本请求
