# Log4Shell 复盘

## 案例定位

Log4Shell 是典型的“基础开源组件漏洞变成全网攻击面”的案例。

它训练的不是某个 Java 漏洞细节，而是：

- 依赖资产是否可见。
- 互联网暴露面是否可快速盘点。
- 漏洞是否能按真实风险排序。
- 临时缓解、补丁、验证和检测是否能并行推进。

## 资料入口

- Apache：<https://security.apache.org/blog/cve-2021-44228/>
- MITRE CVE：<https://cve.mitre.org/cgi-bin/cvename.cgi?name=CVE-2021-44228>

## 资产

- 使用 Log4j 2 的 Java 应用。
- 直接依赖和间接依赖 Log4j 的组件。
- 公网 Web、API、网关、后台、第三方集成入口。
- 容器镜像、fat jar、供应商产品和历史构建产物。

## 攻击面

高风险入口通常是：

- 能被外部输入影响日志内容的服务。
- 公网可达的 Web/API。
- 会解析请求头、参数、User-Agent、错误信息的系统。
- 依赖链不清晰的供应商软件。

## 攻击链抽象

```text
外部输入 -> 应用记录日志 -> 触发脆弱组件行为 -> 远程代码执行风险 -> 权限扩展或数据访问
```

重点不是背 payload，而是看到“日志也可能变成执行路径”。

## 控制缺口

- SBOM / 依赖清单不完整。
- 无法快速回答“哪些系统用了 Log4j”。
- 公网入口、供应商产品、历史镜像不可见。
- 补丁、临时缓解、检测、验证没有统一调度。
- 安全运营缺少 exploit attempt 检测。

## 正确响应

### 立即动作

- 盘点直接和间接依赖。
- 优先处理公网暴露和关键业务系统。
- 应用官方修复或临时缓解。
- 增加 WAF、网关、日志检测等补偿控制。
- 检查是否已有利用迹象。

### 后续动作

- 建 SCA / SBOM。
- 统一依赖治理和镜像扫描。
- 建关键漏洞应急流程。
- 把漏洞响应接入 [[../08-Playbooks/漏洞管理与攻击面管理 Playbook|漏洞管理与攻击面管理 Playbook]]。

## 可迁移教训

- 依赖不可见，是供应链安全的第一风险。
- 漏洞优先级要结合公网暴露、可利用性、数据影响和补偿控制。
- 大规模漏洞响应必须并行做“盘点、修复、检测、验证、沟通”。
- 供应商软件也是你的攻击面。

## 自我演练

用自己的系统回答：

- 哪些服务使用 Java？
- 哪些依赖来自间接依赖？
- 是否能 2 小时内列出公网暴露服务？
- 容器镜像和历史构建产物是否可扫描？
- 如果出现同等级漏洞，谁来协调、谁来修、谁来验证？
