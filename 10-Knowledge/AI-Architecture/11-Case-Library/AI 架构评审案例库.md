# AI 架构评审案例库

> 用于训练架构师的“方案判断力”。不要只问能不能做，要问：是否值得做、能否上线、风险在哪、证据是否足够。

## 评审通用框架

每个 AI 架构方案都按 10 个问题评审：

1. 业务目标是否清楚？
2. 任务边界是否清楚？
3. 是否真的需要 AI？
4. RAG / Agent / Workflow / Fine-tuning / 传统系统的选择是否合理？
5. 数据和权限是否可控？
6. eval 是否能证明质量？
7. trace 是否能复盘问题？
8. 安全和合规风险是否有控制？
9. 成本、延迟和容量是否可接受？
10. 上线、回滚和责任人是否明确？

## 案例 1：客服知识助手

### 方案描述

业务希望给客服做一个知识助手，接入产品文档、工单和 FAQ。用户提问后，系统生成答案，客服复制给客户。

### 初始方案

- 所有文档切片后进入向量库。
- 用户问题直接 embedding 检索。
- 召回 Top 5 片段交给模型生成。
- 没有权限控制，因为客服都能看大部分知识。
- 用 20 个问题人工试了一下，感觉效果不错。

### 架构师评审

| 维度 | 判断 |
|---|---|
| 是否适合 AI | 适合，属于高频知识问答和辅助表达 |
| 主要风险 | 文档过期、引用不准、客户敏感信息泄露、回答口径不一致 |
| 最大缺口 | 没有 eval、引用校验、权限策略和 bad case 闭环 |
| 是否准入 | 不能直接生产，只能 R1 PoC |

### 修改建议

- 改成 hybrid search + rerank，不只依赖向量召回。
- 文档必须有 Owner、版本、更新时间、适用产品线。
- 答案必须带引用；引用不充分则转为“候选资料”。
- 建 eval set：高频问题、相似制度、过期文档、无答案、敏感客户信息。
- 建 trace：query、召回文档、prompt version、model、引用、反馈。
- 首期只做客服辅助建议，不自动发给客户。

### 面试表达点

> 我不会把这个方案直接上线，因为它只是能回答，不代表可信。我的改造重点是把 RAG 做成可验证系统：知识治理、权限、引用、eval、trace 和客服反馈闭环。

## 案例 2：采购审批 Agent

### 方案描述

业务希望 Agent 自动阅读采购申请、查预算、查供应商风险，并给出审批建议。未来希望自动批准低风险申请。

### 初始方案

- Agent 自主规划步骤。
- 工具包括预算查询、供应商查询、历史审批查询、审批通过 API。
- 模型根据历史审批记录学习判断。
- 审批人只看最终建议。

### 架构师评审

| 维度 | 判断 |
|---|---|
| 是否适合 AI | 适合做材料检查和建议，不适合一开始自动批准 |
| 主要风险 | excessive agency、工具误调用、历史审批偏差、金额风险、审计缺失 |
| 最大缺口 | 工具权限过大，workflow 边界不清，高风险动作无人工确认 |
| 是否准入 | 只能做只读建议型 Pilot，不能自动批准 |

### 修改建议

- 从 autonomous agent 改为 bounded workflow。
- 工具分级：只读工具、建议工具、高风险写工具。
- 审批 API 默认关闭，低风险自动化也必须先经过灰度和审计。
- 历史审批只能作为参考，规则和制度优先级更高。
- 每个建议必须输出：事实、规则、工具结果、风险点、人工确认项。
- 建 tool trace 和 approval trace。

### 面试表达点

> Agent 落地不是越自主越高级。企业审批场景的第一原则是 bounded autonomy：让模型理解和建议，让 workflow 控状态，让工具网关控权限，让人审节点控高风险动作。

## 案例 3：企业 LLMOps 平台

### 方案描述

多个团队都在接不同模型，希望建一个统一平台：模型网关、prompt 管理、日志、成本统计、eval。

### 初始方案

- 先做统一模型代理。
- 所有团队都改走新 SDK。
- 网关记录请求日志和 token。
- eval 后续再补。

### 架构师评审

| 维度 | 判断 |
|---|---|
| 是否适合平台化 | 适合，但要从最小治理闭环开始 |
| 主要风险 | 平台太大、团队绕过、日志敏感、eval 缺失、成本归因不准 |
| 最大缺口 | 只做 gateway 不等于 LLMOps，缺 release gate 和治理机制 |
| 是否准入 | 可以做 R2 平台 Pilot |

### 修改建议

- 先选 1-2 个业务应用作为 anchor tenant。
- 平台 MVP 只做：model gateway、prompt registry、trace、cost、eval gate。
- prompt 和模型变更必须记录版本和回滚点。
- trace 默认脱敏，敏感应用单独策略。
- eval 不后置，至少要接入关键 regression cases。
- 推广策略：接入更方便，而不是只靠强制规范。

### 面试表达点

> 我会避免一开始做“大而全 AI 平台”。LLMOps 的最小闭环是：统一调用、版本管理、trace、eval gate、成本归因和发布回滚。没有 eval 和 release gate 的模型网关，只是代理层，不是治理平台。

## 案例 4：数据分析 Agent

### 方案描述

业务希望用户自然语言提问，Agent 自动生成 SQL、查询数据、画图并解释指标。

### 架构师评审

| 维度 | 判断 |
|---|---|
| 是否适合 AI | 适合做分析辅助，但必须控制权限和指标口径 |
| 主要风险 | SQL 越权、指标口径错误、幻觉解释、数据泄露、查询成本爆炸 |
| 最大缺口 | 缺指标语义层、SQL sandbox、权限和审计 |
| 是否准入 | 只能先做只读、低风险数据集 Pilot |

### 修改建议

- 先建立 metric semantic layer，不让模型自由猜指标口径。
- SQL 生成后先 dry-run、成本估算和权限校验。
- 高成本查询需要确认或异步。
- 输出必须包含 SQL、数据来源、时间范围和口径说明。
- 禁止访问明细敏感数据，优先聚合结果。

## 评审训练方法

每次看到一个方案，先写三句话：

1. 这个场景最适合的 AI 模式是什么？
2. 这个方案最大的生产风险是什么？
3. 如果只能补一个上线门槛，先补什么？
