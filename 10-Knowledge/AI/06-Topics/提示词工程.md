# 提示词工程

## 这个主题是什么

`Prompt Engineering` 关注的是：如何通过 `instruction`、`role`、`examples`、输出格式约束和任务拆解，让模型更稳定地理解你想要它做什么。

它是大模型时代最早被广泛讨论的控制手段，所以很多人接触 agent 时，第一反应也是“是不是 prompt 写得不够好”。

## 为什么它仍然重要

- 它决定系统的基础行为边界：语气、目标、格式、禁区、优先级
- 它往往是最低成本、最快可迭代的控制层
- 在很多单轮任务、结构化输出任务、分类任务里，prompt 仍然非常有效

## 你先要抓住什么

- prompt engineering 解决的是“怎么说清楚”
- 它擅长控制任务说明、格式要求、few-shot examples 和高层行为规范
- 但它不等于整个 agent system 的控制能力

换句话说，prompt 是入口，不是全部系统。

## 常见组成

- `system prompt`：定义全局角色、边界和优先级
- `task prompt`：描述当前任务目标
- `few-shot examples`：给模型提供示例
- `output schema`：约束输出格式
- `decomposition hints`：提示模型先分析再执行

## 为什么后来大家觉得“不够了”

在 agent 系统里，失败往往不是因为一句 instruction 写得不漂亮，而是因为：

- 模型没看到对的上下文
- 没有拿到对的工具
- 状态没有被正确延续
- 规划循环没有被约束好
- 执行结果没有被反馈回下一轮

所以 agent 领域后来会自然从 `prompt engineering` 走向 [[上下文工程]]。

## 在 agent 时代的正确位置

- `Prompt Engineering` 仍然是最内层控制面之一
- 但它更像 `context stack` 里的一个部件
- 真正决定系统上限的，通常是 prompt 外面的上下文装配、工具边界、状态管理和 harness 设计

## 什么时候优先看 prompt

- 模型理解错任务
- 输出格式不稳定
- few-shot example 缺失
- 角色边界不清楚
- 简单任务本应可解，却表现波动很大

## 什么时候不要只盯着 prompt

- 系统已经涉及多步任务和 tool use
- 任务依赖外部知识、代码仓库、文件、数据库或 memory
- 失败来自动作执行、权限边界、结果反馈或状态丢失
- 你发现“换十次 prompt 也只是小修小补”
