# 自我进化与持续学习的记忆设计

## 为什么这页容易被聊飘

一说到 `self-evolving agent`，很多讨论会立刻跳到很宏大的叙事：

- agent 会不会自己优化自己
- 会不会自己长出新技能
- 会不会自己重写工作流

但真正落地的系统里，更常见、也更可靠的其实是三层：

1. `self-maintenance`
2. `memory consolidation`
3. `bounded self-improvement`

## 三个层次要分开

### 1. `self-maintenance`

系统通过 heartbeat、hooks、cron、background jobs 去：

- 巡检状态
- 补全日志
- 整理记忆
- 更新 checklist

这更像长期运维，不等于真正“进化”。

### 2. `memory-driven adaptation`

系统基于过去经验调整：

- recall policy
- default workflow
- task checklist
- routing rule
- prompt / instruction bundle

这已经接近准自进化，但仍然主要发生在 workspace / memory / policy 层。

### 3. `offline learning / model improvement`

通过人类反馈、synthetic data、preference optimization、eval loop 等，把经验沉淀回：

- prompts
- tools
- policies
- datasets
- 甚至 model weights

这才更接近真正的“持续学习系统”。

## 为什么 memory 是这条线的枢纽

因为很多所谓“自进化”，本质上不是模型在线变聪明，而是系统开始：

- 持续写入经验
- 整理有效经验
- 淘汰错误经验
- 调整后续任务的默认行为

也就是说，记忆系统如果设计得足够好，就会表现出：

- 更少重复犯错
- 更稳定的默认流程
- 更适合特定用户/项目/团队的行为

## OpenClaw 为什么在这里特别值得看

`OpenClaw` 的价值在于，它把这种“准自进化”做得非常具体：

- workspace files 是显式可编辑的
- `MEMORY.md` 与 daily memory logs 是 durable state
- hooks / heartbeat / cron 可以持续触发整理与维护
- 规则、checklist、memory、artifacts 可以被持续修订

这不是“内核自动进化”，但它非常接近：

`可持续自维护的 agent operating layer`

而 `Self-Improving-Agent` 进一步把这条线具象化成：

- `.learnings` learning ledger
- promotion 到 durable workspace files
- automatic skill extraction
- hook-driven reminders 和 error detection

## 真正可交付的持续学习循环长什么样

### 热路径

- 执行任务
- 产生 event / trace / artifact
- 在必要时写短期 state 或少量 durable memory

### 后台

- 抽取 semantic / episodic / procedural memory
- 去重、压缩、加 provenance
- 评估是否值得升级为稳定记忆

### 评测层

- 检查新记忆是否提升 success rate
- 检查是否引入污染、幻觉放大或不该共享的信息

### 治理层

- 高风险 memory update 进入 review gate
- 对影响 routing / tool use / policy 的变更保守放量

## 最容易踩的坑

- 把所有 reflection 都自动升级成长期记忆
- 没有 provenance，后面不知道一条记忆为什么存在
- 没有 forgetting / invalidation 机制
- 允许 agent 自由修改核心规则文件
- 没有 eval gate，就让新 memory 直接影响生产行为

## 一个更稳的表达

如果我们要避免吹得太大，我更推荐这样理解：

- `self-evolving` 太宽泛
- `memory-driven self-maintenance` 更接近现状
- `controlled self-improvement` 更接近工程可交付路径
