# 上下文工程

## 这个主题是什么

`Context Engineering` 关注的不是“怎么写一句更聪明的 prompt”，而是：

- 给模型什么信息
- 这些信息按什么顺序进入上下文
- 哪些工具、状态、历史、文件、记忆、检索结果要被带进来
- 哪些不该进来

一句更工程化的话是：它是在设计模型的工作上下文，而不是只设计模型的指令文本。

## 为什么它在 agent 时代变得更重要

agent 一旦进入真实任务，系统要处理的就不只是 instruction，而是整个工作环境：

- 当前任务目标
- 用户身份与权限
- 历史步骤
- 文件、代码仓库、网页、数据库
- memory 和 session state
- 可调用工具与工具返回
- 预算、时间、风险边界

所以很多 agent 的上限，不是卡在 prompt，而是卡在 context assembly。

## 你先要抓住什么

- prompt engineering 是 context engineering 的一个子集
- context engineering 决定模型是否“看到对的世界”
- 很多 agent 的表现差异，根源不是模型差异，而是上下文装配差异

## 一个成熟 context stack 往往包含什么

- `instruction layer`：system / developer / task prompt
- `state layer`：session、步骤状态、最近动作
- `knowledge layer`：检索结果、代码、文档、表格、业务记录
- `tool layer`：可用工具、schema、权限说明
- `memory layer`：长期记忆、偏好、约束、历史模式
- `governance layer`：预算、审批、风险控制、禁止动作

## 为什么“less is more”经常成立

上下文不是越长越好。太多无关上下文会带来：

- 注意力被稀释
- 关键信号被淹没
- 工具选择变差
- 决策链变得更脆弱
- latency 和 cost 上升

所以 context engineering 的关键，不只是“加信息”，而是“筛信息、排序信息、刷新信息”。

## 和 prompt engineering 的关系

可以把它理解成三层：

1. `Prompt Engineering`：把话说清楚
2. `Context Engineering`：把模型放进正确环境
3. `Harness Engineering`：把整个 agent 的工作台、反馈回路和执行边界搭起来

## 在 agent 里的典型问题

- RAG 结果该放多少
- 代码仓库上下文怎么裁剪
- 历史对话保留多少轮
- 工具返回是原样塞回去，还是先结构化摘要
- memory 什么时候写入、什么时候召回
- approval 和 policy 信息放在显式 context，还是放在系统执行层

## 什么时候说明你正在做 context engineering

- 你在设计 retrieval strategy
- 你在裁剪 repo context
- 你在做 session / memory packing
- 你在决定 tool schema 与返回格式
- 你在搭“任务状态 + 文件 + 工具 + 历史”的工作集
