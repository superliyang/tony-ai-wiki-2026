# 大模型与 AI 全景题库

> 这组题不是替代前面的专项题库，而是专门训练“从底层原理一路讲到工程落地”的综合能力。
> 适合用在架构师、技术负责人、AI 平台、Agent 系统、应用负责人岗位的压轴准备。

## Q1. 如果让你从 0 到 1 解释大模型技术栈，你会怎么讲？

- 考察点：是否能建立从原理到业务的完整分层。
- 回答骨架：底层原理 -> 数据与训练 -> 后训练 -> 推理服务 -> RAG/Agent -> Eval/Release -> Security/Governance -> Product/ROI。
- 追问：哪一层最容易被普通候选人忽略？

## Q2. Transformer、Pretraining、SFT、RLHF/DPO 之间是什么关系？

- 考察点：是否能把模型架构和训练流程讲成一条能力形成链。
- 回答骨架：Transformer 是架构；pretraining 形成通用能力；SFT 学会指令跟随；RLHF/DPO 调整偏好和行为边界。
- 追问：为什么后训练不能替代高质量 pretraining 数据？

## Q3. 为什么大模型会 hallucinate？工程上如何处理？

- 考察点：是否能从原理和系统两侧解释问题。
- 回答骨架：概率生成、训练分布、缺少真实检索、目标函数不等于事实性；工程上用 RAG、引用、校验、eval、human review、risk-tier fallback。
- 追问：RAG 为什么不能彻底解决 hallucination？

## Q4. 你如何判断一个模型是否适合生产系统？

- 考察点：是否能超越 benchmark，看真实任务适配。
- 回答骨架：任务完成率、事实性、工具调用可靠性、延迟、成本、上下文、稳定性、安全、供应商风险、回归表现。
- 追问：强模型为什么可能不适合某个生产场景？

## Q5. 如果一个 AI 系统效果差，你会从哪些层排查？

- 考察点：是否有分层排障能力。
- 回答骨架：需求定义 -> 数据/知识源 -> prompt/context -> retrieval -> model routing -> tool execution -> eval/metrics -> user workflow -> governance。
- 追问：怎么判断是模型问题还是数据问题？

## Q6. Pretraining、RAG、Fine-tuning、Memory 分别解决什么问题？

- 考察点：是否能区分能力内化、外部知识、场景适配和长期状态。
- 回答骨架：pretraining 学通用能力；RAG 引入外部事实；fine-tuning 改变行为/任务分布；memory 保留跨会话状态和偏好。
- 追问：企业知识库优先 RAG 还是 fine-tuning？

## Q7. 大模型系统和传统软件系统最大的工程差异是什么？

- 考察点：是否理解非确定性、评测和治理差异。
- 回答骨架：输出概率性、质量难定义、版本变化影响大、数据/模型/prompt/tool 多因素耦合、需要 eval harness 和 release gate。
- 追问：传统 SRE 指标为什么不够？

## Q8. 你如何设计一个端到端 AI 应用架构？

- 考察点：开放式架构设计能力。
- 回答骨架：业务目标 -> 用户/权限 -> 数据与知识 -> model routing -> RAG/agent/workflow -> guardrail -> eval -> observability -> rollout -> incident。
- 追问：如果场景从客服变成 coding agent，哪几层会变？

## Q9. Agent 为什么难做稳定？你会如何治理？

- 考察点：是否理解 agent 的执行风险。
- 回答骨架：planning 不稳定、tool schema 漂移、环境状态变化、循环调用、权限风险、不可复现；治理靠 workflow boundary、tool gateway、sandbox、approval、trace、task-level eval。
- 追问：什么时候不应该用 multi-agent？

## Q10. 从 Demo 到生产，一个 AI 项目最关键的门槛是什么？

- 考察点：是否能区分 demo success 和 production readiness。
- 回答骨架：场景边界、golden set、bad case regression、SLO、成本预算、安全审查、灰度、回滚、人工兜底、owner 机制。
- 追问：哪个门槛不过你会坚决不上线？

## Q11. 如果模型升级后 benchmark 变好，但线上投诉增加，你怎么处理？

- 考察点：是否能处理指标冲突和回归。
- 回答骨架：先止血回滚或降级；拆投诉类型；对比流量、人群、任务、prompt、routing；补充 regression；重新灰度。
- 追问：如何防止以后再次发生？

## Q12. 你如何向非技术管理层解释 AI 系统健康度？

- 考察点：是否能把技术健康翻译成业务语言。
- 回答骨架：质量、效率、成本、风险四象限；配合任务完成率、人工接管率、节省时间、投诉率、单位成本、安全事件。
- 追问：如果只能保留三个指标，你选什么？

## Q13. AI 项目如何证明 ROI？

- 考察点：是否理解业务闭环。
- 回答骨架：先定义 workflow fit，再定义节省时间、提升转化、降低错误、减少人力、缩短周期；扣除模型、工程、运维、审核、培训成本。
- 追问：adoption 高但 ROI 不明显怎么办？

## Q14. AI 安全和 AI 质量是什么关系？

- 考察点：是否能把安全从附属项提升为上线条件。
- 回答骨架：质量解决是否有用，安全解决是否可用、可控、可负责；高风险动作需要权限、审批、审计、回滚；安全事件会反过来摧毁 adoption。
- 追问：安全门槛和体验冲突时怎么权衡？

## Q15. 你如何判断一个候选人是真懂 AI，还是只会调 API？

- 考察点：元认知和面试官视角。
- 回答骨架：看他能否解释能力来源、失败根因、系统分层、取舍、评测、事故、业务闭环；只会调 API 的人通常讲不出边界、指标和回滚。
- 追问：你会设计哪三道题筛选？

## Q16. 如果你负责一家公司 AI 能力建设，第一年路线图怎么排？

- 考察点：战略、平台和落地综合能力。
- 回答骨架：先选高价值低风险场景；建 shared platform、model gateway、RAG/data layer、eval/release gate、安全治理；再做试点、复制、平台化。
- 追问：如何避免团队各自为战？

## Q17. 训练、推理、RAG、Agent、Eval、Governance 里，哪个最决定企业 AI 成败？

- 考察点：是否能做情境化判断，而不是绝对化。
- 回答骨架：研究型公司看训练；平台型看推理和 eval；企业应用看 RAG、workflow 和 governance；agent 公司看 execution runtime 和 task eval。
- 追问：你目标岗位最应该强调哪一层？

## Q18. 你如何把 AI 经验沉淀成团队机制？

- 考察点：是否有 senior owner 感。
- 回答骨架：沉淀 prompt/data/model/tool 版本规范、eval suite、release gate、incident runbook、设计评审、复盘模板和健康度看板。
- 追问：怎么让新团队成员复用，而不是继续靠老员工口口相传？

## Q19. 一个 AI 系统最常见的三类失败是什么？

- 考察点：是否有事故分类能力。
- 回答骨架：事实错误 / 任务失败 / 风险越界；分别用 RAG+引用+复核、workflow+tool eval+fallback、permission+approval+audit 治理。
- 追问：哪类失败最危险但最容易被 demo 忽略？

## Q20. 如果让你现场白板讲“AI 系统从底层到落地”，你怎么组织？

- 考察点：综合表达能力。
- 回答骨架：先画能力形成链，再画系统执行链，再画 eval/release 链，最后画业务 rollout 链。
- 追问：如果时间只剩 2 分钟，你保留哪三层？
