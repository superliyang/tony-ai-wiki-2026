# 什么时候该看人物、组织、系统与案例

## 一句先判断

你不是每次都该先看“主题页”。

很多 AI 问题更适合先判断自己缺的是：

- `方向判断`：看人物
- `组织能力`：看组织 / 公司
- `能力表面与产品形态`：看系统
- `真实落地与失败边界`：看案例

## 最实用的判断表

| 你当前问的问题 | 更该先看什么 | 第二步看什么 | 为什么 |
| --- | --- | --- | --- |
| 为什么这家公司/实验室总在押某条路 | `人物` | `公司 / 组织` | 人物最能解释路线、口径和优先级 |
| 这个组织真正擅长什么、边界在哪 | `组织 / 公司` | `系统 / 项目` | 组织比单一产品更能解释长期能力 |
| 某个产品或 runtime 到底是什么 | `系统` | `工程 / 主题` | 系统页最适合看能力边界和定位 |
| 这东西能不能在真实组织里落地 | `案例` | `应用主题 / 治理` | 案例最能看 rollout、adoption 和失败模式 |
| 某个开源项目值不值得跟、值不值得 adopt | `组织` | `项目 / 模式 / 案例` | 先看维护者和生态位置，再看技术细节 |
| 为什么大家都在争某个方向 | `人物` | `系统 / 案例` | 争论先发生在立场和判断，不只是产品形态 |

## 四种资产各自最擅长回答什么

### 人物

适合回答：

- 谁在定义叙事
- 谁在代表某种研究或产品路线
- 为什么外部世界会把某家公司理解成今天这样

不要单独用来回答：

- 产品到底怎么做
- 组织到底怎么 rollout

### 组织 / 公司

适合回答：

- 长期能力是什么
- 生态位置是什么
- 一个系统背后靠的不是单点产品，而是什么组织能力

不要单独用来回答：

- 真实上线效果
- 具体执行闭环

### 系统

适合回答：

- 具体能力边界是什么
- 这个入口到底是 chat、runtime、tooling 还是 observability
- 哪个系统该拿来做样本比较

不要单独用来回答：

- 一个组织为什么会做成这样
- 大规模 rollout 为什么成功或失败

### 案例

适合回答：

- 真实任务边界是什么
- 组织 adoption 为什么成功
- 风险、责任、治理和 release gate 怎么出现

不要只把它当成：

- 产品宣传页
- ROI 神话

## 更稳的连读顺序

### 想理解“路线为什么不同”

1. 人物
2. 公司 / 组织
3. 系统
4. 案例

### 想理解“这东西能不能做成”

1. 案例
2. 应用主题 / 工作流
3. 系统
4. 工程 / 治理

### 想理解“这个开源栈值不值得追”

1. 组织
2. 项目
3. 模式
4. adoption case

## 最常见的 4 个误区

### 误区 1：人物页等于八卦页

不是。

人物页真正有价值的地方，是帮你理解研究路线、产品口径和产业判断。

### 误区 2：系统页可以替代案例页

不行。

系统告诉你“它是什么”，案例才告诉你“它在现实里怎么出问题、怎么产生价值”。

### 误区 3：案例页就是宣传页

也不对。

好的案例页应该帮你提炼任务边界、组织条件、控制点和迁移启发。

### 误区 4：组织页和人物页重复

不重复。

人物解释判断，组织解释能力，系统解释表面，案例解释现实。
