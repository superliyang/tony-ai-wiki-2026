## 2026-06 Refresh Lens

Agent evaluation 需要从“最终回答好不好”扩展成多层评测：

| Layer | 关注点 | 典型失败 | 产物 |
|---|---|---|---|
| Black-Box | final response 是否正确、有用、安全 | 答案看似合理但事实错、遗漏约束、越权建议 | rubric、golden answer、judge prompt |
| Glass-Box | trajectory、tool use、retrieval path、state transition 是否合理 | 调错工具、重复调用、上下文污染、没有检查结果 | trace review、trajectory diff、tool-use score |
| White-Box | 单个工具参数、schema、policy rule 是否合规 | 参数错、路径错、权限错、危险命令未拦截 | schema tests、policy unit tests |
| Release Gate | 是否允许上线、扩大权限或降低人工 review | 回归、成本失控、权限扩大后缺少回滚 | regression suite、approval rule、rollback rule |
| Runtime Monitor | 线上运行是否漂移 | latency/cost 增长、失败率升高、用户修正变多 | trace dashboard、alerts、sample review |

详见 [[10-Knowledge/AI-Engineering/08-Maps/2026-06 AI Engineering 近期变化雷达]]。

## Evaluation Design Checklist

| Step | 问题 |
|---|---|
| 1. 定义任务边界 | 这个 agent / workflow 到底应该完成什么？哪些事情明确不做？ |
| 2. 定义 failure mode | 最怕事实错、越权、漏步骤、成本高、慢、还是不可回滚？ |
| 3. 选择评测层级 | 只测最终答案，还是必须测 trajectory、tool args、release gate？ |
| 4. 构造样本 | 用真实历史任务、合成边界案例、回归 bug、红队样本 |
| 5. 接入 trace | 每次工具调用、输入输出、成本、人工改动都要能回看 |
| 6. 设 release gate | 什么时候可以进入自动化？什么时候必须人工 review？ |
| 7. 定期刷新 | 模型、工具、prompt、权限变了以后，eval 必须重新跑 |
