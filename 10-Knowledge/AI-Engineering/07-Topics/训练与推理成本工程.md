# 训练与推理成本工程

## 为什么这页要单独存在

很多团队一说“AI 成本”，马上想到 token price。

但真正的工程成本通常至少包括：

- 训练 GPU / TPU 成本
- 推理 GPU-second 或 instance-hour 成本
- 存储、日志、checkpoint、cache 成本
- 评测与重跑成本
- 人审、运营和安全复核成本

所以成本工程不是财务附录，而是架构设计的一部分。

## 训练成本和推理成本长得完全不同

### 训练成本

更像大额、集中、计划型投入：

- 预训练或 post-training 的大批量算力消耗
- 数据处理、清洗、标注和 synthetic pipeline
- checkpoint、恢复、失败重跑
- 大规模实验对比

### 推理成本

更像长期、连续、业务驱动型支出：

- 每请求 token 消耗
- GPU 占用时间
- cache 命中率
- 长上下文导致的集群拖累
- 多 tenant 的 tail latency 保护成本

## 训练成本最常被低估的地方

### 1. 重跑成本

没有稳定 checkpoint / resume 时，一次失败不只是掉线，而是把实验资金直接烧掉。

### 2. 数据成本

- 清洗
- dedup
- 标注
- 偏好数据收集
- synthetic data 过滤

这些都经常比团队最初估计更贵。

### 3. 评测成本

每一轮 post-training 如果没有固定 eval gate，就会用更多试错来换“感觉更好”。

## 推理成本最常被低估的地方

### 1. Prefill 很贵

长上下文场景里，真正拖累系统的往往不是 decode，而是 prefill。

### 2. 低 cache 命中率

prefix reuse 不足时，系统会重复做昂贵的 prompt computation。

### 3. 不合适的服务层级

把稳定高负载放在 serverless，或把轻量波动流量放在 dedicated，都会让成本曲线失真。

### 4. 安全和复核不是免费

更严格的 classifier、guardrails、人审和回放系统，都会增加单位任务成本，但经常是必要成本。

## 工程上最值得看的 5 个成本杠杆

### 1. 模型选择

- 更小模型
- mixture routing
- task-specific model selection

### 2. 上下文设计

- prompt 压缩
- retrieval 去冗余
- memory 分层
- 对长上下文做缓存和重用

### 3. runtime 优化

- KV cache
- batching
- prefill / decode 分离
- router 和 data plane 优化

### 4. 服务层级设计

- serverless
- on-demand / dedicated
- reserved capacity
- batch / flex / async tiers

### 5. 流程设计

- 是否必须同步
- 是否必须高阶模型全程参与
- 是否可以先筛后审
- 是否可以 nightly / batch 化

## 成本判断时最容易犯的错

### 只看单位 token 价格

如果瓶颈来自 idle dedicated GPU、低 cache 命中率或 tail latency 保护，盯 token 价会误导判断。

### 只看平均延迟

很多系统真正贵在 tail，而不是 mean。

### 只看模型成本，不看工作流成本

agent 系统里，tool retries、approval loop、browser actions 经常比一次纯文本调用更贵。

## 一个更实用的成本框架

可以按三层看：

### Request-level cost

- tokens
- runtime duration
- cache hit / miss
- tool calls

### Workflow-level cost

- retries
- approval / review
- async execution
- failure recovery

### Fleet-level cost

- utilization
- reserved capacity
- autoscaling waste
- storage / logging / eval infrastructure

## 学这一页最该形成的判断力

### 判断 1：当前优化目标是降 request 成本，还是降 fleet 成本

两者做法不一样。

### 判断 2：这是模型太贵，还是系统太浪费

例如：

- cache 策略差
- routing 差
- dedicated 容量闲置
- batch 和 async 没用起来

### 判断 3：安全成本是不是必要成本

很多所谓“昂贵控制点”其实是在避免更高代价事故。
