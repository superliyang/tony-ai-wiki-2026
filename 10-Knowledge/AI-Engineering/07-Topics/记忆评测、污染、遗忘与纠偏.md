# 记忆评测、污染、遗忘与纠偏

## 为什么 memory 需要单独评测

因为 memory 系统的危险在于：

- 它一开始常常“看起来更聪明”
- 但错误一旦持久化，后面每轮都会放大

所以 memory 不能只看 recall 成功率，还得看：

- 是否提升 success rate
- 是否降低重复错误
- 是否增加幻觉自强化
- 是否引入权限越界或不该共享的信息

## 四类最常见的 failure modes

### 1. `pollution`

临时猜测、错误总结、prompt injection 结果被写成长期记忆。

### 2. `staleness`

记忆曾经正确，但现在已经过期。

### 3. `overshadowing`

召回了太多旧记忆，压过当前任务真正相关的信息。

### 4. `cross-scope leakage`

本该属于某个用户、项目、repo 的 memory 被别的 scope 误读。

## 评测 memory 时至少看什么

### 任务指标

- task success rate
- retry rate
- tool misuse rate
- recovery success rate

### 召回指标

- relevant recall ratio
- irrelevant recall ratio
- recall token budget
- summary drift

### 治理指标

- human correction rate
- stale memory hit rate
- unsafe write rejection rate
- cross-scope recall incidents

## forgetting 不是可有可无

如果系统只有写入和召回，没有 forgetting，就很难长期稳定。

forgetting 常见做法：

- TTL / expiration
- invalidation
- human revoke
- versioned memory replacement
- stale candidate demotion

## 纠偏要有来源，不要只靠覆盖

一条错误记忆最稳的修复方式通常不是“再写一条新的覆盖它”，而是：

- 标记 provenance
- 撤销旧版本
- 写入修正版本
- 在 recall 时优先新版本

## 一个很重要的工程判断

记忆系统越接近：

- routing
- tool use
- safety policy
- personalized defaults

就越应该被当成 `production control surface`，而不是普通笔记系统。
