# 第 4 章实操：微调评测与 Failure Analysis

## 本章目标

这一章不再满足于“我跑了一次 LoRA”。我们要把一次微调实验变成一套可解释、可复盘、可迭代的工程闭环。

## 核心原则

你这章真正要练成的，不是训练命令，而是下面这条链：

- 任务定义
- baseline
- 数据集
- 微调
- post-eval
- failure analysis
- 下一轮改进假设

只要这条链跑通，你的能力就从“会调模型”升级成“会做模型实验”。

## 推荐实验范围

在 `MacBook Pro Max M4` 上，建议你选一个很小、目标非常明确的任务。

例如：

- 把一段技术说明改写成更简洁的风格
- 做一个小型 FAQ 回答任务
- 让模型稳定输出某种结构化格式
- 做一个词汇或术语风格适配任务

不要一开始选：

- 通用聊天增强
- 很大的多任务数据集
- 目标边界模糊的“提升整体智能”

## 推荐实验流程

### 1. 定义任务与成功标准

你至少要先写清楚：

- 任务是什么
- 为什么需要微调，而不是 prompt 就够了
- 你希望改善哪种行为
- 成功的判定标准是什么

例如：

- 输出格式更稳定
- 某类术语使用更一致
- 某一类问答更少跑偏

### 2. 先做 baseline

在微调前，先保留一组固定样本，跑一次基线结果。

建议至少准备：

- `10-30` 条评测样本
- 覆盖正常样本
- 覆盖边界样本
- 覆盖容易失败的样本

记录：

- 模型版本
- prompt 模板
- 输出结果
- 你给它的人工评分

### 3. 准备最小数据集

这一章重点不是“大”，而是“干净”。

数据集建议：

- 规模先小
- 格式统一
- 目标清晰
- 尽量不要同时混很多风格或任务

你要特别警惕：

- label 不一致
- 样本风格冲突
- 数据里本身就有大量噪声

### 4. 跑一次最小 LoRA / SFT

你在这一步要保留这些实验信息：

- 基础模型
- adapter / LoRA 配置
- epoch 或 step
- learning rate
- batch size
- 是否使用 quantization
- 训练时长

### 5. 做 post-eval

用和 baseline 同一组测试样本再跑一遍。

比较：

- 哪些样本真的变好了
- 哪些样本没变化
- 哪些样本反而退化了

### 6. 写 failure analysis

这一节最值钱。

你至少把失败分成这几类：

- 任务没学会
- 风格学歪了
- 格式更稳定了，但事实性变差了
- 某些边界输入变脆弱了
- 输出更像训练集，但泛化能力更差了

## 建议的评测表格

```text
Sample ID:
Task type:
Baseline output:
Tuned output:
Win/Lose/No change:
Observed failure:
Possible reason:
Next fix hypothesis:
```

## 你最该带走的能力

完成这章后，你应该能回答：

- 这次微调到底有没有价值
- 它改善的是哪一类行为
- 它破坏的是哪一类行为
- 下一轮该改数据、改 prompt、改训练配置，还是干脆不该微调

## 本章输出物

- 一份 baseline 数据集
- 一份 post-eval 对比表
- 一份 failure analysis 文档
- 一份下一轮改进假设清单

## 常见误区

- 没 baseline 就直接训练
- 只挑好样本看结果
- 不记录坏例子
- 把一次“主观感觉变好了”误以为实验成功
