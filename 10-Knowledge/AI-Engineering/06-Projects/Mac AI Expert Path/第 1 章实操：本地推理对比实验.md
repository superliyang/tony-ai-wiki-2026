# 第 1 章实操：本地推理对比实验

## 本章目标

这一章不是让你“跑一个模型就结束”，而是让你真正建立本地推理的判断力。

你要比较三条路线：

- `Ollama`
- `llama.cpp`
- `MLX-LM`

最终输出不是一句“哪个好”，而是一张你自己的 runtime 判断表。

## 你要回答的 5 个问题

1. 哪条路线最容易上手？
2. 哪条路线最适合理解底层推理？
3. 哪条路线最适合 Apple silicon 原生实验？
4. 哪条路线最适合做后续本地应用原型？
5. 在你的机器上，速度、内存、体验分别怎么取舍？

## 建议对比维度

### 1. 安装和启动成本

记录：

- 从零到跑通需要多久
- 出错率高不高
- CLI / Python 集成是否顺手

### 2. 模型加载体验

记录：

- 模型格式
- 下载大小
- 首次加载时间
- 二次启动体验

### 3. 生成性能

记录：

- 首 token 延迟
- tokens/sec
- 长上下文时是否明显变慢

### 4. 内存与稳定性

记录：

- 是否容易把机器压满
- 是否容易出现卡顿或退化
- 多轮对话时稳定性如何

### 5. 工程可扩展性

记录：

- 是否容易接 API
- 是否容易接 Python 应用
- 是否适合后面做 RAG / agent

## 实验建议

### 实验 A：同一任务，三条 runtime 全部重放

固定一个 prompt，例如：

- 总结一段技术文档
- 解释一个 Python 函数
- 生成一个简短计划

在三条路线里都跑一遍。

### 实验 B：同一模型，不同量化重放

如果条件允许，观察不同量化下：

- 速度差异
- 内存差异
- 主观质量差异

### 实验 C：长一点的上下文任务

例如：

- 输入较长技术文档
- 让模型先总结，再回答追问

这样你能观察 context 带来的压力。

## 你大概率会得出的结论

通常会是这样的结构：

- `Ollama`：最快建立使用直觉
- `llama.cpp`：最适合理解推理和量化底层
- `MLX-LM`：最适合 Apple silicon 原生路线和后续实验

但重点不是记这个结论，而是你要在你自己的机器上验证它。

## 本章输出物

- 一张三路 runtime 对比表
- 一份你自己的“默认选择规则”
- 一份后续项目的 runtime 选择建议

## 常见误区

- 只看速度，不看集成体验
- 只看主观质量，不记录 context / memory 压力
- 比较时模型、量化、prompt 都不一致，导致结论无效

## 建议的默认收敛

如果你只是开始学习：

- `Ollama` 做最快的上手入口
- `llama.cpp` 做底层理解入口
- `MLX-LM` 做 Apple silicon 深入实验入口
