## 先问 8 个问题

1. 主要消费方是谁：BI、运营、风控、AI、数据科学，还是业务系统
2. 数据延迟要求是多少：小时、分钟、秒级，还是毫秒级
3. 数据是否需要回放、审计和重算
4. 主要数据形态是什么：结构化、日志、事件、文档、多媒体
5. 团队更强的是平台工程、SQL 分析、还是业务数据建模
6. 是否已有云厂商、warehouse、lake、Kafka 或 BI 生态包袱
7. 最怕的问题是什么：成本、延迟、质量、权限、lock-in，还是运维复杂度
8. AI / agent / eval / RAG 是否已经是明确需求

## 决策 1：先选主架构模式

### Warehouse-first

选择条件：

- 核心是 BI / SQL / 报表
- 希望低运维
- 组织主要围绕分析师和 dashboard 运转
- 指标治理比多格式原始数据更急

常见候选：

- Snowflake
- BigQuery
- Redshift
- Microsoft Fabric / Synapse

### Lakehouse-first

选择条件：

- 多格式数据多
- AI / ML / 数据科学强
- 希望开放存储和多引擎
- 团队有平台能力

常见候选：

- Databricks
- Iceberg / Delta / Hudi + Spark / Trino
- cloud object storage + catalog

### Streaming-first

选择条件：

- 低延迟业务动作是核心
- 多消费者需要同一事件轨道
- 需要实时特征、风控、运营或监控

常见候选：

- Kafka / Confluent
- Flink
- cloud pub/sub / streaming services

### Hybrid

选择条件：

- BI、实时、AI、审计和业务系统都要
- 企业级平台要服务多个团队
- 单一架构模式已经不够

常见组合：

- Kafka + Flink + Lakehouse + Warehouse + Semantic Layer + Governance

## 决策 2：托管还是自建

优先托管：

- 团队小
- 业务变化快
- 运维不是核心竞争力
- 希望快速建立稳定分析能力

优先自建或半自建：

- 数据规模极大
- 成本优化空间大
- 有强平台团队
- 需要开放格式、多引擎或深度定制

最常见答案：

- 核心数据模型和治理自控
- 底层计算 / 存储 / streaming 适度托管
- 对 lock-in 敏感的层尽量使用开放格式和标准接口

## 决策 3：先补技术还是治理

先补技术：

- 数据算不动
- 延迟不达标
- 存储和计算瓶颈明显
- 可靠性不足

先补治理：

- 同一个指标多套口径
- 数据没人信
- 权限不清楚
- owner 不明确
- 事故无法追溯

常见误区：用新平台掩盖旧口径问题。

## 决策 4：AI 需求如何影响架构

如果已经有 AI / agent / RAG 需求，必须额外关注：

- dataset versioning
- document metadata
- vector index freshness
- eval dataset isolation
- prompt / dataset / model / index 版本绑定
- PII 和权限传播
- feedback / trace / incident 回流

这会让 lakehouse、catalog、semantic layer、observability 的重要性上升。

## 决策 5：如何避免工具堆叠

每引入一个产品，问：

1. 它替代哪一层
2. 它是否已有现有系统覆盖
3. 它是否带来新元数据孤岛
4. 它的 owner 是谁
5. 它的失败模式是什么
6. 它的成本如何归因
7. 它如何接入治理控制面
