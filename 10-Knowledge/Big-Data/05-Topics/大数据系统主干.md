# 大数据系统主干

## 这页解决什么问题

大数据最容易学散，因为工具名很多：Hadoop、Spark、Flink、Kafka、Hive、Iceberg、Airflow、dbt、Trino、Snowflake、Databricks。

更稳的学法不是先背工具，而是先看一条数据链路到底要过哪些关：

`产生 -> 采集 -> 传输 -> 存储 -> 计算 -> 建模 -> 服务 -> 治理 -> 反馈`

## 一条最短主干

### 1. 数据产生：业务事件与事实源

先问：

- 数据来自用户行为、交易、设备、日志、第三方系统，还是人工录入
- 它是 event、state、document，还是 metric
- 业务事实源是谁，是否允许后续被修正

这一层决定后面能不能追责、回放和统一口径。

### 2. 数据采集：ingestion 与 CDC

采集层解决的是“数据怎么可靠进入平台”：

- batch import
- log collection
- API / file ingestion
- CDC
- event streaming

关键不是“进来了没有”，而是 schema、顺序、重复、延迟和失败重试怎么处理。

### 3. 数据传输：queue、stream 与事件平台

传输层解决的是“生产者和消费者如何解耦”：

- Kafka 类系统承载事件日志和 stream
- queue 更偏任务分发和削峰
- event platform 更关注 schema、topic、权限和 replay

核心判断：这条链路需要的是消息送达，还是可回放的事件历史。

### 4. 数据存储：warehouse、lake、lakehouse

存储层解决的是“数据以什么形态长期存在”：

- `warehouse`：强 SQL、强治理、强分析体验
- `data lake`：低成本、多格式、原始数据沉淀
- `lakehouse`：在开放存储上加入表格式、事务、catalog 和治理能力

不要把 storage 当硬盘看，它会决定数据能否被安全复用。

### 5. 数据计算：batch、streaming、interactive

计算层解决的是“什么时候、以什么代价把数据变成结果”：

- `batch`：稳定、可回放、适合大规模离线计算
- `streaming`：低延迟、状态连续、适合实时业务动作
- `interactive`：低等待、适合探索分析和 ad hoc 查询

多数真实系统不是二选一，而是 batch 做校准和回补，streaming 做实时动作。

### 6. 数据建模：从原始数据到可解释指标

建模层解决的是“数据如何变成可被组织共同理解的语言”：

- ODS / DWD / DWS / ADS
- 维度建模
- metric definition
- semantic layer
- data mart / data product

如果这一层弱，平台越大，争论越多：同一个指标会有多个版本。

### 7. 数据服务：BI、API、feature、vector 与 AI

服务层解决的是“数据最终被谁使用”：

- BI dashboard
- OLAP / ad hoc query
- reverse ETL
- feature store
- vector store
- eval dataset
- business API

不同服务形态对 freshness、latency、权限和质量的要求完全不同。

### 8. 数据治理：quality、lineage、owner、access

治理层解决的是“组织为什么能信这批数据”：

- data quality
- lineage
- catalog
- owner
- data contract
- access control
- privacy / compliance

治理不是文档工作，而是让数据平台可持续运行的控制面。

### 9. 数据反馈：成本、SLA 与业务闭环

最后要问：

- 哪些链路最贵，是否值得
- 哪些指标真的进入业务决策
- 哪些数据产品没人用，应该下线
- 哪些质量事故应该反哺成规则、测试和 owner 机制

大数据平台不是越大越好，而是越能支持可信决策越好。

## 最常见的 5 个误区

### 误区 1：大数据等于 Spark / Flink

Spark / Flink 是计算层，不是完整数据系统。

### 误区 2：实时化一定更高级

实时化会带来状态、一致性、回放、成本和运维复杂度。不是业务强依赖低延迟，就不要为了实时而实时。

### 误区 3：有了 lakehouse 就不用治理

Lakehouse 只提供更好的技术承载，口径、owner、质量和权限仍然要设计。

### 误区 4：数据平台只服务 BI

现代数据平台还服务 AI、风控、推荐、运营自动化和业务系统动作。

### 误区 5：先上工具再补模型

工具可以提高效率，但不能替代数据模型、指标定义和组织责任。

## 一句话记忆

大数据不是“存很多数据”，而是把复杂数据流变成可复用、可信、可治理、可行动的组织能力。
