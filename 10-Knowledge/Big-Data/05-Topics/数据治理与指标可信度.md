# 数据治理与指标可信度

## 这页解决什么问题

很多数据平台的问题，不是“算不出来”，而是：

- 算出来没人信
- 同一个指标有多个版本
- 出问题不知道是谁负责
- 数据质量事故只能靠人肉排查

数据治理的核心目标不是合规文档，而是让组织能够信任、复用和追责数据。

## 指标可信度从哪里来

### 1. Definition：定义清楚

每个核心指标都要回答：

- 业务含义是什么
- 计算公式是什么
- 过滤条件是什么
- 时间口径是什么
- 是否含退款、取消、测试数据、异常数据

如果定义不清，后面所有自动化都会放大混乱。

### 2. Owner：责任清楚

指标和数据表需要 owner。

owner 不只是“联系人”，而是负责：

- 口径解释
- schema 变更
- 质量问题
- 下游影响
- 废弃与迁移

没有 owner，平台会变成无人负责的公共垃圾场。

### 3. Lineage：血缘清楚

Lineage 要回答：

- 这个指标从哪些源表来
- 中间经过哪些任务
- 下游有哪些看板、API、模型和业务流程
- 上游变化会影响谁

血缘的价值是影响评估和故障定位，不只是画图。

### 4. Quality：质量可检测

常见质量规则：

- completeness：是否缺数据
- freshness：是否及时
- uniqueness：是否重复
- validity：格式和取值是否合法
- consistency：跨表是否一致
- distribution drift：分布是否异常变化

质量规则最好和业务语义绑定，而不只是技术校验。

### 5. Contract：变更有契约

Data contract 让生产方和消费方明确：

- schema 能不能改
- 字段含义能不能变
- 延迟 SLA 是什么
- 数据质量底线是什么
- 破坏性变更如何通知和迁移

这对跨团队数据平台尤其重要。

### 6. Access：权限和隐私受控

治理还要回答：

- 谁能看哪些数据
- 敏感字段如何脱敏
- 数据跨境、留存和删除规则是什么
- AI / 分析 / 运营使用数据是否有边界

权限不是阻碍使用，而是让数据可以被安全复用。

## 一个指标可信度检查表

对任何核心指标，问：

1. 有没有唯一业务定义
2. 有没有 owner
3. 有没有权威数据源
4. 有没有血缘
5. 有没有质量规则
6. 有没有 SLA
7. 有没有变更通知机制
8. 有没有下线策略

如果这些都没有，就不要急着做更多看板。

## 治理和平台的关系

治理不应该只靠会议和文档。

它应该沉到平台里：

- catalog 让数据可发现
- lineage 让影响可追踪
- quality checks 让问题可提前发现
- semantic layer 让指标口径可复用
- access control 让权限可执行
- cost attribution 让成本可管理

## 和 AI 数据的关系

AI 系统会放大数据治理问题：

- RAG 需要可信事实源
- eval dataset 需要版本和代表性
- feature 需要 freshness 和训练 / 推理一致性
- agent 需要知道哪些数据能用、能不能行动

所以大数据治理不是传统 BI 遗留问题，而是 AI 系统可靠性的前置条件。
