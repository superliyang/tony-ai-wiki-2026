# 数仓建模与指标口径

## 这页解决什么问题

很多大数据平台“技术上很强”，但业务仍然不信数据。

常见原因不是 Spark 慢、Kafka 不稳，而是：

- 同一个指标有多个口径
- 原始数据没有分层
- 业务事实和分析宽表混在一起
- 看板、实验、AI 数据集各算各的

数仓建模的核心是把数据变成组织能共同理解、复用和追责的业务语言。

## 一条最短分层

### 1. ODS：原始贴源层

ODS 保存接近源系统的原始数据。

作用：

- 保留事实源快照
- 支持回放和重算
- 减少对业务库的直接依赖

原则：

- 少加工
- 保留来源字段
- 记录采集时间和源系统版本

### 2. DWD：明细事实层

DWD 把原始数据整理成可分析的明细事实。

作用：

- 清洗字段
- 统一主键
- 去重
- 标准化时间、状态、币种、地区等维度

原则：

- 尽量表达业务事实
- 不急着做过多聚合
- 明确事实粒度

### 3. DWS：汇总服务层

DWS 面向常见分析主题做汇总。

例如：

- 用户日活汇总
- 订单日汇总
- 支付成功率汇总
- 游戏区服活动汇总
- 风控事件汇总

原则：

- 围绕主题域
- 服务复用
- 避免每个看板重复聚合

### 4. ADS：应用数据层

ADS 面向具体消费场景：

- BI 看板
- 运营报表
- 风控看板
- 推荐特征
- AI eval dataset
- 外部 API

原则：

- 明确使用方
- 明确 SLA
- 明确口径和 owner

## 维度建模的核心问题

维度建模不是背星型模型，而是回答：

- 业务过程是什么
- 事实表粒度是什么
- 维度有哪些
- 指标在哪里定义
- 历史变化怎么处理

典型对象：

- fact table：订单、支付、战斗、登录、点击、审核
- dimension table：用户、商户、商品、游戏版本、渠道、国家、设备
- metric：GMV、DAU、留存、成功率、拒付率、LTV

## 指标口径为什么难

一个指标通常包含：

- 业务定义
- 过滤条件
- 时间窗口
- 去重规则
- 归因规则
- 数据源优先级
- 异常处理
- 延迟和刷新频率

例子：`支付成功率`

要问：

- 分母是发起支付、提交授权，还是可支付订单
- 分子是授权成功、扣款成功，还是最终 capture 成功
- 是否排除用户取消
- 是否排除风控拒绝
- 是否按金额加权
- 是否按国家、通道、支付方式分层

没有这些定义，指标只是同名不同物。

## 建模时最该避免什么

### 误区 1：直接从 ODS 出报表

短期快，长期会让每个看板都复制一套清洗逻辑。

### 误区 2：宽表越宽越好

宽表方便消费，但如果没有事实粒度和 owner，很快会变成不可维护的大泥球。

### 误区 3：指标写在看板 SQL 里

这样会让指标口径散落在 BI 工具、脚本和 notebook 里，无法治理。

### 误区 4：只建技术层，不建业务主题域

真正可复用的数据产品通常围绕业务主题域，而不是围绕任务脚本。

## 一个最短检查表

对任何数据主题，问：

1. 事实表是什么
2. 粒度是什么
3. 维度有哪些
4. 核心指标在哪里定义
5. 口径 owner 是谁
6. 实时结果和离线结果如何对齐
7. 哪些下游依赖这个数据产品
