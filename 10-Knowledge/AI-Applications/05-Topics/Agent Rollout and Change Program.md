# Agent Rollout and Change Program

## 这个主题在讲什么

这一页聚焦“一个 agent 明明技术能用，为什么组织里还是推不起来”。核心问题不是模型能力，而是 rollout 方式、培训方式、使用规范、反馈机制和团队信任建设。

## 为什么 rollout 经常决定成败

- 很多团队把 agent 当成一个工具公告出去，结果发现没人真用
- 真正的 adoption 来自：明确场景、清晰边界、低摩擦接入、持续培训和成功案例复制
- 高信任行业里，change program 本身往往是产品落地的一半

## 一个常见 rollout 路径

1. `seed team`
先在少数高意愿团队做试点，打磨使用方式和评估方法。

2. `bounded workflow`
不要先追求万能 agent，要先把一个高频、低歧义、可评估的 workflow 跑顺。

3. `training and norms`
明确什么能用、什么不能用、什么时候必须人工复核。

4. `feedback loop`
建立每周或双周复盘，把失败模式、prompt 规范、tool usage 和审批条件不断更新。

5. `scale-out`
从一个团队推广到多个团队时，同步复制模板、指标、案例和 guardrails。

## 关键观察点

- 首月活跃率与复用率
- 用户是否真的把 agent 纳入日常流程
- 哪些环节卡在信任、权限或 UI 摩擦上
- 哪些失败模式可以通过培训解决，哪些必须通过产品能力解决

## 常见误区

- 只培训“怎么点按钮”，不培训“什么时候该用、什么时候别用”
- 只看使用量，不看高质量使用率
- 把一次上线当结束，而不是把 rollout 当持续运营项目
