# Autonomous Delivery Loop

## 什么时候进入这个模式

当用户表达出下面这些意图时，默认进入 `autonomous delivery loop`：

- 继续做，不用每步都汇报
- 先做完一轮再告诉我
- 你自己判断下一步
- 不要中间打断
- 做到能交付一个版本为止

## 这套 loop 怎么跑

1. 先看当前图谱、索引、`学习进度.md`、`恢复笔记.md`
2. 列出 2 到 4 个最可能的知识缺口或拓扑缺口
3. 选一个“最高杠杆但边界清晰”的 slice，作为这一轮 batch
4. 一次性做完这一轮 batch 相关内容：主题、实体、地图、索引、进度、恢复页
5. 做完后立刻反思：
   - 哪条主线更清楚了
   - 还缺哪一块才算可教、可学、可恢复
   - 下一轮最值得补什么
6. 如果下一步明显，继续下一轮；如果已经形成一个可交付版本，就停下来汇报

## 什么叫一个好的 batch

一个 batch 不追求把整个领域写完，而追求“交付一个完整片段”。通常至少满足这些条件：

- 有清晰主题，不是零碎补丁
- 相关索引和地图已接上
- `学习进度.md` 和 `恢复笔记.md` 能说明现在学到哪里
- 新增内容已经能被用户顺着读
- wikilink 和 Git checkpoint 可以安全通过

## 什么时候应该停下来汇报

默认不要在中间为了确认而停下来。只有出现这些情况才中断：

- 下一步会改变整个拓扑，而不是局部扩展
- 有多个路线且后果明显不同
- 会涉及破坏性操作或高风险同步决策
- 已经达到一个自然的“版本边界”

## 反思时优先问自己什么

- 现在的知识路径能不能从初学到进阶顺着读
- 当前层次有没有混层问题
- 哪一页只是概念占位，哪一页已经有教学价值
- 现在最缺的是 breadth、depth、maps、case studies，还是 progress memory
- 下一轮补哪一块，能让整个系统更完整而不是更臃肿
