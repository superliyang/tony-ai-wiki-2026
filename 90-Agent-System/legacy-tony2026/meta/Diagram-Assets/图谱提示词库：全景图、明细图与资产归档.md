# 图谱提示词库：全景图、明细图与资产归档

> 这页是可复制提示词库。目标是让你可以直接拿去给大模型生成：全景图、明细架构图、能力地图、决策图、运营闭环图、Obsidian Canvas、Excalidraw、Mermaid、图片版海报，以及归档用的说明页和资产清单。

## 使用方式

1. 先选图谱类型。
2. 替换 `<领域>`、`<主题>`、`<受众>`、`<必须包含>`、`<不要包含>`。
3. 生成后把结果按 [[图谱资产管理规范]] 放进对应领域 `Maps/_assets/`。
4. 如果是正式图，更新领域 `地图索引.md`。

## 通用总提示词：先让模型问清楚再画

适合在你还没想清楚图的边界时使用。

```text
你是一名资深技术架构师、知识图谱设计师和信息设计师。请帮我设计一张关于「<领域>/<主题>」的知识图谱。

目标：
- 这张图要帮助 <受众> 在 5 秒内看懂全局结构，在 2 分钟内知道下一步该看哪里。
- 图不是海报，不追求炫酷，而是追求可学习、可维护、可复盘。
- 每张图只回答一个核心问题：<这张图要回答的问题>。

请先不要直接画图。请先输出：
1. 这张图最适合的图谱类型：全景图 / 分层架构图 / 能力地图 / 决策图 / 运营闭环图 / 路线图 / 对比矩阵。
2. 推荐阅读路径：自上而下 / 从左到右 / 中心辐射 / 闭环 / 泳道 / 象限。
3. 第一层模块，控制在 4-8 个。
4. 每个模块下最多 3-5 个二级点。
5. 哪些细节不应该塞进主图，应该拆成 drill-down 图。
6. 你建议最终输出 Mermaid、Excalidraw、Obsidian Canvas 还是图片版，并说明原因。

视觉约束：
- 第一层模块不超过 8 个。
- 节点标签尽量 4-10 个中文字符，或 2-4 个英文词。
- 不要箭头毛线团。
- 颜色必须有语义：基础、核心、运行、风险、治理、业务。
- 复杂内容拆成全景图 + 明细图。

等我确认结构后，再输出正式图。
```

## Mermaid 全景图提示词

适合长期维护、放进 Markdown、进入 Git 版本管理。

```text
你是一名资深架构图设计师。请为「<领域>/<主题>」生成一张 Mermaid 全景图。

图谱目标：
- 类型：panorama-map / 上帝视角全景图。
- 核心问题：<这张图回答什么问题>。
- 受众：<技术管理者 / 架构师 / 工程师 / 安全团队 / 学习者 / 领导>。
- 使用场景：Obsidian 知识库、技术汇报、学习导航。

内容要求：
- 必须包含：<列出必须出现的模块或概念>。
- 可以包含：<可选模块>。
- 不要包含：<排除项，避免图过密>。
- 第一层模块控制在 5-7 个。
- 每个模块最多 3-5 个关键节点。
- 如果内容太多，请把细节放在图后面的「推荐 Drill-down」列表，不要塞进主图。

Mermaid 要求：
- 使用 `flowchart TB`，除非该主题明显适合 `flowchart LR`。
- 所有中文或含空格标签都用双引号。
- 节点 ID 使用英文短 ID，例如 `Biz`、`Data`、`Runtime`。
- 使用 `subgraph` 表示层、阶段或能力域。
- 使用 `classDef` 做语义颜色：
  - foundation：浅蓝，基础层
  - core：浅绿，核心能力
  - ops：浅紫，运行治理
  - risk：浅红，风险与失败模式
  - business：浅黄，业务或应用层
- 主流程用实线箭头，反馈闭环用虚线箭头。
- 不要在 Mermaid 节点里放 Obsidian wikilink。
- 输出必须是可以直接复制运行的 Mermaid 代码块。

输出格式：
1. 先给 Mermaid 代码块。
2. 再给「图的读法」3-5 条。
3. 再给「推荐 Drill-down 图」3-5 个。
4. 最后给「资产归档建议」：
   - 说明页文件名
   - 源文件名 `.mmd`
   - prompt 文件名 `.prompt.md`
   - 建议放置目录
```

## Mermaid 明细架构图提示词

适合某一条链路、系统、流程、控制点的细化。

```text
请为「<领域>/<主题>/<具体系统或链路>」生成一张 Mermaid 明细架构图。

这不是全景图，而是 drill-down 明细图。请只回答一个问题：
> <例如：手游账号安全从登录到风控处置的端到端链路是什么？>

请按以下结构设计：
- 上层：用户 / 业务入口 / 外部系统
- 中层：核心服务、平台能力、工作流、决策点
- 下层：数据、存储、模型、规则、日志、审计
- 侧边栏：安全、权限、监控、合规、成本、失败模式

必须包含：
- <模块 1>
- <模块 2>
- <模块 3>

必须标出：
- 数据流
- 控制流
- 关键决策点
- 风险点
- 观测点
- 人工介入点或 approval gate

视觉约束：
- 不超过 35 个节点。
- 不要把长解释写进节点。
- 风险节点用浅红色。
- 治理和观测节点用浅紫色。
- 业务入口用浅黄色。
- 基础设施和数据层用浅蓝色。

输出：
1. Mermaid 代码块。
2. 图的读法。
3. 关键控制点表格：控制点 / 解决问题 / owner / 指标。
4. 推荐拆出的下一张明细图。
```

## Excalidraw 汇报图提示词

适合给领导、跨团队、分享会使用。更强调美观和可讲。

```text
你是一名信息设计师和 Excalidraw 架构图设计师。请为「<领域>/<主题>」设计一张 16:9 的 Excalidraw 风格汇报图。

目标：
- 让 <受众> 在 5 秒内知道这张图讲什么。
- 让讲解者可以按从左到右 / 自上而下的路径讲完。
- 不要堆小字，不要复杂网状关系。

画布要求：
- 比例：16:9。
- 风格：专业、清爽、低密度、手绘感但不幼稚。
- 背景：浅色。
- 模块：5-7 个主模块。
- 每个模块最多 3 个关键词。
- 使用清晰分组、对齐和留白。

视觉语义：
- 基础层：浅蓝。
- 核心能力：浅绿。
- 业务应用：浅黄。
- 运行治理：浅紫。
- 风险和失败模式：浅红。
- 反馈闭环：虚线箭头。

内容：
- 标题：<图谱标题>
- 副标题：<一句话定位>
- 主模块：<列出主模块>
- 必须突出：<最想让读者记住的 1-3 件事>
- 不要出现：<排除项>

请输出：
1. 画面布局说明。
2. 元素清单：元素类型 / 文案 / 位置 / 颜色。
3. 箭头关系。
4. 如果可以，请输出 Excalidraw JSON 或可导入 Excalidraw 的结构。
5. 另外给一版用于图片模型生成的英文 prompt。
```

## Obsidian Canvas 导航图提示词

适合做可点击知识导航，而不是一张静态图。

```text
请为 Obsidian 生成一张「<领域>/<主题>」Canvas 导航图设计方案。

目标：
- 这是知识导航，不是海报。
- 每个节点尽量对应一篇已有或计划中的 Markdown 笔记。
- 图要帮助我从「问题 -> 地图 -> 主题 -> 模板/Playbook -> 案例」进入学习。

请输出 JSON Canvas 结构或可转成 `.canvas` 的节点/边清单。

布局要求：
- 左上角放「Start Here」入口节点。
- 中间放 4-7 个主领域 group。
- 右侧放模板、playbook、案例。
- 底部放学习进度和恢复笔记。
- 边只表示真实学习顺序或依赖关系，不要为了好看乱连。

节点要求：
- 每个节点包含：id、type、label、linked_note、x、y、width、height、color。
- 如果 linked_note 不存在，请标记 `planned: true`。
- group 节点要覆盖对应主题卡片。

输出：
1. Canvas 设计说明。
2. JSON Canvas 节点和边。
3. 建议保存路径。
4. 需要新建的笔记清单。
```

## 图片模型全景图提示词

适合生成漂亮的概念海报或汇报封面。注意：图片模型不适合承载大量精确文字。

```text
Create a clean professional 16:9 technical panorama diagram about "<domain/topic>".

Purpose:
Help technical leaders and architects understand the landscape at a glance.

Visual style:
- Minimal, elegant, enterprise architecture poster.
- Light background, subtle grid, clean vector-like shapes.
- Use semantic colors, not decorative colors:
  - light blue for foundations
  - light green for core capabilities
  - light purple for operations and governance
  - light red for risks and failure modes
  - light yellow for business and applications
- Use large readable labels only.
- Keep visible text to a title and 5-8 short module labels.
- Avoid tiny text, dense network graphs, random icons, decorative clutter, or unreadable arrows.

Layout:
- Top: title "<图谱标题>"
- Left or bottom: foundations
- Center: core capabilities
- Right: runtime / operations
- Top or far right: business outcomes
- Side rail: risk / governance
- Add a subtle feedback loop arrow.

Required module labels:
<列出 5-8 个短标签，例如: Foundations, Data, Models, Runtime, Governance, Applications, Risks>

Mood:
Clear, structured, strategic, suitable for an executive technical presentation.

Negative prompt:
No tiny text, no dense mind map, no messy arrows, no random 3D icons, no neon cyberpunk style, no decorative-only elements, no illegible labels.
```

## 图片转 Mermaid / Excalidraw 重构提示词

适合你已经在 ChatGPT 里生成了一张图，但太丑、太密、不可维护。

```text
我会给你一张架构图/全景图。请不要简单复述图片内容，而是把它重构成一张更清晰、可维护的图。

任务：
1. 识别这张图想表达的核心问题。
2. 归纳第一层模块，控制在 4-8 个。
3. 删除重复、装饰性、低价值节点。
4. 把密集细节拆成「主图」和「Drill-down 图建议」。
5. 输出一版 Mermaid，可直接放进 Obsidian。
6. 如果更适合 Excalidraw，也给出 Excalidraw 布局方案。

重构原则：
- 保留业务含义，不保留原图的混乱布局。
- 每个节点标签简短。
- 使用语义颜色。
- 主流程清晰，反馈闭环用虚线。
- 风险、治理、观测作为 side rail 或 guardrail。

输出：
1. 原图问题诊断。
2. 重构后的 Mermaid。
3. 图的读法。
4. 推荐资产归档路径。
```

```text
- 目标受众：<受众>
- 这张图回答的问题：<问题>
- 图片文件名：<图谱标题>-YYYYMMDD-v1.png
- 源文件名：<图谱标题>-YYYYMMDD-v1.mmd / .excalidraw / .canvas
- prompt 文件名：<图谱标题>-YYYYMMDD-v1.prompt.md

请输出：
1. Markdown 图谱说明页，包含 frontmatter：
   - title
   - type: diagram
   - diagram_type
   - status
   - domain
   - version
   - created
   - updated
   - source_image
   - source_file
   - prompt_file

3. `地图索引.md` 应该新增的条目。
4. `专题总览.md` 如果它是主入口，应该新增的快速入口条目。
5. 推荐 Drill-down 图清单。

要求：
- 说明页用中文。
- 保留关键 English technical terms。
- 不要虚构不存在的具体文件路径；用占位路径即可。
```
