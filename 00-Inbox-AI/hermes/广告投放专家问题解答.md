## Q1: D30 ROAS 不到 40% 的中核手游，UA 应该在哪些渠道继续放量？

**场景**：中核手游，D30 ROAS <40%，iOS CPI $3–6，Android CPI $1.5–3，Payback Period 30–75 天。

### 判断框架（三明治模型）

```
          渠道效率       ×      现金约束        =      放量决策
    (D30 ROAS + 趋势)      (Payback Period)           (Scale / Hold / Cut)
```

**不要做的事情：**
- ❌ 不要对所有渠道用同一个 D7 ROAS 目标——ASA 用户 D30 LTV 是 TikTok 用户的 1.5–2x
- ❌ 不要在 D1/D7 达标时就放量——中核游戏 Payback 30–75 天，短窗口信号不够
- ❌ 不要把 UA 预算按 LTV/CPI 比值决定——受现金流转约束，不是比值约束

### 分渠道策略

| 渠道 | 当 D30 ROAS <40% 时的动作 | 原因 |
|---|---|---|
| **Apple Search Ads** | 🔼 保守放量 | D30 LTV 最高（1.5–2x TikTok），即使 D7 指标偏低也值得保留 |
| **Meta** | 🔄 按创意测试节奏放量 | ROAS 取决于创意-产品匹配，先测创意赢家再放量 |
| **Google UAC** | 🔄 40% 预算跑 tCPI（量），60% 跑 tCPA（质） | 中核游戏 Google UAC 通常 ROAS 3.3x |
| **TikTok** | 🔽 如果 D30 LTV 偏低则缩量 | 中核游戏 TikTok 用户 LTV 普遍偏低（年轻用户，付费意愿弱） |
| **Unity Ads / AppLovin** | 🔄 按 D30 ROAS 逐渠道设目标 | 只在 D30 达到品类底线（15–25%）的渠道放量 |

### 放量节奏

```
放量规则：
- D30 ROAS 连续 2 周 > 目标 → 放量 50–100%
- D30 ROAS 连续 2 周 < 底线  → 缩量 20–30%
- 每次放量不超过当前花费的 100%（不要一夜翻倍）

加权：
- 60% 预算给 Payback ≤30 天的渠道（现金流快）
- 40% 预算给 Payback >30 天但 D30 LTV 高的渠道（长期值更高）
```

### 最关键的一条
**Payback Period > 现金跑道 → 放量不会修复问题，只会放大问题。** 每个新增队列锁更多资金更久。增长不缩小缺口，而是拉大缺口。

> 来源: RocketShip HQ (2026-03), Amps33 (2026-03), GameMarketer (2026), Linkrunner (2026)

## Q2: 如何设计 SKAN Conversion Value 映射，使 6-bit 信号区分高/低 LTV 用户？

### SKAN 4.0 基础约束

```
回传窗口:
  Window 1 (D0–D2): Fine Value (0–63, 6-bit) + Coarse Value (low/medium/high)
  Window 2 (D3–D7): Coarse Value only
  Window 3 (D8–D35): Coarse Value only

关键约束:
  - Fine Value 只在 Window 1 可用（还需满足苹果的 crowd anonymity 门槛）
  - Window 2 & 3 只能拿到 Coarse Value
  - Fine Value 总容量：64 个值（0–63）
  - 不能预测某个 campaign 能否拿到 Fine 还是只拿到 Coarse
```

### 推荐策略：预测 LTV 映射法（Predictive LTV-Mapped）

**不要做的事：** 把 Revenue Bucket（$0–$1, $1–$5, $5–$10...）直接编码到 Fine Value。D0–2 的广告收入/IAP 收入太小，信号弱。

**要做的事：** 用 D0–2 的**行为信号**（不是收入信号）来预测 D30/90 LTV。

### Schema 设计实战

**Step 1: 选行为信号（预测力最强→最弱）**

| 信号 | 预测 D7 留存 | 可用性 |
|---|---|---|
| 教程完成率 | ⭐⭐⭐⭐⭐ | D0–1 |
| 第1天留存 (D1 return) | ⭐⭐⭐⭐⭐ | D1–2 |
| Session 深度 (D1 玩了几局) | ⭐⭐⭐⭐ | D1 |
| 第一次 IAP 时间 | ⭐⭐⭐⭐ | D0–7 任意 |
| 社交行为 (加好友/公会) | ⭐⭐⭐ | D1–7 |

**Step 2: 分配合适的比特数（Split Strategy）**

```
应用 "Split" 策略——不要把所有 6 bit 押在单一维度上：

高信号组合方案：
  2 bits → 关卡进度（0/1/2/3+ 关）
  1 bit  → 教程完成（是/否）
  1 bit  → D1 留存（是/否）
  1 bit  → 第一次 IAP（有/无）
  1 bit  → 预留（社交/游戏内事件）

最多 4 × 2 × 2 × 2 × 2 = 64 个值（填满但不超出）
```

**Step 3: Coarse Value 对齐**

Coarse Value 必须与 Fine Value 对齐，保证无论拿到哪种回传，信号可解读：

```
Fine Value 区间      → Coarse Value
  0–21  (低参与)      → LOW
  22–42 (中参与)      → MEDIUM  
  43–63 (高参与/IAP)   → HIGH

原则：Window 1 的 LOW/MEDIUM/HIGH 和 Window 2/3 的 Coarse 必须是同一语义
```

### 踩坑预警

| 坑 | 后果 | 修复 |
|---|---|---|
| 把 Revenue Bucket 直接编码 | D0–2 收入太散，区分力弱 | 改用行为信号预测 LTV |
| 三个 Coarse tier 设得太宽 | Medium 覆盖 80% 用户，区分力报废 | 按用户分布设阈值，确保均衡分布 |
| 忽略 Coarse Value 与 Fine Value 的对齐 | Window 2/3 数据与 Window 1 不可比 | Schema 设计时同步对齐 |
| 不跑历史校准 | 不知道哪个行为信号预测力最强 | 至少用 3 个月历史数据校准 |

> 来源: MonetizationGuy (2026-05), Linkrunner (2026), Singular (2024), Headlight (2024), Game Growth Advisor (2026-04)

## Q3: 平台 ROAS 3.5x vs 增量真实 ROAS 1.8x，如何调整？

### 先理解差距来源

| 因素 | 为什么平台 ROAS 偏高 |
|---|---|
| **有机劫持** | 用户本就会转化，广告只是"抢了有机功劳" |
| **重定向膨胀** | 已经要买的用户看到重定向广告，Last-click 归因全算在广告头上 |
| **跨渠道重叠** | 同一用户在多个渠道看过广告，每个平台都"认领"这个转化 |
| **归因窗口差异** | View-through attribution 把只是看到广告的用户也算进去 |
| **无增量对照组** | 平台无法区分"因为有广告"和"本来就会" |

**行业经验：** 平台 ROAS 通常比真实增量 ROAS 高 1.3–2x。重定向活动差距更大（可到 2–3x），拉新活动差距较小（1.2–1.5x）。

### 应对框架

```
┌────────────────────────────────────────────────────┐
│                 ROI 测量三层模型                      │
├──────────────┬────────────────┬───────────────────┤
│  Layer 1     │   Last-click   │  日常操作信号        │
│  (快速)      │   ROAS/CPA     │  每天看，指导出价     │
├──────────────┼────────────────┼───────────────────┤
│  Layer 2     │   增量测试      │  季度校准           │
│  (准确)      │   真实增量ROAS  │  用于预算分配决策     │
├──────────────┼────────────────┼───────────────────┤
│  Layer 3     │   MMM          │  半年宏观分配        │
│  (全景)      │   营销组合模型   │  跨渠道最优配比       │
└──────────────┴────────────────┴───────────────────┘
```

### 具体调整动作

**1. 预算和 KPI 校准**

```
当平台 ROAS 3.5x 但增量 ROAS 1.8x：

  内部目标 = 平台 ROAS × 校准系数
  校准系数 = 增量 ROAS / 平台 ROAS = 1.8/3.5 = 0.51

  所以：
  - UA 团队仍用平台 ROAS 做日常出价（因为出价算法依赖平台信号）
  - 但 CFO/预算审批看增量 ROAS
  - UA 团队的"内部目标 ROAS"上调：如果原本以为 2x 够，实际需要 4x+
```

**2. 渠道间预算重分配**

比较每个渠道的增量 ROAS（不是平台 ROAS）：

```
渠道 A：平台 ROAS 4x，增量 ROAS 2x  → +50% 预算
渠道 B：平台 ROAS 2x，增量 ROAS 0.3x → 立即缩量/暂停
（渠道 B 几乎所有"效果"都是抢了自然量/其他渠道的）
```

**3. 建立增量测试节奏**

- 至少**每季度**对 Top 3 渠道各跑一次增量测试
- 把校准系数纳入月度预算 review

> 来源: RocketShip HQ (2026-04), Admiral Media (2026-05), Business of Apps (2026-05), Stella (2026)

## Q4: CPI $2 (LTV $2.5) vs CPI $5 (LTV $12) 渠道，预算如何分配？

### 核心算术

```
渠道 A: CPI=$2,  D30 LTV=$2.5 → LTV/CPI=1.25, Payback=很长甚至永远回不了本
渠道 B: CPI=$5,  D30 LTV=$12  → LTV/CPI=2.4,  Payback=如果 D30 能达到 CPI 的 2.4x 就健康

只看 LTV/CPI：
  渠道 B (2.4x) > 渠道 A (1.25x)
  → 渠道 B 显著更健康

但 Payback Period 才是真正的约束！
  - 渠道 B 的 D30 LTV=$12 > CPI=$5，D30 内已经回本，明显优于渠道 A
  - 仍需确认 D7/D14 回收曲线和现金占用，因为 D30 回本不等于短期现金流无压力
  - 现金流约束 = 每日新增获客成本 × 回收天数
```

### 分阶段预算策略

| 阶段 | 渠道 A ($2 CPI) | 渠道 B ($5 CPI) | 原因 |
|---|---|---|---|
| **现金紧张**<br>(跑道 <6 个月) | 0% | 100% | A 根本回不了本；B 虽慢但能回。重 payback 短的渠道 |
| **现金健康**<br>(跑道 >12 个月) | 10% 测试 | 70% | A 做探索预算，B 做主引擎。再拿 20% 试新渠道 |
| **成熟期** | 0% | 60% | LTV/CPI<1.5 的渠道应直接砍掉 |

### 渠道特定的门槛（CPI 越高，门槛越严）

```
不是所有渠道用同一 eLTV/CPI 门槛：
  $4 CPI 渠道 → eLTV/CPI ≥ 3.0x (最低 eLTV $12)
  $10 CPI 渠道 → eLTV/CPI ≥ 4.2x (最低 eLTV $42)
  $18 CPI 渠道 → eLTV/CPI ≥ 4.2x (最低 eLTV $75)

统一门槛会让：
  高 CPI 渠道系统性接受低质量安装
  低 CPI 渠道系统性拒绝高质量安装
```

### 核心原则

**不要优化最低 CPI，优化最高 LTV/CPI 加最低 Payback 的组合。** CPI 是起跑线，不是终点线。

> 来源: Paul Levchuk (2026-04), RocketShip HQ (2026-03), GameMarketer (2026)

## Q5: Open Exchange vs PMP vs Programmatic Direct ——三维比较

### 量化对比

| 维度 | Open Exchange (RTB) | Private Marketplace (PMP) | Programmatic Guaranteed |
|---|---|---|---|
| **CPM** | $1–4 | $5–15 | $10–30 |
| **Viewability** | 50–60% | 70–85% | 80–92% |
| **IVT Rate (欺诈)** | 8–15% | 2–5% | <1% |
| **品牌安全** | ⭐ 低（需验证工具补偿） | ⭐⭐⭐ 中高 | ⭐⭐⭐⭐ 最高 |
| **透明度** | ⭐ 低（不知广告会出现在哪） | ⭐⭐⭐ 中高（知道发布商是谁） | ⭐⭐⭐⭐ 最高（精确到广告位） |
| **归因可见性** | ⭐ 低（长尾站点信号弱） | ⭐⭐⭐ 中高 | ⭐⭐⭐⭐ 最高 |
| **量级** | ⭐⭐⭐⭐⭐ 最大 | ⭐⭐⭐ 中等 | ⭐⭐ 有限 |
| **竞价** | 公开拍卖 | 邀请制拍卖 | 固定价格 1:1 |
| **质调节省** | 无 | PMP 的质量调整 vCPM 常与 Open Exchange 持平或更低 | 虽然 CPM 贵但质最高 |

### 适用场景

```
Open Exchange:
  → 目标：最大化曝光量 + 预算有限
  → 必须配：投前 IVT 过滤 + 品牌安全黑名单 + 排除列表
  → 绝对不要：品牌安全敏感的广告（金融/医疗/政府）

PMP:
  → 目标：精准受众 + 质量中保证
  → 溢价 20–40% 但欺诈低 70–80%，质量调整后可能更便宜
  → 适合：UA 投放主力渠道

Programmatic Guaranteed:
  → 目标：固定量级 + 确定性交付
  → 适合：产品发布/品牌活动/特定时间段/特定位置
  → 最贵但最干净
```

**核心权衡：Open Exchange 便宜但 8–15% 是假量；PMP 贵 2–3x 但假量只有 2–5%。质量调整后的真实成本（vCPM）往往 PMP 更优。**

> 来源: OwlClaw (2026), Rajiv Gopinath (2026), Attekmi (2024), The Marketing Juice (2026)

## Q6: IVT 率 15%→30%，排查优先级链路？

### 第一步：确认不是数据口径问题（排除 false alarm）

```
1. 确认 DSP 统计和验证商 DV/IAS 口径一致
2. 检查是否是某个单一渠道/campaign/广告位的孤立 spike
3. 排除刚启用的新渠道、新 geo、新 audience（初期噪音大）
```

### 第二步：按流量源分层排查（按优先级）

```
Priority 1: 开放交易所流量（最高风险）
  ├── 检查最近新增了哪些 exchange/SSP 接入
  ├── 检查 ads.txt / app-ads.txt / sellers.json 授权链
  ├── 抓可疑流量源的 ASN: 数据中心 IP > 住宅 IP 很可疑
  └── 检查异常 pattern: CTR 过高 (>5%)、Sessions 极少、地理位置集中

Priority 2: CTV 流量（欺诈率 12.4%，行业最高危）
  ├── 应用名/包名校验（防止 app spoofing）
  └── 检查 viewability < 30% 的 CTV 曝光

Priority 3: 移动 App 流量
  ├── 排除 SDK spoofing（点击注入 / 点击泛滥）
  └── 检查 Install Rate 突变的 campaign

Priority 4: 已信任的 PMP / Direct
  └── 理论上不应有大幅变化，若出现说明 PMP 发布商本身出问题
```

### 第三步：六层防御栈（从投前→投后）

```
Layer 1: 投前过滤 (Pre-bid, <20ms)
  ├── 拦截数据中心 ASN
  ├── 拦截已声明 bot 的 User-Agent
  ├── 拦截已知 bot IP（实时 feeds）
  └── 拦截未授权的 sellers.json

Layer 2: 实时曝光评估
  └── JA3 指纹 + ASN + 行为信号

Layer 3: 实时点击评估
  └── 点击到转化的时序异常检测

Layer 4: 点击后行为验证
  └── 页面停留时间、Session 深度

Layer 5: 转化窗口分析
  ├── 每个来源的点击→转化比例
  └── 与 baseline 偏差 >30% → 标记可疑

Layer 6: 跨源对账
  ├── DSP 报告 vs 验证商报告的差异
  └── 差异 >5% → 触发调查
```

### 第四步（阻止而非追讨）

```
投前阻止 > 投后追讨：
  ├── 立即：把可疑流量源加入 Hard Blocklist（DSP 源级排除）
  ├── 立即：把验证商 Pre-bid 开关全开（DV/IAS/HUMAN）
  ├── 短期：切向 SPO (Supply Path Optimization) 经过认证的供给路径
  └── 长期：优先 PMP + Programmatic Guaranteed（IVT 可降至 <1%）
```

> 来源: Fraudlogix (2026), Tuvoc (2026-03), Improvado (2026-04), Adsafe (2026-04), AdExchanger (2025-03)

## Q7: Google 放弃强制淘汰 Cookie 后，为什么仍不能回归 Cookie 时代？

### 三个独立但叠加的现实

**1. Chrome 不是 Web 的全部**

```
各浏览器现状（2026）：
  Safari:  第三方 Cookie 默认阻止（2017 年起）+ 7 天 JS-set cookie 限制
  Firefox: 全部 Cookie 按站点隔离（Total Cookie Protection）
  Brave:   默认 Block 所有第三方 Cookie
  
  → 17–20% 全球流量天然无 Cookie，与 Chrome 无关
```

**2. Chrome 自身也变了**

```
Google 2025 年 4 月调整第三方 Cookie 路线 → 但 Privacy Sandbox 的尝试不是白跑的
  → Chrome 用户可手动在设置中关闭第三方 Cookie
  → Topics / Protected Audience 等 API 仍代表隐私优先广告基础设施方向，不能简单回到旧 Cookie 时代
```

**3. iOS/移动端才是归因断裂的主战场**

```
iOS 14.5+ ATT → 75–85% 用户拒绝 IDFA
SKAdNetwork → 延迟几十小时 + 聚合 + 粗粒度
  → 移动游戏 UA 的 iOS 归因早在 2021 年就断裂了，跟 Chrome Cookie 无关
```

### Cookie 时代 vs 现在：五个维度对比

| 维度 | Cookie 时代 (pre-2020) | 现在 (2026) | 差距能否弥合 |
|---|---|---|---|
| 跨站受众定向 | 第三方 Cookie 串联 | 一方数据 + Clean Room | 只能部分恢复 |
| 频次控制 | Cookie 跨站追踪 | 需在 Walled Garden 内做 | 开放 Web 无法恢复 |
| 归因 | 用户级多触点 | Last-click + 增量校准 | 不再有确定性用户级 |
| 受众构建 | 第三方数据市场 | 一方数据 + 上下文定向 | 精度下降 30–50% |
| 测量 | Cookie-based attribution | MMM + 增量 + SKAN | 从精确→概率 |

### 实际操作结论

```
不能回退的三件事：
  1. 归因：从"精准用户级"变"建模聚合级"，回不去了
  2. 定向：从"第三方数据"变"一方数据+上下文"，信号少了很多
  3. 频控：开放 Web 上基本丢了，只能在 Walled Garden 内做

必须继续推进的事：
  1. 一方数据收集（登录/email/hashed PII）
  2. 服务端转化 API（Meta CAPI / Google Enhanced Conversions / sGTM）
  3. 增量测试作为真相层（不是 nice to have，是 must have）
```

> 来源: Consenteo (2026-04), Digital Applied (2026-04), Leadgen Economy (2026-05), Pxlpeak (2026-01), SignalBridge (2026-03)

## Q8: UA/产品/财务 KPI 冲突，组织和激励如何设计？

### 冲突根因

```
UA 团队 KPI: D7 ROAS, CPI, 安装量
     ↓ 冲突区域 ↓
产品团队 KPI: D30 留存, 功能采用率
     ↓ 冲突区域 ↓
  财务 KPI: Payback Period, Net Revenue

典型场景：
  UA 放量 D7 ROAS 达标 → D30 留存下降 → 财务发现 Payback 从 45 天延到 90 天
  → 三个团队都"达标了"但公司亏了
```

### 根因诊断

1. **KPI 时间窗口不一致**：UA 看 D7，产品看 D30，财务看 D90+
2. **没有人拥有"全漏斗"**：三个团队的 OKR 各讲各的，没有单一 owner
3. **北极星指标缺失**：没有向上聚合的 shared metric
4. **数据不透明**：UA 用 MMP（Gross Revenue），财务用 Net Revenue，差了 30–40%（App Store 费+VAT）

### 修复方案

**方案 1：统一北极星指标**

```
共享 KPI：D30 Net ROAS（或 D60 Net ROAS）
  - UA：为 D30 Net ROAS 负责（而不仅是 D7 CPI）
  - 产品：为 D30 留存+ARPU 负责（直接贡献 D30 ROAS）
  - 财务：统一用 Net Revenue 核算

  → 每个人都看同一条线，冲突自然减少
```

**方案 2：跨职能激励联动**

```
UA Manager 的奖金 =
  60% 基于 D30 ROAS（渠道加权）
  20% 基于 D30 留存率
  20% 基于 Payback Period ≤ 目标

产品 Manager 的奖金 =
  40% 基于 D30 留存
  30% 基于 UA 通道的 D30 LTV
  30% 基于功能采用率
```

**方案 3：Clean-Room Accounting（干净记账法）**

```
把 UA + Monetization 作为一个独立闭环：
  流入：UA 预算
  流出：队列 LTV 回款
  
  隔离运营成本（工资/服务器），只看 UA 引擎本身的效率
  → UA 和财务看同一组数字，没有"口径战争"
```

**方案 4：分流排障——不是所有 ROAS 下降都是 UA 的问题**

```
当 ROAS 恶化时：
  Stage 1: 是"根本没激活"（广告到产品不匹配）→ 创意团队的问题
  Stage 2: 是"激活后掉了"（onboarding 没钩住）→ 产品团队的问题
  Stage 3: 是"长期不付费"（经济系统弱）→ 变现/LiveOps 的问题
  Stage 4: 是"渠道质量真的差"→ UA 渠道策略的问题

不同阶段路由到不同团队，避免互相甩锅
```
