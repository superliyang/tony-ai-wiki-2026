# 广告投放领域全貌图

> 初稿由 Hermes 基于 Exa 搜索生成（2026-06-05），Codex 已做第一轮结构和边界修正。本文可作为领域学习入口；涉及行业 benchmark、比例和趋势的判断，后续应按 [[来源与校验]] 继续补 URL 级来源。

## 一、目标（核心优化方向）

广告投放领域**不是优化单一指标**，而是在一个多维约束空间里持续做 ROI 最大化：

| 方向 | 核心目标 | 典型约束 |
|---|---|---|
| **广告主侧** | 用最低成本获取最高价值的用户/转化 | 预算、目标 CPA、ROAS、品牌安全 |
| **平台/媒体侧** | 最大化每次广告展示的收益（Yield） | 用户体验、库存利用率、底价 |
| **中间层** | 高效撮合，降低交易摩擦 | 延迟、数据质量、合规 |

**核心张力：广告主想便宜买优质用户，媒体想高价卖每次曝光。中间平台在这两个方向之间做拍卖设计。**

## 二、角色与核心对象

```
┌─────────────────────────────────────────────────────────────┐
│                    广告投放生态角色图                          │
├───────────────┬─────────────────┬───────────────────────────┤
│   需求侧       │     中间层       │        供给侧              │
│  (Buy Side)   │  (Transaction)  │      (Sell Side)          │
├───────────────┼─────────────────┼───────────────────────────┤
│ 广告主         │  广告交易平台    │  媒体/发布商 (Publisher)    │
│ (Advertiser)  │  (Ad Exchange)  │                           │
│               │                 │                           │
│ 代理商(Agency) │  数据管理平台    │  SSP (供应方平台)           │
│               │  (DMP)          │  - 库存管理                 │
│ DSP (需求方平台)│                 │  - 底价策略                 │
│ - 受众定向     │                 │  - 竞价路由                 │
│ - 竞价策略     │                 │                           │
│ - 创意管理     │                 │  用户 (User)               │
│ - 预算控制     │                 │                           │
├───────────────┼─────────────────┼───────────────────────────┤
│ 辅助层: MMP(移动监测)/归因 | 验证(反作弊/可见性/品牌安全) | 隐私合规 |
└───────────────┴─────────────────┴───────────────────────────┘
```

**核心对象：**
- **广告位 (Ad Slot / Inventory)**：被交易的商品，单位是"一次曝光机会"
- **竞价请求 (Bid Request)**：包含用户画像、广告位信息、上下文数据的结构化请求
- **出价响应 (Bid Response)**：DSP 返回的出价+广告素材
- **用户画像 (User Profile)**：定位和出价的依据（正在被隐私政策持续削弱）
- **归因事件 (Attribution Event)**：安装/点击/转化——衡量广告效果的基石

## 三、关键流程（RTB 竞价全链路）

```
用户在浏览器/App中加载页面
        │
        ▼
媒体 SSP 发送广告请求 ──→ Ad Exchange
                              │
                    DMP 数据富化（用户画像）
                              │
                    Ad Exchange 分发 Bid Request
                         ┌────┼────┬────┐
                         ▼    ▼    ▼    ▼
                       DSP1 DSP2 DSP3 DSP4
                       (受众匹配/出价计算/预算检查)
                         │    │    │    │
                         ▼    ▼    ▼    ▼
                    Ad Exchange 收集所有出价
                              │
                    拍卖淘汰（Second Price / First Price）
                              │
                    返回获胜广告素材
                              │
                    SSP → 媒体页面渲染广告
                              │
                    曝光/点击/转化回传（归因闭环）

全程耗时：<100ms（DSP 内部决策 <30ms）
通信协议：OpenRTB (Protobuf/JSON over HTTP)
```

**三种交易模式：**
1. **Open Exchange（公开竞价）**：所有买家公平竞价，量最大但质量参差
2. **Private Marketplace / PMP（私有市场）**：受邀买家竞价，质量更高
3. **Programmatic Guaranteed / Direct Deal（程序化保量/直采）**：固定价格保量，品牌广告常用

## 四、核心指标体系

### 4.1 漏斗指标（从上到下）

```
曝光 (Impression)
  ├── CPM = 每千次曝光成本
  │
  ▼
点击 (Click)
  ├── CTR = 点击/曝光（衡量创意吸引力）
  ├── CPC = 成本/点击
  │
  ▼
转化 (Conversion: 安装/购买/注册)
  ├── CVR = 转化/点击（衡量落地页+产品匹配）
  ├── CPA = 成本/转化（衡量获客效率）
  │
  ▼
收入 (Revenue)
  ├── ROAS = 收入/广告花费（核心效率指标）
  ├── LTV = 用户生命周期价值
  └── Payback Period = 回收周期（LTV > CPA 所需天数）
```

### 4.2 按角色的重点指标

| 角色 | 一级指标 | 二级指标 | 陷阱 |
|---|---|---|---|
| **广告主/UA** | ROAS, CPA, LTV | CTR, CVR, 留存率, ARPPU | CPI 低不代表用户好；短窗口 ROAS 掩盖长期亏损 |
| **DSP** | 竞价胜率, eCPM, 响应延迟 | QPS, 填充率, 拒量率 | 过度优化 CTR 可能导致低意图流量 |
| **SSP** | 填充率, eCPM, 收益 | 底价策略, 超时率 | 底价太高丢填充率, 太低压收益 |
| **Ad Exchange** | 撮合率, 延迟, 吞吐量 | 买家数/拍卖, 出价密度 | 延迟是硬天花板 |

### 4.3 移动游戏 UA 关键 Benchmark（2025-2026）

| 品类 | iOS CPI | Android CPI | D7 ROAS | D30 ROAS |
|---|---|---|---|---|
| 超休闲 | $1.00-2.00 | $0.50-1.00 | 15-20% | 30-45% |
| 休闲 | $2.00-4.00 | $1.00-2.50 | 7-10% | 15-25% |
| 中核 | $3.00-6.00 | $1.50-3.00 | 4-7% | 10-20% |
| 棋牌/博彩 | $5.00-15.00 | $3.00-8.00 | 8-15% | 30-50% |
| RPG | $3.00-8.00 | $2.00-5.00 | 5-10% | 15-30% |

## 五、关键问题（What Matters）

### 5.1 信号损失（Signal Loss）—— 头号结构性问题

```
因果链：
隐私法规(GDPR/CCPA) → 用户拒绝追踪 → 第三方 Cookie 消亡
   ├── iOS ATT (75-85% 用户拒绝 IDFA)
   ├── Safari ITP (阻止第三方 Cookie)
   ├── Android Privacy Sandbox (GAID 逐步受限)
   └── Chrome 不再强制走一次性淘汰路线，但 Safari、Firefox、iOS 和用户选择已经让大量流量处在无第三方 Cookie 或弱标识环境

后果：
   ├── 归因断裂：无法确知哪个广告带来了哪个转化
   ├── 受众定向退化：不能跨站追踪用户行为
   ├── 频控困难：同一用户在不同 App/网站看到同一广告的频次失控
   └── 测量扭曲：平台自报 ROAS 通常高估 30-100%（vs. 增量测试真实 ROAS）
```

### 5.2 广告欺诈（Ad Fraud）

- **全球 IVT 率（无效流量）**：~20.6%，即 1/5 的广告曝光是假的
- **程序化欺诈损失**：约广告花费的 8.7%
- **常见手段**：点击注入、点击泛滥、SDK 欺骗、Bot 流量、域名伪装
- **最高危渠道**：CTV（联网电视）欺诈率 12.4%，开放交易所远高于 PMP

### 5.3 归因难题

| 归因模型 | 原理 | 问题 |
|---|---|---|
| 最后点击 (Last Click) | 100% 归给最后触点 | 高估底漏斗渠道，低估品牌渠道 |
| 多触点 (MTA) | 按规则/数据平分各触点 | 信号丢失后不可靠 |
| SKAdNetwork (iOS) | 苹果聚合归因，延迟回传 | 粗粒度、延迟、无法用户级 |
| MMM (营销组合模型) | 统计建模，宏观分配 | 需 12-18 个月数据起步 |
| 增量测试 (Incrementality) | A/B 对照组测真实增量 | 成本高、需统计显著性 |

**2026 年共识：Last-click 做日常操作信号 + 季度增量测试校准 + MMM 做预算分配**

## 六、核心技术架构（DSP 视角）

```
┌─────────────────────────────────────────────────────┐
│                    DSP 系统分层                        │
├─────────────────────────────────────────────────────┤
│  产品层    │ 广告主管理台: 投放设置/报表/创意管理         │
├───────────┼─────────────────────────────────────────┤
│  接入层    │ OpenRTB 解析 / 隐私校验 / 验签去重限流      │
│           │ 超时护栏 (tmax, 统一 UTC+USD)              │
├───────────┼─────────────────────────────────────────┤
│  策略层    │ 受众定向 → 频控 → Pacing → 出价计算        │
│           │ CTR/CVR 预估模型                           │
├───────────┼─────────────────────────────────────────┤
│  数据层    │ DMP: 用户画像 / 行为标签                    │
│           │ 实时+离线报表 / 日志 / 反作弊               │
├───────────┼─────────────────────────────────────────┤
│ 基础设施    │ 三级缓存 (Heap/Redis-MS/Redis-Cluster)    │
│           │ In-memory data stores / 分布式微服务        │
│           │ SLO: 端到端≤100ms / 可用性≥99.9%           │
└─────────────────────────────────────────────────────┘
```

核心性能约束：**每天百亿级竞价请求，每个请求 <100ms 内必须完成决策并返回**。

|---|---|---|

## 八、常见失败模式

| # | 失败模式 | 根因 | 后果 |
|---|---|---|---|
| 1 | **只看 CPI 不看 LTV** | 激励结构：UA 团队被 KPI 在 CPI 和短窗口 ROAS | 便宜的安装=垃圾用户，长期亏损 |
| 2 | **信平台自报 ROAS** | 平台归因无增量对照组 | 实测 ROAS 常被高估 30-100% |
| 3 | **忽略品牌安全和反作弊** | 贪便宜开放交易所流量 | 广告出现在色情/盗版网站；20% 曝光是无效流量 |
| 4 | **Cookie 时代归因惯性** | 继续按用户级归因优化 iOS 投放 | 数据不准，决策依据是错的 |
| 5 | **短窗口优化陷阱** | D1/D7 ROAS 达标就放量 | D30/D90 可能亏损，回收周期无限拉长 |
| 6 | **创意疲劳不轮换** | 同一素材跑太久 | CTR/CVR 持续下滑，不及时刷新损失 30%+ |
| 7 | **归因模型切换不做校准** | 从 Last-click 切到 First-click 或 Data-driven | 横向对比失效，通道评估完全错乱 |
| 8 | **忽略增量测试** | 增量测试成本高、周期长 | 不知道什么预算真的在起作用 |

## 九、新手误区 vs 高手权衡

### 新手误区

| 误区 | 实际 |
|---|---|
| "出价最高就能赢" | 出价只是因素之一，CTR/CVR 预估、频控、创意质量都影响胜率 |
| "CPI 越低越好" | 低 CPI 用户往往是低质量/假量，应优化 ROAS 而非 CPI |
| "Last-click 归因就够了" | Last-click 系统性高估底漏斗渠道，需增量测试校准 |
| "第三方 Cookie 消亡没影响" | 即使 Chrome 不强制停用，17-20% 流量已无 Cookie，iOS 已归因断裂 |
| "程序化=RTB 公开竞价" | PMP 和 Programmatic Direct 对品牌安全和质量有巨大差异 |

### 高手权衡

| 权衡 | 两方面 |
|---|---|
| **量 vs 质** | 开放交易所量最大化但欺诈率高；PMP 质量高但要量有限 |
| **短窗口 vs 长窗口** | D7 ROAS 快速反馈但可能掩盖长期亏损；D90 真实但要等 |
| **精准定向 vs 隐私合规** | 数据越多定向越准 → 但法规和平台限制越来越严 |
| **自动化 vs 可控性** | Google PMax/Meta Advantage+ 省力但黑盒；手动出价灵活但人力成本高 |
| **多方归因 vs 单一归因** | Last-click 统一口径；Data-driven/MMM 更准但跨平台不一致 |
| **自建 vs 平台依赖** | 自建 DSP 差异化可控但研发成本极高；用 TTD/Google 成熟但受制于人 |

## 十、2026 关键趋势

1. **AI 接管竞价和创意**：Meta Advantage+、Google PMax、TTD Kokai → 人工调参变"目标管理"
2. **服务端归因成标配**：Meta CAPI、Google Enhanced Conversions、sGTM 是恢复信号的核心手段
3. **隐私优先测量栈**：SKAN / AdAttributionKit + 增量测试 + MMM 三层模型成为标准
4. **CTV（联网电视）爆发**：CPM $30-60（远超 Display $3-8），但欺诈率最高（12.4%）
5. **Walled Garden 集中化加剧**：Google/Meta/Amazon 占数字广告 60%+，开放 Web 承压
6. **CPI 优化→LTV 优化**：行业共识从"便宜安装"转向"高价值用户"
7. **程序化创意 (DCO)**：AI 自动生成/测试/轮换素材，降低 CPI 达 30%

## 十一、专家问题清单（初版 10 问）

> 使用你的 5 要素公式：对象→变量→权衡→边界→演化

1. **（目标/角色）** 对于一款 D30 ROAS 不到 40% 的中核手游，UA 团队应该在哪些渠道继续放量？放量的决策依据是 D7 信号还是 D30 信号？
2. **（流程）** 在 iOS ATT 下，如何设计 SKAN Conversion Value 映射策略，使得粗粒度的 6-bit 信号仍然能有效区分高 LTV 用户和低 LTV 用户？
3. **（指标）** 当平台自报 ROAS 是 3.5x 但增量测试真实 ROAS 只有 1.8x 时，预算和 KPI 体系应该如何调整？
4. **（权衡）** CPI $2 的渠道和 CPI $5 的渠道——如果前者的 D30 LTV 只有 $2.5，后者的 D30 LTV 是 $12，你如何决定预算分配的比例和节奏？
5. **（分层）** Open Exchange / PMP / Programmatic Direct 三种交易模式，在欺诈率、CPM、和归因可见性三个维度上，各自的最优适用场景是什么？
6. **（失败）** 当一个月内 IVT 率从 15% 突然上升到 30%，排查的优先级链路应该是什么？如何在投前阻止而非投后追讨？
7. **（策略）** Google 放弃强制淘汰第三方 Cookie 后，为什么仍然不能回归 Cookie 时代的归因和定向策略？已损失的那部分信号到底是什么？
8. **（组织）** UA 团队 KPI 在 D7 ROAS，产品团队 KPI 在 D30 留存，财务关心 Payback Period——当三者冲突时（比如放量 D7 达标但 D30 恶化），组织架构和激励应该如何设计？
9. **（演化）** Meta CAPI / Google Enhanced Conversions / sGTM 的服务端方案，各自在信号恢复率、接入复杂度、和隐私合规上的差异是什么？如果一个广告主只能选一个先上，应该选哪个？
10. **（边界）** 在多渠道 UA 投放中，如何区分某个渠道的高 ROAS 是真正的"增量效果"还是"抢了自然量/其他渠道的功劳"？增量测试的对照组设计在移动游戏场景下的最小可行方案是什么？

## 十二、推荐学习路径

```
Phase 1: 生态全貌（1-2天）
  └── RTB 竞价的完整流程（从用户打开页面到广告展示的 100ms 全链路）
  └── DSP/SSP/ADX/DMP 四者关系

Phase 2: 核心指标（1天）
  └── CPM→CPC→CPA→ROAS→LTV 的漏斗关系和算数推导
  └── 归因模型的演变：Last-click → MTA → SKAN → MMM → 增量测试

Phase 3: 技术架构（1-2天）
  └── OpenRTB 协议基础
  └── DSP 内部的分层架构和性能约束（<100ms, 百亿级/天）
  └── 缓存策略、CTR/CVR 预估模型、预算 Pacing 算法

Phase 4: 行业痛点（1-2天）
  └── 信号损失：ATT、Cookie 消亡、Privacy Sandbox
  └── 广告欺诈：IVT 类型、检测手段、投前过滤
  └── 游戏 UA 专项：SKAN 4.0/5.0, MMP 选型, LTV 预测

Phase 5: 实战视角（按需）
  └── 读 The Trade Desk 的年报/投资者信（最好的 DSP 视角）
  └── 读 AppsFlyer/Adjust 的年度报告（最好的移动归因视角）
  └── 自己用 Meta Ads / Google Ads 投一个最小预算的测试活动
```
