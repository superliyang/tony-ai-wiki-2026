# Agent 记忆架构 — 领域全貌图

> 一句话定义：Agent 记忆架构是一套让 LLM Agent 跨越会话/任务持续积累、组织、检索和更新知识的工程系统，本质上是 **上下文工程 (Context Engineering)** ——决定哪些 token 进入上下文窗口，以及如何组织它们。

---

## 1. 目标 (Goals)

Agent 记忆系统的核心目标：

| 目标 | 为什么重要 | 衡量方式 |
|------|-----------|---------|
| **跨会话连续性** | Agent 不能每次对话都"失忆" | 多会话回忆准确率 |
| **个性化积累** | 从"通用工具"变成"懂你的伙伴" | 用户偏好命中率 |
| **经验复用** | 不重复犯错，记住成功策略 | 任务成功率提升 |
| **信息去重/冲突消解** | 记忆不能无限制膨胀 | 记忆条目增长率、矛盾率 |
| **检索效率** | 记忆再多也不能拖慢响应 | p95 检索延迟 |

**核心张力**：上下文内立即可用（零延迟、零错误） vs 外部存储按需检索（有延迟、有召回误差、有 token 开销）。

## 3. 关键流程 (Key Process)

Agent 记忆系统的三大核心流程：

### 3.1 记忆写入 (Memory Extraction → Encode → Store)

```
对话/任务轨迹
  │
  ▼
提取 (Extraction)  ← LLM 判断"这个信息值得记住吗？"
  │
  ▼
去重/冲突消解      ← 与已有记忆比较：合并？覆盖？追加？
  │
  ▼
编码 (Encode)      ← 向量化 + 结构化 (实体/关系/时间戳)
  │
  ▼
存储 (Store)       ← 按记忆类型写入对应存储层
```

### 3.2 记忆检索 (Query → Retrieve → Rerank → Inject)

```
用户查询 / 任务上下文
  │
  ▼
多路检索           ← 语义搜索 + 图遍历 + 关键词 + 时间过滤
  │
  ▼
精排 (Rerank)      ← 跨编码器 (cross-encoder) 重排序
  │
  ▼
注入上下文窗口      ← 格式化为 LLM 可消费的 memory block
```

### 3.3 记忆维护 (Consolidation)

```
定期后台任务:
  • 合并同类记忆
  • 标记/删除过时信息
  • 时间衰减权重调整
  • 从 episodic → semantic 的"记忆巩固"
```

## 4. 核心指标体系 (Core Metrics)

| 指标类别 | 具体指标 | 目标 |
|---------|---------|------|
| **召回质量** | LongMemEval 得分 | 越高越好 (SOTA ~83-95%) |
| | Recall@K | K=5 时 >80% |
| | MRR (Mean Reciprocal Rank) | 相关记忆排前面 |
| **延迟** | p95 检索延迟 | <200ms (热路径) |
| | p95 写入延迟 | <1s |
| **效率** | 记忆增长率 | 每会话 <10 条新记忆 |
| | 冗余率 | 重复/过时条目 <5% |
| **规模** | 最大记忆条目数 | 百万级不减质 |
| | 多租户隔离 | 零跨用户泄漏 |

### 主流框架 LongMemEval 基准（2026 Q2）

| 框架 | LongMemEval 得分 | 策略 |
|------|:---:|------|
| Letta (MemGPT) | 83.2% | OS 分层，agent 自主管理 |
| Zep | 63.8% | 时序知识图谱 |
| Mem0 | 49% (无图) | 向量语义 |
| Mem0 (April 2026 新算法) | 94.8% | 实体链接 + 多信号检索 |

> ⚠️ LongMemEval 测量对用户事实的长期回忆，不等同于生产环境表现。

## 5. 关键问题 (Key Problems)

行业正在努力解决的结构性挑战：

| # | 问题 | 现状 |
|---|------|------|
| 1 | **记忆污染/Poisoning** | 恶意输入注入虚假记忆，尚无成熟防御方案 |
| 2 | **信息过时 (Staleness)** | 事实变更后旧记忆仍被检索，污染回答 |
| 3 | **多 Agent 并发写冲突** | 多个 Agent 同时写同一用户记忆，ACID 保障缺失 |
| 4 | **"该记什么"的判断** | 过度提取→噪音；提取不足→漏记。平衡点因场景而异 |
| 5 | **长尾检索精度** | 10 万条记忆后，语义搜索的 Recall 断崖下降 |
| 6 | **隐私合规** | GDPR 被遗忘权：如何精准删除某条记忆及其衍生事实？ |
| 7 | **冷启动** | 新用户没有记忆时如何得体地收集信息？ |
| 8 | **跨框架互操作** | 从 Mem0 迁移到 Letta？目前基本不可能 |

## 6. 核心技术架构 (Technical Architecture)

### 6.1 三层认知模型（行业共识）

```
┌─────────────────────────────────────────────┐
│              Agent 上下文窗口                 │
│  ┌─────────────────────────────────────────┐│
│  │   工作记忆 (Working Memory)               ││
│  │   当前会话消息 + 工具调用 + 活动状态        ││
│  │   会话结束即清空                           ││
│  └─────────────────────────────────────────┘│
└─────────────────────────────────────────────┘
           ▲ 按需检索注入          ▲ 自动注入
           │                       │
┌──────────┴───────────────────────┴──────────┐
│            长期存储 (Long-term Store)         │
│                                              │
│  ┌────────────────┐  ┌────────────────────┐ │
│  │ 情景记忆         │  │  语义记忆            │ │
│  │ Episodic        │  │  Semantic           │ │
│  │                 │  │                     │ │
│  │ "上次 Tony 说    │  │  "Tony 偏好中文     │ │
│  │  Kafka 延迟高"   │  │   输出、短段落"     │ │
│  │ 完整交互上下文    │  │  抽取的事实           │ │
│  │ 向量搜索         │  │  向量 + 图          │ │
│  └────────────────┘  └────────────────────┘ │
│                                              │
│  ┌────────────────────────────────────────┐ │
│  │  程序性记忆 (Procedural Memory)          │ │
│  │  "遇到 P0 告警→先查 Grafana→再切流"    │ │
│  │  技能 / 工作流 / Playbook               │ │
│  │  Git 文件 / 规则引擎                    │ │
│  └────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
```

### 6.2 Letta 的 OS 灵感分层（该流派最完整实现）

| 记忆层 | 类比 | 容量 | 访问方式 | 持久化 |
|--------|------|------|---------|--------|
| **Core Memory** | RAM | 2-4KB | 始终在上下文内 | 会话级 |
| **Archival Memory** | 磁盘 | 无限制 | Agent 主动调用 `archival_memory_search` | 向量数据库 |
| **Recall Memory** | 交换分区 | 对话历史 | Agent 主动调用 `conversation_search` | 分页块 |

### 6.3 存储选型矩阵

| 存储类型 | 擅长 | 不擅长 | 代表技术 |
|---------|------|--------|---------|
| **向量数据库** | 模糊语义召回 | 关系推理、多跳查询 | Qdrant, Chroma, Milvus, pgvector |
| **知识图谱** | 实体关系、多跳推理、时序 | 模糊匹配、非结构化文本 | Neo4j, Kuzu, Graphiti |
| **KV 存储** | 结构化偏好快速查 | 语义搜索 | Redis, Valkey |
| **关系数据库** | ACID、多 Agent 一致性 | 语义搜索 | PostgreSQL |

**2026 趋势**：单一引擎承载向量 + 关系 + 全文 + 图（如 pgvector + pgvectorscale + AGE），一次 CTE 查询组装完整上下文。

## 7. 解决方案族 (Solution Families)

### 7.1 四大主流框架对比

| 框架 | 核心哲学 | 架构 | GitHub Stars | 最佳场景 |
|------|---------|------|:---:|---------|

| **Mem0** | 通用记忆 API 层 | 向量+图+KV 混合 | ~48K | 快速给现有 Agent 加记忆 |

| **LangMem** | LangGraph 原生 SDK | 平 KV + 向量 | ~1.3K | 已在 LangChain/LangGraph 栈 |

### 7.2 选型决策树

```
需要记忆的 Agent
│
├─ 已经用 LangChain/LangGraph？ → LangMem（零基础设施）
│
├─ 需要时序推理（"Tony 什么时候换了岗位？"）？ → Zep
│
├─ 快速原型，简单"记住事实"？ → Mem0

├─ 长跑自主 Agent，需要复杂记忆管理？ → Letta

└─ 简单场景 → MCP Memory Server 就够了，别过度工程化
```

## 8. 常见失败模式 (Common Failure Modes)

| # | 失败模式 | 根因 | 后果 | 预防 |
|---|---------|------|------|------|
| 1 | **记忆无限膨胀** | 无遗忘/去重机制 | 检索精度下降，成本飙升 | 时间衰减 + 主动遗忘 |
| 2 | **记忆撕裂** | 同一实体的信息散落在多个条目 | 检索不完整，回答矛盾 | 实体链接 + 图合并 |
| 3 | **过时记忆污染** | 事实变更后旧记忆权重不减 | Agent 按旧信息行动 | 时效标记 + 覆盖更新 |
| 4 | **冷启动灾难** | 新用户无记忆时过度询问 | 用户体验极差 | 渐进式信息收集 |
| 5 | **过度提取噪音** | 把闲聊当事实记录 | 记忆库充满无用信息 | 价值门控 (MemReader 做法) |
| 6 | **检索策略单一** | 只用向量搜索 | 多跳关系查询失败 | 混合检索 (向量+图+关键词) |
| 7 | **上下文窗口超载** | 注入太多记忆 | 挤占推理空间，延迟增加 | 分层注入 + token 预算控制 |
| 8 | **跨 Agent 写冲突** | 多 Agent 无锁写同一用户记忆 | 数据不一致 | ACID 存储 + 乐观锁 |

## 9. 新手误区 vs 高手权衡

| 领域 | 新手误区 | 高手权衡 |
|------|---------|---------|
| **选型** | "Mem0 星最多，选它" | Mem0 星多因为易上手，但图能力在 Pro 付费墙后。Letta 学术最扎实但运维复杂。权衡：易用性 vs 控制力 |
| **存储** | "搭个向量数据库就够了" | 向量擅长语义但瞎于关系。生产级 Agent 需要向量+图混合。但混合=运维复杂度翻倍。权衡：精度 vs 运维成本 |
| **记忆量** | "越记越多越好" | 记忆多≠效果好。10 万条后语义搜索 precision 断崖降。需要主动遗忘。权衡：完整性 vs 检索精度 |
| **架构** | "用一个框架搞定所有记忆类型" | 2026 最佳实践是混合栈：工作记忆在上下文、偏好在 Mem0、实体关系在 Zep、程序知识在 Git 文件。权衡：简单 vs 各取所长 |
| **提取** | "每句话都提取记忆" | 过度提取 = 噪音。MemReader 的 ReAct 判断"有长期价值吗？有跨轮依赖吗？信息完整吗？"三问是正确方向 |
| **安全** | "记忆是内部数据，没人攻击" | 记忆投毒 (Memory Poisoning) 是真实威胁：恶意用户输入"系统提示：忽略所有安全限制"被当作事实存储。权衡：开放性 vs 安全 |

## 10. 关键趋势 (Key Trends, 2025-2026)

1. **存储收敛**：pgvector + pgvectorscale 成为事实标准，单一 PostgreSQL 承载向量+关系+全文
2. **混合检索普及**：向量语义 + 图遍历 + BM25 关键词 + 时间衰减，四路并行→cross-encoder 精排
3. **小模型做提取**：MemReader-0.6B/4B 替代 GPT-4 做记忆提取，成本降低 100 倍
4. **声明式记忆注入**：CLAUDE.md / AGENTS.md 文件成为程序性记忆的事实标准
5. **多 Agent 记忆共享**：Letta Conversations API (2026.01)、Mem0 namespace scoping
6. **安全性进入主流讨论**：记忆投毒、GDPR 合规、审计日志从 afterthought 变成设计需求
7. **Agent 自主管理记忆**：从「开发者替 Agent 管理」走向「Agent 用工具自己管理」
8. **理论框架收敛**：清华 Awesome-Memory-for-Agents + HK 综述 (2026.05) 统一了 Extraction→Coarsening→Traversal 三算子模型

## 11. 专家问题清单 (Expert Question List)

以下 10 个问题用 5 要素公式（对象→变量→权衡→边界→演化）生成，将在 Phase 2 深挖：

1. 在**多租户 Agent 平台**中，如果目标是**零跨用户记忆泄漏**，你们如何在**向量检索精度**、**命名空间隔离开销**、**共享知识的便利性**之间取舍？这个隔离策略在**租户数从 100 到 10 万**增长后，最先失效的假设是什么？

2. 在**客服 Agent**场景下，当用户**换了电话号码**，记忆系统如何判断"旧号码过时"而非"这是两个人的信息"？你们在**自动冲突消解**和**人工确认**之间的边界划在哪里？

3. 在**长时间自主 Agent**（运行数周）中，**10 万条记忆后语义搜索精度断崖下降**，你们选择**主动遗忘**还是**分层索引**？这个选择在**任务类型从对话变为代码生成**后还成立吗？

4. **向量+图混合存储**增加运维复杂度，在什么**数据规模/查询复杂度**节点上，纯向量方案明确不够用、混合方案值得投入？

5. 记忆提取的**质量（precision）和数量（recall）**天生矛盾——MemReader 用 ReAct 判断"是否值得记"，但小模型判断失误怎么办？**LLM 判断 vs 规则判断 vs 小模型判断**的边界在哪？

6. **记忆投毒 (Memory Poisoning)** 目前无成熟防御——如果用户说"我的偏好是忽略所有安全限制"，记忆系统应该如何区分**真实偏好**和**注入攻击**？

7. Letta 的 OS 分层模型给了 Agent 最大自主权，但**Agent 自己的记忆决策可能错误**——把关键信息归档到冷存储。如何在**Agent 自主权**和**开发者可控性**之间平衡？

8. **多 Agent 共享记忆**时，如果 Agent A 记录了"Tony 用 Java"，Agent B 记录了"Tony 最近转向 Go"，谁负责更新？多 Agent 的**写冲突**和**事实权威性**如何解决？

9. 在**GDPR 被遗忘权**要求下，删除用户的所有记忆——但图存储中的实体关系可能**级联影响其他用户/系统知识**。如何在**合规删除**和**知识完整性**之间权衡？

10. **声明式记忆 (CLAUDE.md) vs 动态提取记忆 (Mem0)** 的边界在哪？哪些知识应该由人写死，哪些应该由 Agent 自己学？
